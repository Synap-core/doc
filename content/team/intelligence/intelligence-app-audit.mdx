# Intelligence App – Audit & Implementation Plan

**Purpose:** Align with the cofounder’s product direction (5 surfaces: Threads, Commands, Agents, Connections, Runs), add command-palette integration and “create your own intelligence service,” and deliver a phased plan with backend/frontend separation, package restructure, migrations, routers, and UI.

**References:** Cofounder proposal (5 surfaces, Commands as UX primitive, Raycast placeholders, Agents, Connections, Runs/Audit), [INTELLIGENCE_APP_VISION_AND_IMPLEMENTATION.md](./INTELLIGENCE_APP_VISION_AND_IMPLEMENTATION.md), [AI_CHAT_UPDATE_PLAN.md](./AI_CHAT_UPDATE_PLAN.md).

**Cofounder feedback incorporated (this revision):** Single source of truth for command inputs (template → compiled_template_ast + derived_inputs; optional input_overrides). Formal selectionContext type. Every command_run has a thread_id (provenance). Streaming transport defined up front (SSE or tRPC subscription). Explicit permission model (allowed_tools, allowed_entity_types, max_entities_created_per_run, can_create_views; snapshot on run). Connections: passive detection untrusted until validated; extend existing webhook_subscriptions (no new table). Service manifest for “create your own” (name, version, capabilities[], optional endpoints, auth type). UI: same handling as other apps (global sidebar; local header + tabs inside panel content only). Runs early in phasing; Connections after Commands + Agents feel good.

---

## 1. Product Summary (Locked)

- **One app, five surfaces:** Threads | Commands | Agents | Connections | Runs (tabs or left-nav in the Intelligence app).
- **Commands:** Reusable prompt+tool wrapper; Raycast-style placeholders (`{argument name="Topic"}`, `{selection}`, options/defaults/modifiers); run sheet → stream + proposals → approve.
- **Agents:** Directory of built-in + user-created agents; enable/disable, permissions, “what they can do”; model + system instructions + allowed tools.
- **Connections:** External integrations (n8n, ActivePieces, custom webhook) – explicit registration + passive detection; store provider, status, webhook subscriptions, delivery logs.
- **Runs/Audit:** Unified timeline of what ran, inputs, proposals, approved, what changed (event-sourced).
- **Command palette:** Search and run Commands from palette; “Create command” from palette; type-to-run, duplicate, quick edit.
- **Intelligence app shell:** **Same handling as other apps** (Views, Data, etc.): keep the global left icon sidebar; use the same single props/layout pattern. **Inside** the Intelligence panel content only, render a **local header with tabs** (Threads | Commands | Agents | Connections | Runs), like Home’s “Workspace | My Home.” No special-casing of navbar/sidebar for this app.
- **User’s own intelligence service:** Show “currently connected to &lt;service&gt;” and explain how to create/connect their own; backend fetches and displays which service is used and its data (address, skills/capabilities). Use a **service manifest** contract (see §3.7) so UI can show capabilities without guessing.

## 3. Backend – Required Updates

### 3.1 Data Model (New Tables / Columns)

#### 3.1.1 Single source of truth for Command inputs (no manual input_schema)

- **Store only `prompt_template`** as the source of truth. Do **not** maintain a separate `input_schema` by hand (drift risk).
- **On create/update:** Run the template compiler; persist:
  - **compiled_template_ast** (or equivalent) – parsed structure for substitution.
  - **derived_inputs** – argument definitions **derived from** the template (e.g. all `{argument name="X"}` placeholders).
- **Optional input_overrides** (JSONB): user-provided overrides for **labels**, **defaults**, **options** only. The parser owns the base set of argument names and presence; overrides cannot add/remove arguments that the template doesn’t declare.
- This prevents “template says `{argument name="Tone"}` but schema forgot Tone.”

| Entity                                      | Purpose                                           | Suggested columns                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------------------------- | ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **intelligence_commands**                   | User-created Commands (Raycast-style)             | id (uuid), workspace_id, created_by (user_id), title, **prompt_template** (source of truth), **compiled_template_ast** (JSONB, from parser), **derived_inputs** (JSONB, from parser), **input_overrides** (JSONB, optional labels/defaults/options), **allowed_tools** (JSONB array or null = all), **allowed_entity_types** (JSONB array or null), **max_entities_created_per_run** (int or null), **can_create_views** (boolean), output_mode (text \| proposal \| view), **permissions_profile** (read_only \| propose_writes), shared_scope (workspace \| user), created_at, updated_at. |
| **command_runs**                            | Audit / Runs timeline; **every run has a thread** | id (uuid), **thread_id** (uuid, FK to chat_threads, required – provenance), command_id (FK), workspace_id, user_id, **permissions_snapshot** (JSONB – copy of command’s permissions at run time), inputs (JSONB), selection_context_snapshot (JSONB, optional), output_summary (text or JSONB), proposed_actions (JSONB), approved_actions (JSONB), status (running \| completed \| failed), started_at, completed_at, error_message (optional).                                                                                                                                             |
| **integration_connections**                 | External automations (n8n, ActivePieces, custom)  | id (uuid), workspace_id, created_by, provider (n8n \| activepieces \| custom \| webhook), name, status (healthy \| degraded \| offline), metadata (JSONB), created_at, updated_at, last_health_at.                                                                                                                                                                                                                                                                                                                                                                                           |

Cofounder: webhookSubscriptions[] on connection; extend existing table instead of creating integration_webhook_subscriptions.                                                                                                                                                                                                                                                                                                                |

#### 3.1.2 selectionContext – formal typed payload

`{selection}` is the most-used placeholder; it must have a strict shape so Commands behave predictably across Canvas/Data/Views/Docs.

- **selection.type:** `entities` | `viewRows` | `documents` | `text`
- **selection.entityIds[]** – when type is `entities` (e.g. selected entities from Data).
- **selection.viewId** + **selection.rowEntityIds[]** – when type is `viewRows` (rows from a View).
- **selection.documentIds[]** – when type is `documents`.
- **selection.text** – when type is `text` (e.g. clipboard or selected text from Docs).

Define this type in shared types (e.g. `@synap-core/types`) and use it in `runCommand` and in the template compiler when resolving `{selection}`.

#### 3.1.3 Permission model (explicit at Command-level)

- **permissions_profile:** `read_only` | `propose_writes` (already in plan).
- **Also store on command (and snapshot on run):**
  - **allowed_tools** (array or null = all)
  - **allowed_entity_types** (array or null)
  - **max_entities_created_per_run** (int or null)
  - **can_create_views** (boolean)
- **command_runs.permissions_snapshot:** Copy of the command’s permission fields at run time so history is not rewritten if the command is edited later. This avoids “one command accidentally becomes a super-admin.”

### 3.4 Template Compiler (Placeholders)

- **Location:** `packages/api/src/utils/command-template.ts` (or under intelligence).
- **Parse:** `{argument name="Topic"}`, `{argument name="Tone" options="a,b,c" default="a"}`, `{selection}` (and modifiers: trim, uppercase, raw).
- **Output:** (1) **derived_inputs** for UI form (parser owns base set); (2) function to substitute argument values + **selectionContext** (typed payload) into template.
- **Validation:** All required arguments from derived_inputs must be provided before run; selectionContext must match selection.type when template uses `{selection}`.

### 3.5 Execution (Run Command) – thread + transport

- **Execution primitive:** Every command run **creates a thread** (or reuses one by policy). **command_run** always stores **thread_id**. Threads = “narrative log”; runs = “operational log.” Users can open the provenance conversation from a run.
- **Flow:** `runCommand` → create row in command_runs (status=running, thread_id set) → create thread + first message (compiled prompt + selection context) → stream via Hub → update command_runs on completion/failure.
- **Streaming + proposals – define transport now** (so Commands tab doesn’t feel laggy and we don’t re-architect mid-build):
  - **Option A:** SSE endpoint e.g. `GET /runs/:id/stream` (or tRPC equivalent) that streams chunks for that run.
  - **Option B:** tRPC subscription (if the app already uses WebSockets for real-time) – subscribe to runId, receive stream events.
- **Recommendation:** Pick one and document it in this plan; implement in Phase 1 so runCommand returns runId + threadId and the client can attach to the stream via the chosen transport.

### 3.6 Connections – passive detection (untrusted until confirmed)

- **Do not** create “Detected Connection” records from arbitrary webhook traffic. Treat passive detection as **untrusted** until confirmed.
- **Create or attach a “Detected” connection only if** you can validate via:
  - a **shared secret signature header** (e.g. webhook signing), or
  - a **token-bound webhook subscription** (user already registered this URL with a token), or
  - **user-confirmed mapping in UI** (e.g. “This webhook belongs to connection X”).
- Otherwise you risk noisy or forged “connections” records. Prefer explicit registration; use passive detection only when one of the above validates.

### 3.7 Service manifest (create your own intelligence service)

- Store a minimal **service manifest** in `intelligence_services.metadata` (or a dedicated JSONB column) so custom services declare capabilities predictably.
- **Manifest shape (minimal):**
  - **name**, **version**
  - **capabilities[]**
  - **optional endpoints supported:** e.g. `agents.list`, `skills.list`, `runs.execute`
  - **auth type required**
- Then the UI can show “Connected to X” and “Supports agent discovery” (or not) without guessing. getEffective(workspaceId) returns this manifest so the Intelligence app can render it.

### 3.8 “Current Intelligence Service” and “Create Your Own”

- **getEffective(workspaceId):** Resolve service; return public info + **manifest** (name, version, capabilities[], optional endpoints, auth type). Used by Intelligence app to show “Connected to: &lt;name&gt;” and capabilities.
- **List registered services:** Already `intelligenceRegistry.list`; workspace admin can link workspace to a different service (workspace.settings.intelligenceServiceId).
- **Docs:** “Create your own intelligence service” doc (or link to repo + Hub protocol spec); include manifest contract so implementors know what to expose.
