## 📋 What We've Built So Far

### ✅ **1. Unified View Renderer System**

We've created a complete view rendering architecture that handles all view types:

**Core Package**: `@synap/core/view-renderer`

- **`ViewRenderer`**: Main entry point that loads views and routes to appropriate renderers
- **`StructuredViewRenderer`**: Routes structured views (table, list, grid, kanban) to adapters
- **`ViewTypeSwitcher`**: UI component for switching between view types
- **`useViewConfig`**: Hook for managing view configuration with auto-sync
- **`useViewData`**: Hook for fetching entities based on `EntityQuery`

**Architecture Flow**:

```
ViewRenderer
  ├── Loads view from backend (via tRPC)
  ├── Merges pending config changes (viewConfigStore)
  ├── Routes by category:
  │   ├── 'canvas' → WhiteboardCanvas or Bento
  │   └── 'structured' → StructuredViewRenderer
  │       ├── 'table' → TableAdapter → EnhancedEntityTableView
  │       ├── 'list' → ListAdapter → ListView
  │       ├── 'grid' → GridAdapter → GridView
  │       ├── 'kanban' → KanbanAdapter
  │       ├── 'calendar' → CalendarAdapter
  │       ├── 'flow' → FlowAdapter
  │       └── 'branch_tree' → BranchTreeAdapter
```

### ✅ **2. Standard Table View (Complete)**

**Package**: `@synap/entity-table-view`

**Features**:

- ✅ Standard table mode with columns, sorting, filtering
- ✅ Column management (show/hide, reorder, resize)
- ✅ Flexible column widths (auto-fill 100% width)
- ✅ Multi-select rows
- ✅ View mode selector UI (standard/pivot/kanban)
- ✅ Integration with view API (load/save config)
- ✅ Offline state management (pending changes)

**Current State**: Fully functional ✅

### ✅ **3. Backend Integration**

**Backend Types** (from `@synap-core/types`):

- `ViewConfig`: `{ category: 'structured' | 'canvas', query: EntityQuery, render?: RenderSettings }`
- `EntityQuery`: Filters, sorts, pagination, `groupBy` (for Kanban)
- `RenderSettings`: Columns, `groupByField`, `cardFields`, formatting rules

**Frontend Types** (in `@synap/stores`):

- Aligned with backend types
- Extended with client-only state (selections, UI state)

### ✅ **4. View Management Page**

- Full page for viewing any view type

### **What We Need to Build**

#### **1. Pivot Table Mode** 🔄

**Concept**: Dynamic columns generated from metadata values

**Example**:

- Pivot by `metadata.status`
- Columns: "To Do", "In Progress", "Done", "Archived"
- Entities grouped into columns based on their status value

**Configuration** (already defined):

```typescript
interface PivotTableConfig {
  pivotField: string; // e.g., 'metadata.status'
  columnGeneration: "enum" | "dynamic" | "custom";
  columnOrder?: string[];
  defaultValueColumn?: string; // For entities without pivot value
  aggregateFunctions?: Record<string, "count" | "sum" | "avg">;
}
```

**Implementation Needs**:

1. **`PivotTableView` component**:
   - Generate columns from `pivotField` values
   - Group entities by pivot value
   - Render as table with dynamic columns
   - Support aggregations (count, sum, etc.)

2. **Column Generation Logic**:
   - **Enum mode**: Use `ENTITY_SCHEMAS` to get enum values
   - **Dynamic mode**: Analyze entities to find unique values
   - **Custom mode**: User-defined columns

3. **Integration**:
   - Add to `EnhancedEntityTableView` switch statement
   - Convert `RenderSettings.groupByField` to `PivotTableConfig`
   - Handle config changes back to `RenderSettings`

#### **2. Kanban View Mode** 📋

**Concept**: Specialized pivot mode with drag-and-drop cards

**Example**:

- Kanban board with columns: "To Do", "In Progress", "Done"
- Cards show entity title, assignee, priority
- Drag cards between columns to change status

**Configuration** (already defined):

```typescript
interface KanbanConfig {
  pivotField: string; // Usually 'metadata.status'
  columns: KanbanColumn[]; // Explicit column definitions
  allowCustomColumns?: boolean;
  defaultColumnId?: string;
  cardHeight?: "compact" | "normal" | "large";
  showCardFields?: string[]; // Which fields on cards
}

interface KanbanColumn {
  id: string;
  value: string; // The metadata value
  label: string; // Display label
  order: number;
  color?: string;
  limit?: number; // WIP limit
}
```

**Implementation Needs**:

1. **`KanbanView` component**:
   - Render columns as vertical lanes
   - Render entities as cards within columns
   - Drag-and-drop support (react-beautiful-dnd or dnd-kit)
   - Update entity metadata on drop
   - Show WIP limits and warnings

2. **Card Component**:
   - Display entity fields (title, assignee, priority, etc.)
   - Support different card sizes
   - Click to open entity detail

3. **Integration**:
   - Add to `StructuredViewRenderer` switch statement
   - Create `KanbanAdapter` (like `TableAdapter`)
   - Convert `RenderSettings.groupByField` + `cardFields` to `KanbanConfig`
   - Handle drag-and-drop updates via entity API

## 🏗️ **Architecture Decisions**

### **1. View Categories**

**Structured Views** (`category: 'structured'`):

- Table, List, Grid, Kanban
- Query-driven (fetch entities dynamically)
- Configuration in `metadata.config`
- No document backing

**Canvas Views** (`category: 'canvas'`):

- Whiteboard (only one currently)
- Document-driven (content in `documentId`)
- Uses document versioning system
- Heavy content (Yjs blob)

### **2. Pivot Mode Philosophy**

**Why "Pivot Mode" instead of separate view types?**

- ✅ More abstract, agnostic layer
- ✅ Can handle any metadata field as columns
- ✅ Enables many data visualization patterns
- ✅ Kanban is just a specialized pivot with drag-and-drop

**Backend Support**:

- `EntityQuery.groupBy` field exists
- `RenderSettings.groupByField` exists
- Backend can handle pivot queries

### **3. Configuration Storage**

**Backend** (`views` table):

- Lightweight metadata in `metadata.config` (JSONB)
- Heavy content in `documents` table (via `documentId`)

**Frontend**:

- `viewConfigStore` (Zustand): Pending changes (offline)
- Auto-sync to backend on changes
- Merges saved config + pending changes

## 💡 **Key Questions to Resolve**

1. **Pivot Column Generation**:
   - How to access enum values from `ENTITY_SCHEMAS` at runtime?
   - Should backend provide metadata schema introspection endpoint?
   - How to handle custom metadata fields?

2. **Kanban Drag-and-Drop**:
   - Which library? (`react-beautiful-dnd`, `@dnd-kit/core`, or custom?)
   - How to handle optimistic updates?
   - Should we batch updates for multiple moves?

3. **Mode Switching**:
   - Should we preserve column order when switching?
   - How to handle incompatible configs?
   - Should we show a preview before switching?

4. **Performance**:
   - How many entities can we handle in pivot/kanban?
   - Should we virtualize columns/cards?
   - Do we need pagination for pivot tables?
