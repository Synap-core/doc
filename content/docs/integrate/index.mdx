# Build & integrate

This section is for **builders and operators**: how Synap’s **data pod**, **intelligence layer**, and **external tools** fit together — not a flat list of endpoints.

---

## Why this matters

Synap is designed so **your graph is portable**: anything you import or connect becomes **entities, views, and channels** you can reuse across apps. Integrations are not sidecars; they feed the same event-sourced core and the same governance (permissions, proposals) as the product UI.

---

## Cold start: import on sign-up, reuse forever

A new pod is empty by default. The **browser extension** (Chrome MV3, in `synap-app`) addresses **cold-start** by reading open tabs, clustering them, and **bulk-importing** pod-wide entities so the space starts full, not empty — then every capture stays in the graph for search, views, and AI. See **`EXTENSION.md`** in `synap-app/packages/apps/extension` and **[Synap products](/team/home/synap-products)**.

**Integration takeaway:** treat onboarding imports as **first-class data**, not a one-off wizard — the same entities power automations and agents later.

## Hub Protocol: the stable surface for “anything else”

**Hub Protocol** (`/api/hub/*` on the pod) is the **main HTTP contract** for Intelligence ↔ Pod traffic and for **external agents** (other AIs, scripts, bridges). It keeps access **Bearer-key–scoped** and workspace-aware, instead of scraping the web app.

- **[Integrations overview](/docs/integrate/integrations/overview)** — skill invocation, chat stream, SDK / tRPC / Hub REST (options C–E)  
- **[AI agents & external tools](/docs/start/getting-started/ai-agents-and-tools)** — mental model in one page

## Channels & external streams

**Channels** are where conversation and many **external imports** land — alongside entities and views they complete the product triptych. See **[Channels](/docs/architecture/concepts/channels)**.

**RSS and feeds:** the pod can ingest **RSS/Atom** (and, in hosted stacks, **RSSHub via the control-plane proxy**) into feed jobs that become messages you can route and reason about. Deep architecture (executors, CP proxy): **[Unified feed](/team/platform/unified-feed)** · self-hosters: backend repo **`docs/RSS-SETUP.md`** (operators) and **`docs/FEED-API.md`** (HTTP surface).

## Templates: ship a whole app shape on top of the system

Workspace **templates** (presets, sidebar, bento, profiles) let you **stand up a full application pattern** — CRM, second brain, etc. — from a definition rather than clicking every entity into existence. See **[Roadmap / templates](/docs/start/strategy/roadmap)** and how templates wire relationships in **[Entity connections](/docs/architecture/concepts/entity-connections)**. **[Extending Synap](/docs/integrate/development/extending/overview)** covers internal vs remote vs hybrid modules.

## OpenClaw, CLIs, and skills (concrete path)

| Tool | Role |
|------|------|
| **`npx @synap/cli init`** | Guided **agent bridge** — point OpenClaw (or similar) at your pod, API keys, laptop flows. |
| **Backend `synap` script** | **Pod lifecycle** when self-hosting from the backend repo — install, update, migrate (Docker). |
| **Synap skill (`skills/synap/`)** | OpenClaw-facing skill that uses **Hub only** — no direct DB access from the agent. |
| **`deploy/setup-openclaw.sh`** | Self-hosted: provisioning token → agent user + API key on the pod. |

Details: **[AI agents & external tools](/docs/start/getting-started/ai-agents-and-tools)**. Skills exposed for HTTP invocation: **[Skill invocation](/docs/integrate/integrations/skill-invocation)**.

## Governance (don’t skip this)

User mutations and **agent** mutations are not the same code path: high-impact agent work goes through **proposals** where policy requires it. When you integrate, **do not bypass** `checkPermissionOrPropose()` patterns — see **[Contributing](/docs/contributing)** and product docs on governance.
