# Decisions

**Target Users**: Engineering leads, product managers, architects, team members making cross-cutting decisions  
**Marketing**: Decision transparency — why we chose what we chose, what we considered, and how we'll know if we were right  
**Key User Stories**:
- As an engineer, I want to understand the rationale behind past decisions so I don't repeat mistakes or re-debate settled questions
- As an architect, I want a structured ADR format so I can document cross-service decisions with clear context and trade-offs
- As a team member, I want to track decision status and review dates so old decisions stay current
**Key Features**: ADR template, decision log, quality checklist, cross-referencing  
**Current Status**: Template and log active; decision log empty — use it

## Overview

This space exists to document **why** a decision was made, **what alternatives** were considered, and **how we'll measure** whether it was correct. It covers architecture decisions with cross-service impact, product scope and prioritization, operational policy changes, and GTM or positioning shifts that affect execution.

The space is organized into three pages:

1. **Index** — scope, quality checklist, how to use this space
2. **ADR Template** — structured format for writing decisions
3. **Decision Log** — chronological index of accepted and proposed decisions

## Architecture

Synap's decision process follows an **ADR (Architecture Decision Record)** pattern:

```
Problem/Context → Alternatives Evaluated → Decision Made → Trade-offs → Rollout → Validation
```

Every decision is held to a **quality checklist**:
- Context is explicit
- Alternatives are listed
- Trade-offs are concrete
- Owner and review date are set
- Follow-up metrics are defined

Decisions can be in one of four states: **Proposed** → **Accepted** → **Superseded** (by a later ADR) or **Rejected**.
