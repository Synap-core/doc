## 1. TL;DR

`@synap-core/auth` is now a public, MIT-licensed npm package (`@synap-core/auth@1.0.0`). It is the single auth foundation for **every** Synap-compatible app — Hub, Studio, Canvas, CRM, DevPlane, the-arch, and Eve. Sign in once on any of them and:

- A shared session is written to `localStorage["synap:session"]` plus a cross-subdomain cookie at `Domain=.synap.live`.
- Every other app's `SynapProvider` boot effect reads that session via `getSharedSession()` and the user is signed-in there too — no second prompt, no postMessage handshake.
- A `storage` event listener in each provider propagates sign-out across tabs.

There are exactly **two valid auth modes**:

- **Synap-hosted** (default) — sign in with a Synap CP account, then connect to a pod via the CP handshake. Marketplace, cross-pod sharing, and entitlements work.
- **Self-hosted** — bypass the CP entirely, sign in with pod-local Kratos credentials. Apps that depend on the CP gracefully degrade.

Eve is the same shape as the other apps with **one extra step**: after a successful CP sign-in it mirrors the session into `~/.eve/secrets.json` so the on-host CLI / daemon can act as the user without prompting again.

## 2. Architecture

```
                   ┌────────────────────────────────────────────────┐
                   │             User's browser (one tab)           │
                   │                                                │
   sign in once ──►│ ┌─────┐  ┌────────┐  ┌─────┐  ┌─────┐  ┌─────┐ │
                   │ │ Hub │  │ Studio │  │ CRM │  │ Eve │  │ ... │ │
                   │ └──┬──┘  └───┬────┘  └──┬──┘  └──┬──┘  └──┬──┘ │
                   │    └─────────┴──────────┴────────┴────────┘    │
                   │                  @synap-core/auth              │
                   │                                                │
                   │  storeSharedSession()  /  getSharedSession()   │
                   │                                                │
                   │  ┌──────────────────────────────────────────┐  │
                   │  │  localStorage["synap:session"]           │  │
                   │  │  cookie "synap-session" Domain=.synap.live  │
                   │  └──────────────────────────────────────────┘  │
                   │                       │                        │
                   │              storage event │  cross-tab clear  │
                   └───────────────────────┼────────────────────────┘
                                           │
                                           ▼ (Eve only)
                                ┌──────────────────────┐
                                │ POST /api/auth/sync  │
                                │ owner-only writer    │
                                └──────────┬───────────┘
                                           ▼
                                ~/.eve/secrets.json
                                  cp.userSession
```

A second user in a separate browser hits the same server and is rejected from the disk slot with `{ ok: true, skipped: "host-owner-already-set" }` — their browser session still works, the disk just stays bound to the first signed-in user. See §6.

## 3. Credential Layers

There are three storage layers. Each answers a different question and they don't substitute for each other.

| Layer | Storage | Per-user scope | Holds | Lifetime |
|---|---|---|---|---|
| Browser session | `localStorage["synap:session"]` + cookie `synap-session` at `.synap.live` | Per-browser (each operator's own browser has their own) | CP Bearer token, user identity, pod URL, active workspace | Until sign-out or token expiry |
| Pod session | Kratos session minted on demand via [JWT-Bearer exchange](./eve-credentials.mdx#4-how-poduser-token-is-minted-rfc-7523-jwt-bearer-grant) | Per-user-per-pod | Pod auth token used on `/trpc/*` | Re-mintable silently from a valid CP session |
| Host secrets | `~/.eve/secrets.json` (mode 0600) | **Owner only** — first signed-in user wins, others no-op | `cp.userSession`, `pod.userToken`, `agents.eve.hubApiKey`, Eve issuer keypair | Persistent on disk |

The first two exist on every app that uses `@synap-core/auth`. The host layer is **Eve-only** because Eve is the only Synap surface running on the operator's box with on-host CLI/daemon clients that need to act without a browser tab.

## 4. Mode A — Synap-hosted (default)

```
operator                Eve UI / Hub          api.synap.live           pod.<domain>
   │                        │                      │                        │
   │ 1. email + password    │                      │                        │
   ├───────────────────────►│  signInToCP() via    │                        │
   │                        │  @synap-core/auth    │                        │
   │                        ├─────────────────────►│                        │
   │                        │ ◄── CPSession ───────┤                        │
   │                        │                      │                        │
   │                        │ storeSharedSession() │                        │
   │                        │ writes localStorage  │                        │
   │                        │ + .synap.live cookie │                        │
   │                        │                      │                        │
   │                        │ (Eve only)           │                        │
   │                        │ POST /api/auth/sync  │                        │
   │                        │ owner-gate → disk    │                        │
   │                        │                      │                        │
   │ 2. pick a pod          │                      │                        │
   ├───────────────────────►│ POST /pods/handshake │                        │
   │                        ├─────────────────────►│                        │
   │                        │ ◄── ES256 JWT ───────┤                        │
   │                        │                      │ POST /api/handshake    │
   │                        ├──────────────────────────────────────────────►│
   │                        │ ◄── PodSession ────────────────────────────────┤
   │                        │                      │                        │
   │                        │ storeSharedSession() │                        │
   │                        │ now carries podUrl + workspaceId              │
```

This is the `apps/hub/lib/auth.ts` flow, mirrored verbatim by `apps/studio`, `apps/canvas`, `apps/crm`, `apps/devplane`, `apps/base`, `the-arch`, and `eve-dashboard`. The shared bits — `signInToControlPlane`, `signUpToControlPlane`, `verifyCPTotp`, `fetchUserPods`, `handshakeConnectToPod` — all live in `@synap-core/auth` and are wrapped by each app's `lib/auth.ts` for app-specific persistence (e.g. Hub's `synap-cp-session` cookie for the platform proxy).

## 5. Mode B — Self-hosted (CP bypass)

For operators who don't want a Synap account at all. The Eve sign-in screen surfaces it as a top-level "Self-hosted" tab; the same flow is reachable from `PodConnectGate`'s "Claim this pod" CTA.

```
operator           Eve UI               pod.<domain>
   │                  │                       │
   │ pod URL +        │                       │
   │ Kratos email/pw  │ POST /api/pod/        │
   ├─────────────────►│ bootstrap-claim       │
   │                  ├──────────────────────►│
   │                  │ ◄── invite + Kratos ──┤
   │                  │                       │
   │                  │ storeSelfHostedSession(podUrl, email)
   │                  │   writes synap:session with mode: "self-hosted"
   │                  │   sessionToken = "" (no CP token to share)
```

The shared session is still written so other Synap surfaces on the same machine see "an operator is signed in," but the `mode: "self-hosted"` marker (narrowed via `isSelfHostedSession`) lets gates downgrade the experience:

- Marketplace tabs render a "Requires a Synap account" hint instead of catalog data.
- Cross-pod sharing UIs hide the share button.
- Anything that only needs `pod.userToken` (data, agents, governance) works exactly the same.

This is intentional: **self-hosted is not a CP-with-a-fake-token** — it is "no CP, period." Apps that want a CP-only feature must check `getSharedSession()?.sessionToken !== ""` before using it.

## 6. Multi-User Behaviour

### Browser

Browser sessions are **per-user-per-browser**. Two operators sharing one machine but using separate browsers (or separate browser profiles) each have their own `synap:session`. There is no global "current user" — the session is whatever localStorage holds for that browser context.

### Pod

The pod-signin route (`/api/auth/pod-signin`) mints a Kratos session for any user the JWT-Bearer exchange resolves. Non-owner users get the session **in the response body only** — it is not written to disk. This means:

- Owner's browser → pod token is cached on disk for the on-host CLI.
- Teammate's browser on the same Eve → pod token works for the duration of the tab, but the on-host CLI cannot impersonate them.

This matches the file-permission model: `~/.eve/secrets.json` is `0600` and tied to the host owner uid. Persisting another user's token there would silently grant the owner access to that user's pod identity.

### Host owner (Eve only)

The "host owner" is whoever the disk slot is bound to. Defined operationally:

> The first user to complete a CP sign-in through this Eve dashboard. Subsequent users sign in normally in their own browser; their session stays in their browser only. The disk slot does not rotate.

The owner gate is enforced server-side in `app/api/auth/sync/route.ts`:

```
if (existing && existing.userId && existing.userId !== incoming.userId) {
  return NextResponse.json({ ok: true, skipped: "host-owner-already-set" });
}
```

This is intentionally a no-op (200 OK), not an error — the second user's browser flow still proceeds successfully; only the host mirror is skipped.

## 7. Cross-App SSO Mechanics

### Storage layout

```ts
// localStorage
"synap:session" -> JSON { podUrl, sessionToken, workspaceId, userId, userName, podId?, tier? }

// cookie at .synap.live (production) / no domain (localhost)
synap-session    -> same JSON, URL-encoded
SYNAP_INSTANCE_URL -> podUrl only (used by server proxies that need to route)
```

The cookie domain is computed dynamically in `cookieDomainAttr()`:

- `localhost` / `127.0.0.1` → no Domain attribute (browsers treat it as host-scoped but port-agnostic, so :3020 / :3002 / :3030 all share).
- `*.synap.live` or `synap.live` → `Domain=.synap.live`.
- Anything else (custom Eve domain, tunnel) → no Domain (host-only, the cookie still works for that single host).

`Secure` is added when `location.protocol === "https:"`.

### Read order

`getSharedSession()` is fault-tolerant on purpose:

1. Read `localStorage["synap:session"]`. Fast path; same app or already migrated.
2. Read each legacy key in `LEGACY_KEYS` (`hub:session`, `studio:session`, `crm:session`, `devplane:session`, `base:session`). On hit, promote to `synap:session`, write the cookie, drop the legacy key. This is the migration path for users with pre-1.0 sessions.
3. Read the cookie `synap-session`. This is the cross-app fallback — another Synap surface on the same domain wrote the cookie, our localStorage doesn't have it yet, we hydrate ours from the cookie.

### Cross-tab sign-out

Every `SynapProvider` registers a `storage` listener:

```ts
useEffect(() => {
  function handler(e: StorageEvent) {
    if (e.key !== "synap:session" || e.newValue !== null) return;
    setSession(null);
    setStage("sign-in");
  }
  window.addEventListener("storage", handler);
  return () => window.removeEventListener("storage", handler);
}, []);
```

Note the asymmetry: we only react to **clear** (newValue === null), not to writes. Sign-in propagation already happens via the `.synap.live` cookie and is picked up on the next provider mount in each tab. Reacting to writes would force every open tab to swap users mid-session, which breaks more than it fixes.

## 8. Token Entry Surfaces

There are exactly **three** legitimate places the user types or sees a token. Anything else is a smell — file an issue.

1. **Sign-in screen "Self-hosted" tab.** Pod URL + Kratos email/password. The only place where the user explicitly opts out of the CP path. Implemented by `EveSignInScreen` (Eve) and the equivalent self-host card on Hub/Studio.
2. **Settings → Advanced → "Connect remote pod"** *(deferred)*. Pod URL + credentials/handshake. Lets an operator point an existing Eve at a pod they didn't bootstrap from this Eve. Currently not implemented; tracked under §10.
3. **Settings → Pod → Agent API key viewer.** Read-only display of the `agents.eve.hubApiKey` slot, plus a "Rotate" button. For OpenClaw / external skills that need a Hub key. Never used to authenticate the dashboard itself.

NOT legitimate:

- Bootstrap email entry on a fresh Eve. The bootstrap flow goes through `/api/admin/bootstrap/claim` with the deploy-side `ADMIN_BOOTSTRAP_TOKEN` and a Kratos signup link — no token is ever pasted into the dashboard.
- Pasting a pod token into normal flow. The JWT-Bearer exchange (see [eve-credentials.mdx §4](./eve-credentials.mdx#4-how-poduser-token-is-minted-rfc-7523-jwt-bearer-grant)) makes this unnecessary.
- Manually editing `synap:session` or the `.synap.live` cookie. If you find yourself doing this in a debugger, the bug is in the auth client.

## 10. Open Follow-Ups

Tracked but not shipped today:

- **Settings → Advanced → "Connect remote pod"** UI (token-entry case 2). Today an operator who already has Eve installed cannot point it at a pod they didn't bootstrap through Eve; they have to re-run the sign-in flow.
- **Hub's `synap-cp-session` cookie is still per-app** — it is kept because the Next API proxy at `/api/platform/[...path]` reads it server-side to forward the CP Bearer. Long-term, switch the proxy to read the CP token from the shared session server-side and drop this cookie. Low priority.
- **The CP cookie is not shared across `.synap.live`.** Each app fetches a fresh CP session from the shared-session token if it needs one, so this is a non-issue in practice. Logged here so nobody re-discovers it as a bug.

## 11. Glossary

- **`synap:session`** — The single localStorage key every Synap-compatible app reads on boot. Contains the shared session JSON. Legacy per-app keys (`hub:session`, `studio:session`, ...) are migrated transparently on first read.
- **`synap-session` cookie** — Cross-subdomain mirror of the same JSON. `Domain=.synap.live` in production, no Domain on localhost (port-agnostic). Set by `writeCookie()` in `shared-session.ts`.
- **Synap-hosted mode** — The user has a Synap CP account at `api.synap.live`. CP marketplace, entitlements, cross-pod sharing all work.
- **Self-hosted mode** — The user has a pod but no CP account. Stored as `mode: "self-hosted"` on the shared session. CP-only features degrade gracefully.
- **Host owner** — On Eve, the first user to complete CP sign-in through this dashboard. The only user whose CP session is mirrored to `~/.eve/secrets.json`. Subsequent users sign in normally; their session lives in their browser only.
- **`cp.userSession`** — The slot in `~/.eve/secrets.json` (Eve only) that mirrors the host owner's CP session. Used by the on-host CLI / daemon to act as the user without re-prompting. Single-slot, owner-only, mode 0600.
