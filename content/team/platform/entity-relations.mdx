## Two Linking Systems

Synap has two orthogonal ways to link entities. Use the right one for the right job.

```bash

┌──────────────────────────────┬────────────────────────────────────────────────────────┐
│     STRUCTURAL LINKS         │           SEMANTIC RELATIONS                           │
├──────────────────────────────┼────────────────────────────────────────────────────────┤
│ Table: entity_property_index │ Table: relations                                       │
│ Defined in profile schema    │ Created on-the-fly (user, AI, automation)              │
│ One-directional              │ Bi-directional, traversable                            │
│ e.g. Task.assignee = person  │ e.g. Note "relates_to" Person                         │
│ Form-based UI, schema-first  │ Free-form, graph-first                                 │
│ No metadata                  │ JSONB metadata (status, scores, timestamps)            │
│ Not traversable by graph     │ Powers the knowledge graph + IS graph tools            │
└──────────────────────────────┴────────────────────────────────────────────────────────┘

```

**Rule of thumb:**  
- Predefined schema links (Task.assignee, Deal.contact) → **structural** (entity_id property)  
- Ad-hoc connections, notes about people, campaign contacts, graph edges → **semantic** (`relations`)

The `getConnections()` procedure returns BOTH unified in one response.

## Semantic Relations — Complete Model

### Table: `relations`

```
id            uuid PK
userId        text  NOT NULL          — creator
workspaceId   uuid  NOT NULL          — workspace scope
sourceEntityId uuid FK → entities.id  — origin
targetEntityId uuid FK → entities.id  — destination
type          text  NOT NULL          — relation type slug
metadata      jsonb default '{}'     — extensible (status, scores, etc.)
createdAt     timestamp
```

**Indexes (migration 0063):**
```sql
relations(source_entity_id, workspace_id)  — getRelated(), getConnections()
relations(target_entity_id, workspace_id)  — incoming traversal
relations(type, workspace_id)              — type-filtered queries
relations(user_id, workspace_id)           — user-scoped listing
```

### Table: `relation_defs`

Workspace-scoped type catalog. Every semantic relation type is registered here.

```
id, workspaceId, userId
slug          text UNIQUE per workspace  — "relates_to", "assigned_to"
displayName
isDirectional boolean                    — whether A→B ≠ B→A
uiHints       jsonb                      — category, color, icon, inverse labels
```

### Table: `profile_relations`

Optional schema-level constraints: "contacts can work_at companies."  
Junction: `(sourceProfileId, targetProfileId, relationDefId)`.  
Not yet exposed in UI — exists for power-user validation.

## tRPC API (relations router)

Mounted at `trpc.relations.*`:

| Procedure | Input | Notes |
|-----------|-------|-------|
| `listTypes()` | — | All `relation_defs` for workspace, grouped by category |
| `list()` | type?, limit, offset | Workspace-wide relation list |
| `get()` | entityId, type?, direction | Raw relation records |
| `getRelated()` | entityId, type?, direction | Resolved entity objects |
| `getStats()` | entityId | Outgoing/incoming/total counts by type — uses SQL COUNT |
| `getConnections()` | entityId, limit | **Unified**: graph + structural + threads |
| `create()` | sourceEntityId, targetEntityId, type, metadata?, workspaceId? | Governance-gated |
| `update()` | id? OR (sourceEntityId + targetEntityId + type), metadata?, workspaceId? | ← ADDED 2026-04-07 |
| `delete()` | id, workspaceId? | Governance-gated, reverse-syncs properties |
| `backfill()` | workspaceId | One-time job to backfill from entity_id properties |

### The `update()` signature

The Relay app doesn't always have the relation `id` — it identifies by `(sourceEntityId, targetEntityId, type)` triple. `update()` accepts both:

```typescript
// By ID (canonical)
await trpc.relations.update.mutate({ id: 'rel-uuid', metadata: { status: 'reached' } })

// By triple (Relay campaign contacts)
await trpc.relations.update.mutate({
  sourceEntityId: campaignId,
  targetEntityId: personId,
  type: 'target_contact',
  metadata: { reached: true, reachedAt: new Date().toISOString() }
})
```

## Property ↔ Relation Sync

`property-relation-sync.ts` keeps structural links and semantic relations in sync:

- **On create**: If the relation type maps to an `entity_id` property via `profile_relations.propertyDefId`, auto-sets that property on the source entity.
- **On delete**: Auto-clears the corresponding property.

This means you can model the same link two ways and they stay consistent. Example: Task has a structural `assignee` property (entity_id). When a `assigned_to` relation is created Task→Person, the `assignee` property is auto-set. When the relation is deleted, the property is cleared.

## The "Add Note to Person" Flow

### Option A — Two calls (explicit, most flexible)

```typescript
// 1. Create the note
const { id: noteId } = await trpc.entities.create.mutate({
  profileSlug: 'note',
  title: 'Meeting notes - John',
  properties: { content: '...' }
})

// 2. Link it
await trpc.relations.create.mutate({
  sourceEntityId: noteId,
  targetEntityId: personId,
  type: 'relates_to'
})
```

### Option B — Capture pipeline (one call, atomic)

Best for AI-driven flows or when creating multiple entities + links at once:

```typescript
await trpc.capture.execute.mutate({
  entities: [
    { tempId: 'note', profileSlug: 'note', title: '...', properties: { content: '...' } },
    { tempId: 'person', existingEntityId: personId }   // reference existing, don't create
  ],
  relations: [
    { sourceTempId: 'note', targetTempId: 'person', relationType: 'relates_to' }
  ]
})
```

### Option C — With relation metadata (Relay campaigns)

```typescript
await trpc.relations.create.mutate({
  sourceEntityId: campaignId,
  targetEntityId: personId,
  type: 'target_contact',
  metadata: { status: 'not_reached', reached: false, addedAt: new Date().toISOString() }
})

// Later, update status
await trpc.relations.update.mutate({
  sourceEntityId: campaignId,
  targetEntityId: personId,
  type: 'target_contact',
  metadata: { reached: true, reachedAt: new Date().toISOString() }
})
```

## Relay App Usage (Real-World Examples)

### Campaign contacts (`useCampaignDetail.ts`)

Campaign entity links to Person entities via `target_contact` relation. The relation metadata stores the contact's pipeline status:

```yaml

Campaign ──target_contact──▶ Person
                    metadata: {
                      status: 'not_reached' | 'reached' | 'interested' | 'converted',
                      reached: boolean,
                      reachedAt?: ISO string
                    }

```

### Workspace automations (`relay-automations.ts`)

Automations 1 and 2 update entity properties directly via `entity_update` output nodes. No relations are created by automations currently (though the `create_relation` IS tool is available for custom flows).

## Future Roadmap

### Short-term (blocks current features)

1. ~~**Note sync in Relay**~~ — ✓ DONE (2026-04-07). Notes saved as `note` entities linked via `relates_to`. AsyncStorage migration included.
2. **Person timeline** — add `chat.getMessagesForEntity(entityId)` endpoint

### Medium-term (improves power)

3. **Custom relation type creation in UI** — `relation_defs.create` tRPC procedure exists, no UX for it yet
4. **Profile-level relation constraints** — `profile_relations` table is populated but invisible to users. UI in profile editor: "This profile can be linked to: [Company] via [works_at]"
5. **Relation batch create** — `capture.execute` already supports it; expose a dedicated `relations.createBatch()` for performance-critical flows
6. **Relay `as any` cleanup** — `useCampaignDetail.ts` and other hooks call relations procedures via `(client as any)`. Should use typed `@synap/synap-client` types.

### Long-term (graph power)

7. **`traverseGraph()` depth-3 BFS** — already implemented, not exposed in any UI yet. Powers "2-hop connection" features ("you know X who knows Y")
8. **Graph view** — relations table has all the edges. Missing: a graph cell type that visualizes them with React Flow or d3-force
9. **AI relation inference** — IS creates `mentioned_in` / `discussed_with` / `works_on` relations automatically during conversation, all via the `create_relation` tool + proposals
