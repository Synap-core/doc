# Relay — Next-Gen Capabilities Ship Plan

**Date:** 2026-04-10
**Horizon:** 3 sprints over ~6 weeks, then stabilization. Targets iOS 18 + Android 14.
**Companion docs:** `NEXT-GEN-IOS-PLAN.md`, `NEXT-GEN-ANDROID-PLAN.md`, `RELAY-NEXT-GEN-CAPABILITIES.md`.

This is the ordered execution plan for the six next-gen capabilities, organized into three sprints plus a foundations sprint. Each capability is sized on both platforms with effort, files, dependencies, minimum viable shape, power shape, and architectural prerequisites.

---

## Overall shape

```
Sprint 0 (3 days)    — Foundations: App Group, entity IDs, shared storage module
Sprint 1 (2 weeks)   — App Intents / App Actions + Spotlight / AppSearch + Share
Sprint 2 (2 weeks)   — Widgets + Live Activities / Live Updates
Sprint 3 (2 weeks)   — Vision / Live Text / ML Kit + polish + power-mode upgrades
```

Total engineering budget: **6 weeks × 1 engineer, or 3 weeks × 2 engineers** (one per platform with one week for cross-platform glue + JS).

---

## Sprint 0 — Foundations (3 days, blocking everything else)

Not a product feature. Pure architectural prerequisite work.

### Tasks

| Task | Effort | Owner | Files |
|---|---|---|---|
| App Group entitlement `group.com.synap.relay` | 0.5d | iOS | `app.json` `ios.entitlements`, Apple Dev Portal |
| Shared Keychain access group for auth token | 0.5d | iOS | `src/lib/auth.ts`, Keychain access group config |
| `canonicalRelayUrl(entity)` helper + route table | 0.5d | Both | `src/lib/deep-links.ts` |
| `modules/relay-shared-storage` Expo module (iOS App Group UserDefaults, Android SharedPreferences) | 1d | Both | `modules/relay-shared-storage/{ios,android,src}` |
| `modules/relay-vision`, `modules/relay-intents`, `modules/relay-appsearch`, `modules/relay-live-updates` scaffolds | 0.5d | Both | `modules/` skeletons via `npx create-expo-module --local` |

### Why this sprint is non-negotiable
Every capability in Sprint 1-3 writes or reads cross-process state. Without the shared storage module, Widgets can't see feed data, Share Extension can't persist captures before the main app reads them, Live Activities can't be updated from the Share Extension, and App Intents can't pass parameters to the JS app. This is the single load-bearing piece.

### Shipped artifacts
- App builds on iOS and Android with new entitlements
- `sharedStorage.getString/setString/getJSON/setJSON` works from both main app and future extension targets
- Every entity has a stable `relay://{kind}/{id}` URL

## Sprint 1 — Zero-UI capture + searchable content (2 weeks)

**Theme:** make Relay capture-able and findable without opening the app.

Three capabilities in parallel: App Intents + App Actions (voice-driven capture), Spotlight + AppSearch (searchable content), Share Extension + Share Target (share-driven capture). All three ship together because they share the same `AppEntity` / `IndexedEntity` declarations on iOS and the same `canonicalRelayUrl` routing on both platforms.

### 1A. App Intents (iOS) + App Actions (Android)

| | iOS | Android |
|---|---|---|
| **Effort** | 3 days | 2 days |
| **Files** | `targets/RelayIntents/*.swift`, `modules/relay-intents/` | `android-plugin/shortcuts.xml`, `android/app/.../CreateNoteActivity.kt` + 2 more Activities |
| **Dependencies** | Sprint 0 (App Group, stable IDs, shared storage) | Sprint 0 |
| **Expo plugin?** | None — custom via `@bacons/apple-targets` + local module | None — custom local config plugin |
| **JS API** | `subscribeIntents(cb)` + `donate(kind, params)` | Same interface on Android via DeviceEventEmitter |
| **Unlocks** | Siri voice capture, Shortcut automations, Spotlight Actions, iOS 18 AppShortcuts, widget button targets | Google Assistant + Gemini Assistant voice, long-press launcher shortcuts, Dynamic Shortcuts |

**Minimum viable:** one intent — `CreateNoteIntent(title: String)` — that writes to shared storage + posts a Darwin/broadcast notification. "Hey Siri, new Relay note about X" creates the note on next app open. Shipped on both platforms = 2 days.

**Power version:** four intents (Note / Task / Person / Capture), parameter resolution against existing entities (`EntityQuery` on iOS / BII slot filling on Android), donation on every in-app capture, AppShortcutsProvider with localized phrases, `openAppWhenRun = false` so capture is fully silent. **This is the sprint deliverable**.

**Architectural prerequisite:** stable entity ID scheme + canonical URL (done in Sprint 0).

| | iOS | Android |
|---|---|---|

**Architectural prerequisite:** unified `IndexableEntity` type in `@synap-core/types` (one day of plumbing — see cross-cutting work below).

## Sprint 2 — Ambient state + glanceable surfaces (2 weeks)

**Theme:** make Relay visible at a glance, and make async AI visible while it's working.

### 2A. Widgets (iOS WidgetKit) + Glance (Android)

| | iOS | Android |
|---|---|---|
| **Effort** | 3-4 days | 3-4 days |
| **Files** | `targets/RelayWidgets/*.swift` (Quick Capture, Today's Captures, Space Switcher, Voice Capture) | `android/.../widgets/*.kt` same four widgets |
| **Dependencies** | App Intents (Sprint 1) — widget buttons ARE intents | Sprint 1 Activities + local config plugin |
| **Expo plugin?** | `@bacons/apple-targets` generates target scaffold; SwiftUI by hand | Custom plugin — `@bittingz/expo-widgets` is pre-Glance, not worth it |
| **JS API** | `sharedStorage.setJSON('widget-state', state)` + `WidgetCenter.reload` via native side-effect | Same + `GlanceAppWidgetManager.updateAll` |
| **Unlocks** | Home screen quick capture, Today's feed at a glance, Space switching without app, voice capture entry point | Same |

**Minimum viable:** Quick Capture widget only. One tap → CaptureModal opens. 1.5 days each platform.

**Power version:** four widgets per platform. Today's Captures shows last 3 entities with thumbnails and tap-to-open. Space Switcher is a 3-button medium widget backed by shared storage. Voice Capture button starts a Live Activity (iOS) or foreground service (Android). Material You tinting on Android. Lock Screen widgets on iOS. **Sprint deliverable.**

**Architectural prerequisite:** `widget-state.json` written by JS on feed mutation via shared storage module. Without it, widgets are stuck on the 15-minute timeline budget.

**Total Sprint 2 effort:** iOS 6 days + Android 6 days + 1 day glue = **~13 engineer-days.** Two engineers in parallel = 1 week each, plus 1 week testing across device matrix.

## Cross-platform shape

**JS can be shared:** capture hooks, shared storage API, deep-link routing, indexable entity types, recording state hooks, widget data contract, OCR result types.

**JS cannot be shared:** any code that touches a native target (Swift in `targets/`, Kotlin in `android/.../widgets/`). This is expected — extensions are native-only processes.

Roughly **80% of the JS-level code is shared; 0% of the native target code is shared**. Surface area is small (maybe 2000 lines of shared JS) because most of the value is in the native targets.

### Package reuse
- `@synap-core/types` — add `IndexableEntity` and `LiveUpdateState` types once, both platforms consume
- `@synap-core/capture-pipeline` — already handles most of the capture logic; extensions feed into it
- `modules/relay-*` Expo local modules — the bridge layer

### Ecosystem strength by capability
From strongest Expo plugin support to weakest:

1. **Share / Share Target** — `expo-share-intent` + `expo-share-extension` are production-ready.
2. **Vision / Text Recognition** — `react-native-vision-camera-ocr-plus` is the substrate everyone uses.
3. **Spotlight (iOS)** — `expo-spotlight` is an Expo-native module, works today.
4. **Widgets (iOS)** — `@bacons/apple-targets` is battle-tested but you still write SwiftUI by hand.
5. **Live Activities (iOS)** — `expo-live-activity` is Software Mansion-backed but new.
6. **App Intents (iOS)** — `@bacons/apple-targets` gets you the target; Swift code is still yours.
7. **Glance Widgets (Android)** — no good Expo plugin; custom local plugin required.
8. **AppSearch (Android)** — no plugin at all; custom Kotlin module required.
9. **App Actions (Android)** — no plugin; custom local plugin for `shortcuts.xml`.
10. **Live Updates (Android)** — `@notifee/react-native` is the closest; still need custom code for A14 chip promotion.

**Surprising finding:** Android is the *weaker* ecosystem slot for 3 of the 6 capabilities despite Android's generally simpler native model. The React Native + Expo community has invested more in iOS extensions because iOS extensions are harder to build manually, so the demand for abstractions is higher. On Android, devs just drop into Kotlin.

## Prioritized one-line summary

| Capability | iOS priority | Android priority | Combined recommendation |
|---|---|---|---|
| App Intents / App Actions | **P0** | **P0** | **Sprint 1 — foundational for everything else on iOS** |
| Spotlight / AppSearch | **P0** | P1 | **Sprint 1 — shares code with App Intents** |
| Share Extension / Share Target | **P0** | **P0** | **Sprint 1 — cheapest, highest daily-frequency value** |
| Widgets / Glance | P1 | P1 | Sprint 2 — depends on App Intents already landing |
| Live Activities / Live Updates | P1 | P1 | Sprint 2 — premium feel for voice capture |
| Vision / ML Kit | P1 | P1 | Sprint 3 — standalone; can slip without blocking anything |

**If we can only ship three:** App Intents + Spotlight + Share on both platforms. These are the foundation for Siri/Assistant discovery, system-wide search, and the highest-frequency daily capture path. Widgets without App Intents is an island; Live Activities without Widgets is a gadget; Vision without the rest is a side feature. Start with the base.

## Risks and blockers

1. **Apple Developer Portal access** for App Group + access group entitlement — if we don't have admin on the team account, Sprint 0 slips.
2. **`expo-spotlight` maturity** — if the package doesn't support iOS 18 semantic search attributes, we fork. Add half a day to Sprint 1.
3. **Android 14 foreground service types** — Play Store now rejects apps that use `foregroundServiceType` without a declared user-visible reason. The `microphone` type is fine; the `shortService` path for AI structuring needs a fallback for users on Android 13.
4. **`@bacons/apple-targets` is still marked experimental** — API could change across updates. Pin to an exact version, don't auto-upgrade.
5. **New Architecture (Fabric + TurboModules)** — Relay has `newArchEnabled: true`. Most community packages listed above support it but confirm before install; the riskiest is `react-native-vision-camera-ocr-plus` which depends on `react-native-worklets-core`.
6. **Shared code vs shared types** — JS types should live in `@synap-core/types`; native targets cannot import TS types. Manually keep the contracts in sync via codegen or comments.
