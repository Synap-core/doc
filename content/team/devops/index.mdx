# DevOps

**Target Users**: New engineers, DevOps operators, self-hosters, contributors running full-stack environments  
**Marketing**: Developer experience — frictionless local setup, clear ops runbooks, production-grade security posture  
**Key User Stories**: 
- As a new engineer, I want to get all services running locally in under 15 minutes so I can start contributing immediately
- As a DevOps operator, I want standardized deployment, backup, and incident workflows so I can maintain reliability
- As a self-hosted user, I want Docker-based installation with clear architecture docs so I understand my deployment
**Key Features**: Progressive setup paths (local → hybrid → remote), Docker Compose orchestration, secret management, rolling updates, AI governance audits, auth consolidation  
**Current Status**: Core setup docs active; security audit findings documented; auth consolidation phase 1 recommended

## Overview

This space covers everything needed to run, deploy, and maintain Synap infra­structure — from onboarding a new developer on a weekday morning to operating production pods at scale. It is organized into four areas:

1. **New Developer Onboarding** — progressive paths from zero to running the full stack
2. **Operations** — provisioning, monitoring, updates, backups, incident response
3. **Security** — hardening practices, governance audits, authentication model
4. **Infrastructure** — Docker architecture, backlink strategy

## Architecture

Synap's development and deployment stack follows a clear separation:

```
Frontend (Next.js / Electron) → Backend (Hono API) → Services (Docker)
                                    │                    │
                              env vars             PostgreSQL, Kratos,
                              for connections      Redis, MinIO, Typesense
```

Key principles:
- **One backend URL** for the frontend — all requests proxy through a single Hono API
- **Env var driven** — every service connection is configured via environment variables
- **No Docker locally for the API** — developers run the API process directly, connect to remote services
- **Progressive setup** — start local, gradually migrate to remote as needed
