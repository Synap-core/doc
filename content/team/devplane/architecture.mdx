## Data flow

```
React component
  ‚Üí lib/devplane/hooks.ts         (useDevplaneEntities, useCreateDevplaneEntity‚Ä¶)
      ‚Üí @synap-core/client        (tRPC + TanStack Query)
          ‚Üí Next.js /trpc proxy
              ‚Üí Synap backend pod :4000/trpc
                  ‚Üí Drizzle ORM ‚Üí PostgreSQL
```

Components never call tRPC directly ‚Äî always through a hook in `lib/devplane/hooks.ts`. This is the three-layer rule.

## Provider tree

Split into two tiers. The first tier (in `app/providers.tsx`) is always active. The second tier (in `app/(app)/layout.tsx`) only mounts after workspace readiness is confirmed.

```
providers.tsx (always active):
  NextThemesProvider
    HeroUIProvider
      SynapProvider              ‚Üê auth context
        AppStoresSync            ‚Üê Zustand hydration
          AppTRPCProvider        ‚Üê tRPC client
            RegistryBoot
              CellRuntimeWrapper
                CommandPaletteProvider
                  {children}

app/(app)/layout.tsx (gated ‚Äî only when state === "ready"):
  AppSocketProvider              ‚Üê Socket.IO ‚Üí pod
    RealtimeBridge               ‚Üê cache invalidation via socket events
      AppShell + page content
```

**Why Socket.IO is gated:** tRPC requests go through the Next.js `/trpc` proxy, but Socket.IO connects directly to the pod URL (e.g. `antoine-perso.synap.live`). Connecting before the workspace is confirmed ready produces cross-origin WebSocket errors with no benefit. The readiness gate ensures Socket.IO only connects once the app is fully operational.

## Startup state machine

`hooks/useWorkspaceReadiness.ts` runs inside `WorkspaceReadinessGate` before any app content renders.

```
idle
  ‚Üí no workspace in localStorage  ‚Üí no_workspace ‚Üí redirect /onboarding
  ‚Üí workspace found               ‚Üí checking

checking  (fetching profiles.list)
  ‚Üí all 13 profiles present       ‚Üí ready ‚úì
  ‚Üí profiles missing, first time  ‚Üí bootstrapping

bootstrapping  (createFromDefinition with existing workspaceId)
  ‚Üí done                          ‚Üí rechecking

rechecking  (fetching profiles.list again)
  ‚Üí all 13 profiles present       ‚Üí ready ‚úì
  ‚Üí still missing                 ‚Üí waiting

waiting  (scheduled retry every 3.5s)
  ‚Üí profiles appear               ‚Üí ready ‚úì
  ‚Üí after 4 retries               ‚Üí show "Check again" button
  ‚Üí manual retry                  ‚Üí reset bootstrappedRef ‚Üí checking
```

Key invariants:
- `bootstrappedRef` prevents calling `createFromDefinition` more than once per mount
- Manual retry resets `bootstrappedRef` so re-bootstrap is possible
- `mountedRef` guards all state updates to prevent stale-closure updates after unmount
- `createFromDefinition` is idempotent ‚Äî safe to call with an existing `workspaceId`

## WebSocket endpoints

Both connect directly to the pod (not via the Next.js proxy). The session token is passed as a query param because cookies don't transfer on cross-origin WebSocket upgrades.

### SSH Terminal ‚Äî `/api/devplane/ssh`

```
?envId=<entity_id>&token=<kratos_session_token>

1. Validate token via Kratos
2. Fetch devplane_environment ‚Üí envHost, sshUser, sshKeyVaultRef
3. Resolve vault://uuid ‚Üí plaintext SSH key (server-side only)
4. ssh2.shell() ‚Üí PTY proxy

Client ‚Üí Server:  binary (input) | JSON { type:"resize", cols, rows }
Server ‚Üí Client:  binary (output) | JSON { type:"ready"|"error"|"closed" }
```

### Recipe Runner ‚Äî `/api/devplane/recipe-run`

```
?recipeId=<entity_id>&envId=<entity_id>&token=<kratos_session_token>

1. Validate token
2. Fetch recipe ‚Üí parse recipeSteps JSON
3. Fetch environment ‚Üí resolve SSH credentials
4. For each step: ssh2.exec() ‚Üí stream stdout/stderr
5. Save devplane_recipe_run entity

Server ‚Üí Client (JSON events):
  { type: "step_start", index, name, command }
  { type: "output", index, data }
  { type: "step_done", index, exitCode, success }
  { type: "recipe_done", success, duration }

Client ‚Üí Server:  { type: "cancel" }
```

## Infrastructure model

Canonical deployment stack (provider-agnostic, SSH-based):

```
Local (dev)
  ‚îî‚îÄ‚îÄ pnpm dev ‚Äî no SSH, no DevPlane management needed

Staging (1 server)
  ‚îî‚îÄ‚îÄ Any SSH-accessible VPS (Hetzner CX21, DO Droplet, etc.)
  ‚îî‚îÄ‚îÄ Server entity: { envType: "staging", kamalRole: "all" }
  ‚îî‚îÄ‚îÄ Recipe: kamal template ‚Üí kamal deploy --destination staging

Production (fleet)
  ‚îî‚îÄ‚îÄ Multiple servers with explicit roles:
      web √óN    ‚Üí { kamalRole: "web" }
      worker √óN ‚Üí { kamalRole: "worker" }
      db √ó1     ‚Üí { kamalRole: "db" }
  ‚îî‚îÄ‚îÄ DevPlane graph view visualizes topology + live status
```

Recipe templates: `kamal` (primary), `shell`, `custom`. Docker Compose and git-pull are custom recipes.

## Design decisions

**Frontend owns profiles.** The pod/backend has no knowledge of DevPlane-specific profiles. `DEVPLANE_WORKSPACE_DEFINITION` is the source of truth. The old `ensureDevplaneProfiles()` in `synap-backend` is dead code ‚Äî wrong scope (system vs workspace), never called.

**Idempotent bootstrap.** `createFromDefinition` with an existing `workspaceId` adds missing profiles without touching existing ones. This means the readiness gate can repair workspaces created before new profiles were added to the definition ‚Äî no manual migration needed.

**Server ‚âÝ Environment tier.** `devplane_environment` entities represent individual server nodes, not environment tiers. `envType` (staging/production) is a property, not the entity type. Multiple Server entities can share `envType: "production"` with different `kamalRole` values ‚Äî this is how the Kamal fleet is modelled.

**SSH over CI.** Many self-hosted setups don't have a CI runner. SSH is the lowest common denominator and Kamal specifically expects it.

**Recipe steps as JSON string.** Synap property values are typed scalars. `JSON.stringify(steps)` keeps the data in the property system without schema changes.

**Provider-agnostic servers.** `serverProvider` is metadata only ‚Äî DevPlane connects via SSH regardless of provider. Hetzner, DO, AWS, bare metal ‚Äî same SSH + vault pattern.
