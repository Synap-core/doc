## 1. The Vision

Eve turns a bare server into a **personal AI operating system**: a beautiful, marketplace-driven launcher for everything the operator runs ŌĆö proprietary apps, automation packs, messaging bridges, agents ŌĆö with the Synap pod as the single source of truth underneath.

This document is the north star. It defines:

- The **agentic triangle** invariant (Synap = brain, OpenClaw = ingress, Hermes = execution)
- The **OS surface map** (what lives at each Eve route)
- The **marketplace model** (apps + packs + entitlements)
- The **Agents app** (live flow + chronological timeline; see [Eve Agents design](./eve-agents-design.mdx))
- The **auth handshake** between Eve and the Control Plane
- A **phased build plan** with explicit "not shipping yet" per phase

What it does NOT do: prescribe pixels. UI design happens after this doc is merged, in the Frontend Design Skill.

### What each agent actually does *today* in our deployment

| Agent | Upstream capability | Our integration today | Gap |
|---|---|---|---|
| **OpenClaw (`@eve/arms`)** | MCP standard support ŌåÆ access any MCP server in the ecosystem; native adapters for WhatsApp (Baileys), iMessage (BlueBubbles), Telegram, Signal, Matrix, voice | Telegram + Signal + Matrix + voice configured; WhatsApp/iMessage adapters exist upstream but not wired in our env (BlueBubbles needs a Mac, Baileys needs QR session) | Setup work, not code work |
| **Hermes (`@eve/builder`)** | `ThreadPoolExecutor` for parallel sub-agents (up to 3), periodic SOP-nudge learning loop, native cron | Polling daemon over Synap task queue. Sequential. Cron disabled deliberately (Synap owns state). Sub-agent delegation + SOP loop **disabled accidentally** by integrating Hermes as a thin API caller. | Re-enable sub-agent delegation + SOP loop while keeping Synap as task source ŌĆö fast-follow for Phase 2/3, see ┬¦10 |
| **Synap (`@eve/brain`)** | Hub Protocol (76+ REST endpoints), entity graph, proposals, channels, A2AI routing, semantic + full-text search | All shipped and working | None |

### The invariants (do not violate)

1. **Synap owns state.** No agent maintains long-term memory in its own DB. OpenClaw's SQLite, Hermes' internal task DB ŌĆö both stripped.
2. **REST-only inter-agent.** Every cross-agent action is a Synap entity write or read. No direct WebSocket, no shared queue outside Synap.
3. **Proposals gate everything consequential.** Anything that mutates state outside Synap (sending an external message, deploying code, triggering a webhook) goes through `checkPermissionOrPropose()` unless the action is on the `DEFAULT_AUTO_APPROVE` whitelist.
4. **OpenClaw and Hermes never overlap.** OpenClaw never executes code. Hermes never reads a WhatsApp message. If you find code that crosses this line, it's a bug.

> Realtime visibility into this triangle relies on the typed event registry ŌĆö see
> [Event Chain ŌåÆ Realtime event naming](./event-chain.mdx#realtime-event-naming--actorentityaction).
> Every cross-agent action emits an `{actor}:{entity}:{action}` event so Synap (the brain)
> stays the canonical witness of what the other organs are doing.

## 4. The Marketplace as Proprietary Centerpiece

This is the architectural unlock. Today the Synap CP marketplace is schema-only and only models automation packs. We extend it to model **everything proprietary**:

### What the marketplace catalogs

| `app_type` | Example | Install effect |
|---|---|---|
| `url` | The Arch, CRM, DevPlane, custom dashboards | Eve registers a tile. Click ŌåÆ opens external URL. App is hosted however you want (Coolify, Vercel, anywhere). |
| `eve_component` | OpenWebUI, Dev Agent | Surfaces an Eve-managed local component as an app. Install triggers Eve's install lifecycle for the component identified by `metadata.componentId`. Lets the marketplace become the unified catalog of *everything proprietary* ŌĆö including components Eve already manages. |
| `workspace_pack` | "CRM workspace template", "Builder workspace" | Installs into the user's Synap pod via `POST /api/hub/workspaces/from-definition`. Pre-existing logic, no change. |
| `bundle` | "The Arch + its companion automation pack" | Composite ŌĆö installs the children. |

**`metadata` JSONB column** carries type-specific config:

- `eve_component` ŌåÆ `{ componentId: string, engineOptions?: string[], defaultEngine?: string }`. The Dev Agent uses `engineOptions: ["openclaude", "opencode", "claude-code"]` so the install flow can prompt for engine choice and write it to `secrets.builder.codeEngine`.
- `workspace_pack` ŌåÆ `{ templateSlug }`
- `bundle` ŌåÆ `{ childAppSlugs: string[] }`
- `url` ŌåÆ typically null

We do NOT model in Phase 1:

- ŌØī **Coolify provisioning.** Marketplace does not deploy infra. Apps are pre-hosted, marketplace just serves the link. Re-evaluate in Phase 3.
- ŌØī **Per-tenant containers.** Same reason.

### Entitlements (per-user gating)

The CP issues entitlements that determine what each user sees:

```sql
CREATE TABLE entitlements (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL,
  app_id UUID NOT NULL,             -- references marketplace_apps
  granted_at TIMESTAMPTZ NOT NULL,
  expires_at TIMESTAMPTZ,           -- nullable ŌĆö for trials/subscriptions
  source TEXT NOT NULL,             -- 'manual' | 'stripe' | 'beta' | 'free'
  stripe_subscription_id TEXT,      -- nullable ŌĆö ties to billing
  status TEXT NOT NULL DEFAULT 'active'  -- 'active' | 'revoked' | 'expired'
);
```

Eve's marketplace query:

```
GET /api/marketplace/apps?for_user=me
Authorization: Bearer <cp.userToken>

ŌåÆ CP joins entitlements + public catalog
ŌåÆ Returns:
  - Free apps (always visible)
  - Paid apps user is entitled to (full install enabled)
  - Paid apps user is NOT entitled to (visible as "Buy" CTA)
```

The Arch becomes invisible to users without an active entitlement. CRM might be visible-but-locked with a "Subscribe" CTA. Free open-source apps are always visible.

### Where the code lives

| Layer | Location | Purpose |
|---|---|---|
| **Shared types + UI components** | `synap-app/packages/core/marketplace/` (new package `@synap/marketplace`) | App card, install button, pricing display, entitlement state UI |
| **Admin curation** | `synap-control-plane-dashboard/src/pages/marketplace/` | Founder-facing: create/edit apps, set pricing, grant manual entitlements, see install metrics |
| **User-facing browse** | `synap-app/packages/apps/marketplace/` (or extend existing browser MarketplaceApp) | Customer-facing browse + install |
| **CP API** | `synap-control-plane-api/src/routes/marketplace.ts` | List apps with entitlement join, install endpoint, entitlement CRUD |
| **Eve consumer** | `hestia-cli/packages/eve-dashboard/app/page.tsx` | Reads marketplace via `cp.userToken`, renders app grid |

### Behaviors

- **Live by default.** Real-time activity flows along edges as messages arrive. Subtle pulse animation on the edge for each event.
- **Click a node ŌåÆ side-panel.** Clicking "WhatsApp" opens config for the WhatsApp bridge (API keys, session QR). Clicking "OpenClaw" opens its skill list and persona settings. Clicking "Synap Channel" shows recent messages in that channel. Clicking "Hermes" shows the task queue.
- **Connect/disconnect via the panel.** No separate forms. Disconnected nodes are greyed out; clicking offers an inline onboarding flow ("Connect WhatsApp" ŌåÆ 3 steps ŌåÆ done).
- **Recent activity tape** at the bottom. Last 10 events, plain language: "WhatsApp ŌåÆ 'build a login page' ŌĆö 2s ago".

### Onboarding for external channels

A user adding (say) Discord to their Eve should see a single guided flow:

1. Click `+ Add` ŌåÆ channel picker (Discord / Slack / WhatsApp / Telegram / Signal / Matrix)
2. Auth (OAuth or QR) ŌåÆ handled by OpenClaw
3. Choose default Synap channel to route incoming messages into (defaults to user's personal channel)
4. Done. Edge appears in the flow, lights up on first message.

After this onboarding, the user should never need to touch it again. The flow visualizes itself thereafter.

### What this is NOT

- **Not a workflow editor.** Users do not draw edges. The topology is fixed by the agentic triangle invariant ŌĆö we visualize what's there, we don't let users redraw it.
- **Not a chat UI.** That's OpenWebUI's job (and eventually a native Synap chat). The channels page shows traffic; reading/replying to messages happens elsewhere.

## 6. Auth Handshake (Eve Ōåö CP)

> Canonical reference for credential mechanics (two-channel rule, three-credential model, JWT-Bearer exchange flow, bootstrap path) is **[Eve Credentials & the Two-Channel Rule](./eve-credentials.mdx)**. This section sketches the CP side only.

Eve queries the CP marketplace as the user, not as a service. This means Eve needs a user-scoped JWT.

### The flow

1. User opens Eve dashboard for the first time after install
2. Dashboard sees no `cp.userToken` in `~/.eve/secrets.json`
3. Dashboard redirects to `https://cp.synap.sh/oauth/eve?return=http://localhost:3000/auth/callback`
4. CP authenticates user (existing Better Auth session) and shows consent screen: "Allow Eve to read your installed apps and entitlements?"
5. User approves ŌåÆ CP redirects to `http://localhost:3000/auth/callback?code=...`
6. Dashboard exchanges code for JWT via `POST /oauth/token`
7. Dashboard writes `cp.userToken` to `~/.eve/secrets.json` (under existing secrets management)
8. Dashboard reloads, queries marketplace with bearer token, renders app grid

### Token scope

The JWT is **strictly scoped to marketplace read access**. Specifically:

- `marketplace:read` ŌĆö list apps, list entitlements
- `marketplace:install` ŌĆö record local installation (for tracking)

It does NOT grant:

- ŌØī Pod admin access (the user already has pod access via their pod-local credentials)
- ŌØī Billing access (Stripe Customer Portal handled by CP, not Eve)
- ŌØī Cross-user data

Token rotation: 30-day expiry, refresh handled by CP refresh token flow (standard OAuth).

## 7. OpenWebUI's Place in the OS

OpenWebUI is **a prominent tile in the app grid, pinned by default** as "Chat" (or whatever name the operator prefers).

Until Synap has a native chat surface (out of scope for the Eve OS work), OpenWebUI is how the user actually talks to their sovereign data + agents. Hiding it in settings would be a UX failure.

The grid renders OpenWebUI as a first-class app:

- Tile labeled "Chat" with appropriate icon
- Pinned position (top-left or similar prominent slot)
- Click ŌåÆ opens OpenWebUI in a new tab/window
- Status indicator if OpenWebUI is offline

OpenWebUI's existing pipelines (already shipped ŌĆö Tracks A and B per `openwebui-integration-strategy.md`) handle Synap memory injection and channel sync. The user sees one OpenAI-compat backend (Synap IS); providers are abstracted.

## 8. What We're NOT Building

Explicit list to prevent scope creep:

- ŌØī **Paperclip integration.** Synap is the better version of Paperclip's coordination layer (proposals + entity graph). Revisit only if/when Synap's automation engine cannot model what Paperclip does.
- ŌØī **Custom Eve chat UI.** OpenWebUI fills this slot. A native Synap chat is out of scope for this work.
- ŌØī **Direct OpenClaw Ōåö Hermes communication.** Violates the agentic triangle invariant.
- ŌØī **Coolify component in Eve.** Coolify stays a dumb hosting provider. Apps in the marketplace are pre-hosted; we serve the link.
- ŌØī **A workflow editor on the channels page.** The topology is fixed by the architecture. We visualize, we don't let users redraw.
- ŌØī **Multi-tenant Eve.** One Eve install = one operator. Multi-user marketplaces are a CP concern, not an Eve concern.

## 9. Phased Build Plan

Each phase is independently shippable. No phase blocks another from starting in parallel if resourcing allows, but the listed order is the sensible default.

### Phase 0 ŌĆö Protect the Brain (PREREQUISITE)

**Goal:** Synap PostgreSQL is durable, backed up, and recoverable before anything else ships on top of it.

Audit (2026-05-05) rated current protection at **1.5/5** ŌĆö manual dashboard backups only, named Docker volume, no scheduled backups, no documented restore for self-hosted Eve, retention policy absent.

- **Bind-mount Postgres volume** to `${EVE_HOME}/data/postgres` instead of named Docker volume (one-time migration tooling required for existing installs)
- **Nightly `pg_dump` ŌåÆ S3** automated backup script in pod-agent (`synap-backend/scripts/backup-postgres.sh`) wired to a cron job
- **Retention policy** ŌĆö keep last 30 days local, last 12 months in S3 (cold tier)
- **Restore procedure documented** in `hestia-cli/README.md` and dashboard Doctor page
- **Monthly restore test** in CI ŌĆö spin up test Postgres, take backup, wipe, restore, verify
- **`recreate` safety assertion** ŌĆö explicit guard preventing `recreate` on Synap component without confirmation

**Not shipping:** off-site replication / hot-standby (Phase 4+), volume encryption at rest (operator-specific deployments only).

### Phase 1 ŌĆö Marketplace API + Entitlements (CP)

**Goal:** CP can serve a list of apps gated by user entitlements.

- Schema: `marketplace_apps` (id, name, description, icon_url, category, app_type, app_url, pricing, status), `entitlements` (above)
- API: `GET /api/marketplace/apps`, `POST /api/marketplace/install`, `GET /api/entitlements`, `POST /api/entitlements` (admin)
- Admin UI: `synap-control-plane-dashboard/src/pages/marketplace/` ŌĆö create/edit apps, manual entitlement grant
- Auth: extend OAuth to issue user-scoped JWT for Eve consumption
- **Not shipping yet:** user-facing browse, Stripe wiring, automated provisioning

### Phase 2 ŌĆö Eve OS Home (`/`)

**Goal:** Eve dashboard home is a beautiful app grid pulling from CP marketplace + local Eve components.

- New `@synap/marketplace` package (shared UI components)
- New Eve dashboard route `/` ŌåÆ app grid
- OAuth handshake to obtain `cp.userToken`
- Auto-discovery of Eve components with `webUrl` (current `/apps` page logic, ported)
- Move existing dashboard pages under `/settings`
- OpenWebUI tile pinned by default
- **Fast-follow:** Hermes un-neutering ŌĆö re-enable sub-agent delegation + SOP loop while keeping Synap as the task queue source. (Phase 2.5 once OS home ships.)

### Phase 3 ŌĆö Agents App (`/agents`)

## 11. Decision Log

Decisions locked in for this north star (do not re-litigate without explicit revisit):

1. **Marketplace location: BOTH** ŌĆö admin in CP dashboard, user-facing in `synap-app`, shared package `@synap/marketplace`.
2. **Install model: (A) + (C)** ŌĆö pure registry for apps, workspace seed for automation packs. NO Coolify provisioning in Phase 1.
3. **Auth: user-scoped JWT** ŌĆö OAuth handshake from Eve to CP. Marketplace-read scope only.
4. **Agents app: Flow + Timeline** ŌĆö n8n-style live node graph plus chronological event stream over the same data, click-to-configure. Not a workflow editor.
5. **OpenWebUI: pinned tile in app grid** ŌĆö first-class default, not hidden in settings.
6. **Hermes management: Synap macro-queue + Hermes micro-parallelism** ŌĆö Synap owns state; Hermes' internal sub-agent delegation re-enabled per task. Cron stays disabled (Synap-as-state golden rule).
7. **Paperclip: not integrated** ŌĆö Synap is the better Paperclip.

## 12. Glossary

- **Agentic triangle** ŌĆö The invariant relationship between Synap, OpenClaw, and Hermes: Synap owns state, the other two communicate only through it.
- **Entitlement** ŌĆö A CP record granting a specific user access to a specific marketplace app. The basis for per-customer gating.
- **OS surface** ŌĆö The user-facing routes of the Eve dashboard: `/`, `/agents`, `/settings`. Distinct from the underlying agent infrastructure.
- **App tile** ŌĆö A clickable card in the Eve home representing one installed app (proprietary, open-source, or auto-discovered Eve component).
- **Live node flow** ŌĆö The Agents app's Flow view, rendering message traffic as animated edges between agent nodes in real time. Sister view: the Timeline (chronological event stream).
