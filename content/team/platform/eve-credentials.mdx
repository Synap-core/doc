## 1. TL;DR

Eve speaks to two backends — the **Control Plane** at `api.synap.live` (marketplace, entitlements, releases) and the **operator's pod** at `synap.<domain>` (data, agents, governance). Two backends, two transport channels, three credentials, one rule of thumb:

> **If I were doing this manually as the operator in a terminal, would I authenticate as me-the-human or me-the-service?** That's the channel.

| Slot in `~/.eve/secrets.json` | Identity | Used for |
|---|---|---|
| `cp.userToken` | CP user | CP marketplace + entitlements |
| `pod.userToken` | Pod user (Kratos session) | tRPC against the pod, human operator actions |
| `pod.agentApiKey` | Agent / service user | Hub Protocol against the pod, agent and service actions |

The third credential was previously misnamed `synap.apiKey`; new code uses `pod.agentApiKey`.

---

## 2. The Two Channels

Eve only ever talks to a pod over one of two channels. They are not interchangeable.

| Channel | Identity in claims | Use it for |
|---|---|---|
| `tRPC /trpc/*` | **User** (Kratos session) | Anything a *human operator* does |
| `Hub Protocol /api/hub/*` | **API-key holder** (agent or service user) | Anything a *service or agent* does |

### The rule of thumb

For every new Eve UI feature, ask the test question:

> If I were doing this manually as the operator in a terminal, would I authenticate as me-the-human or me-the-service?

Examples:

- **Read my inbox / mark notification read / approve a proposal** — me-the-human → tRPC.
- **Edit a profile / invite a teammate / change workspace settings** — me-the-human → tRPC.
- **Submit a proposal from an agent / post a proactive nudge / sync from a connector** — me-the-service → Hub Protocol.
- **OpenClaw skill calls / IS↔backend memory writes** — me-the-service → Hub Protocol.

When in doubt: **tRPC**. The agent channel exists for genuinely agentic actions, not as a bypass for "I don't have a user session yet."

## 3. The Three Credentials

Eve persists all three credentials in `~/.eve/secrets.json` (under the existing secrets-management contract — see `@eve/dna/src/secrets-contract.ts`). Each lives in a clearly named slot.

| Credential | Provenance | Used for |
|---|---|---|
| `cp.userToken` | PKCE redirect (loopback) or RFC 8628 device flow against `api.synap.live` | CP marketplace + entitlements |
| `pod.userToken` | RFC 7523 JWT-Bearer exchange against the operator's pod (§4) | tRPC against the operator's pod, user identity |
| `pod.agentApiKey` (formerly `synap.apiKey`) | `eve auth provision` against `/api/hub/setup/agent`, gated by a [Trusted Issuer](./trusted-issuers.mdx) approval or `PROVISIONING_TOKEN` for self-hosted | Hub Protocol against the pod, service identity |

**Never use one credential where another belongs.** The proxy layer (§7) enforces this; individual UI pages stay simple because the proxy paths themselves carry the meaning.

### When each credential is minted

- `cp.userToken` is minted on first install of Eve when the operator approves the OAuth consent screen. Refresh handled by the standard OAuth refresh-token flow. 30-day expiry.
- `pod.userToken` is minted **on demand** every time Eve needs to call tRPC and the cached token is missing or expired. The exchange is silent — the operator never re-authenticates separately.
- `pod.agentApiKey` is minted at install time by `eve auth provision`. Per-agent keys are documented in [Eve — Per-Agent Hub Keys](./eve-agent-keys.mdx); the slot referenced here is the `eve` agent's key, used for Doctor probes and dashboard introspection. Other agents (`openclaw`, `hermes`, `coder`, …) live in `agents.<slug>.hubApiKey`.

## 4. How `pod.userToken` is Minted (RFC 7523 JWT-Bearer Grant)

Eve registers itself as a trusted issuer of its own pod and exchanges short-lived JWTs for Kratos session tokens. This is the [JWT-Bearer Grant](https://datatracker.ietf.org/doc/html/rfc7523) (RFC 7523), used over the `urn:ietf:params:oauth:grant-type:jwt-bearer` grant type.

### One-time setup (install)

1. Eve generates an ES256 keypair on first launch.
2. Eve publishes its JWKS at `https://eve.<domain>/.well-known/jwks.json` (public key only).
3. Eve registers itself in the pod's `trusted_issuers` table with `allowed_scopes: ["auth:exchange-user"]`. The pod admin approves the registration once — or it is auto-approved during admin bootstrap (§5).

### Per-session flow

```
                 Eve dashboard                     Pod (api.synap.<domain>)
                       │                                    │
  1. operator signs    │                                    │
     into Eve (eve-    │                                    │
     session cookie)   │                                    │
                       │                                    │
  2. mint JWT          │                                    │
     iss = eve URL     │                                    │
     sub = email       │                                    │
     aud = pod URL     │                                    │
     exp = now + 5min  │                                    │
     jti = nonce       │                                    │
     signed w/ ES256 ──┼───────── POST /api/hub/auth/exchange
                       │            grant_type=urn:ietf:...:jwt-bearer
                       │            assertion=<jwt>         │
                       │                                    │
                       │                                    │  3. lookup issuer in
                       │                                    │     trusted_issuers
                       │                                    │  4. fetch JWKS, verify
                       │                                    │     signature + aud + exp
                       │                                    │     + jti (replay cache)
                       │                                    │  5. lookup user by email
                       │                                    │     (must be userType=human
                       │                                    │     with kratosIdentityId)
                       │                                    │  6. POST Kratos admin
                       │                                    │     /admin/identities/:id
                       │                                    │     /sessions
                       │                                    │
                       │ ◄── 200 OAuth envelope ─────────── │
                       │     access_token=<kratos session>  │
                       │     token_type=Bearer              │
                       │     expires_in=86400               │
                       │     scope=user                     │
                       │                                    │
  7. store as          │                                    │
     pod.userToken     │                                    │
     use as Bearer for │                                    │
     /trpc/* calls     │                                    │
```

### Endpoint contract

`POST /api/hub/auth/exchange` is **not** API-key gated — it *is* the auth primitive. Authentication happens via the JWT signature plus the `trusted_issuers` allowlist. The endpoint returns standard OAuth 2.0 error envelopes per RFC 6749 §5.2 (`invalid_request`, `invalid_grant`, `invalid_client`, `insufficient_scope`, `user_not_found`, `server_error`).

Hard constraints enforced by the pod:

- `iss` claim must be an HTTPS URL and present in `trusted_issuers` with `status='approved'`.
- The issuer's `allowed_scopes` must include `auth:exchange-user`.
- `aud` must equal the pod's `PUBLIC_URL`.
- Assertion lifetime (`exp - iat`) must be ≤ 300 seconds (5 min).
- `jti` is tracked in an in-process replay cache for 15 minutes.
- The `sub` user must be `userType='human'` with a `kratosIdentityId`. Agents cannot be exchanged into.

### Refresh

When `pod.userToken` is rejected (Kratos session expired), Eve transparently re-mints a fresh assertion and re-exchanges. The operator never re-authenticates. From the operator's perspective, signing into Eve once is sufficient for the lifetime of `cp.userToken`.

### Multi-instance pods (gap)

The replay cache is currently in-process. For multi-node pods, the implementation note at the top of `auth.ts` flags a follow-up: a `used_assertion_nonces` table so a replay against a different node is also rejected. Single-node pods are not affected.

## 5. Bootstrap Path (No Admin Yet)

A freshly installed pod has zero users. Eve must be able to create the first admin without pre-existing credentials.

1. Eve calls public `trpc.setup.status`.
2. If `initialized: false`, Eve surfaces a "Create first admin" card on the home screen.
3. The operator submits **email + name**. Eve POSTs to the pod's `/api/admin/bootstrap/claim` with:

   ```json
   {
     "token": "<ADMIN_BOOTSTRAP_TOKEN>",
     "email": "alice@example.com",
     "role": "admin"
   }
   ```

   `ADMIN_BOOTSTRAP_TOKEN` is the one-shot pod-deploy secret the operator already has access to (it lives in `deploy/.env` next to `PROVISIONING_TOKEN`).
4. The pod creates a pod-wide invite tied to that email and returns the invite link.
5. The operator completes Kratos signup in their browser using the invite link.
6. Once the user exists, Eve's first JWT-Bearer exchange (§4) issues a `pod.userToken`. The operator is now authenticated and the bootstrap card disappears.

After bootstrap, the trusted-issuer registration for Eve is auto-approved as part of the same flow — the admin doesn't need to visit `/admin/trusted-issuers` to click through.

## 6. Multi-User Onboarding (Admin → Invitee)

For every additional human on the pod, the flow is invite-driven and stays inside the user channel.

1. Admin uses Eve UI (or any other pod UI) to call `trpc.workspaces.createInvite({ type, email, role })` where `type` is `"pod"` or `"workspace"`. Returns `{ token, expiresAt }`.
2. The invite URL is `https://eve.<domain>/invite/<token>` (Eve's invite page proxies to the pod) or the pod URL directly — both work.
3. Invitee opens the URL → Eve fetches invite metadata via the existing public procedure `acceptInviteViaCp` → presents a Kratos signup form with email pre-filled and locked.
4. On signup, the invite is consumed; the invitee is added to the workspace(s) per the invite's `type` and `role`.
5. Same email = same identity across CP and pod once the operator next signs into Eve and the JWT-Bearer handshake runs.

No admin ever issues an API key to a human user. Hub keys are exclusively for agents and services.

## 7. Proxy Namespace on Eve

Eve's local server exposes three proxy roots. The path itself encodes which credential is attached.

| Path on Eve | Forwards to | Auth header |
|---|---|---|
| `/api/marketplace/*` | CP `https://api.synap.live/api/marketplace/*` | `cp.userToken` (Bearer) |
| `/api/pod/*` | pod `/trpc/*` (mostly) or pod public REST | `pod.userToken` (Bearer or cookie) |
| `/api/hub/*` | pod `/api/hub/*` (Hub Protocol REST) | `pod.agentApiKey` (Bearer) — **only for genuinely agentic actions** |

Apply the rule of thumb when picking which proxy a new feature should call:

- `/api/pod/*` is the default for any operator-driven UI. If you're not sure, this is the answer.
- `/api/hub/*` is the answer when the operation is on behalf of an agent — submitting a proposal, posting a proactive nudge, an OpenClaw skill round-trip.
- `/api/marketplace/*` is reserved for CP-hosted concerns (catalog, entitlements, releases). The pod never sees these calls.

The proxy layer attaches the credential. Pages don't pick credentials directly — they pick a path. This keeps the channel choice auditable in code review.

---

## 8. Why This Matters

Mixing channels is not a stylistic concern. It causes three concrete failures:

1. **Audit-log lies.** Inbox-style features called over `/api/hub/*` were briefly attributed to the `eve` agent in audit logs because the agent's name carried over from the API-key context. The operator's own actions appeared as service writes. Channels exist precisely so that "who did this" is recoverable from the access log.
2. **Wrong RBAC checks.** Service users typically have broader scopes than humans (memory writes, cross-workspace reads). Routing a human action through the agent key gives that action elevated privileges it should not have, and conversely a human reading their own inbox over Hub Protocol bypasses the user-scoped filters in tRPC.
3. **Impossible refactoring later.** Once a UI surface mixes the two, you cannot tighten Hub key scopes without breaking the surface. Per-agent scope tightening (the open question at the bottom of [Eve — Per-Agent Hub Keys](./eve-agent-keys.mdx)) requires that no human-driven UI relies on agent scopes by accident.

Channels are enforced at the proxy layer. The cost of getting this wrong rises sharply once features ship.

## 9. Glossary

- **Two-channel rule** — The Eve invariant that `tRPC /trpc/*` carries user identity and `Hub Protocol /api/hub/*` carries API-key (agent/service) identity. Each new feature picks one and stays in it.
- **Trusted issuer** — An entry in the pod's `trusted_issuers` registry that grants an external party the right to mint JWT-Bearer assertions. For Eve, the trusted issuer is Eve itself (its own dashboard URL). Approved by an admin or auto-approved during pod bootstrap.
- **JWT-Bearer Grant** — RFC 7523 OAuth 2.0 grant type (`urn:ietf:params:oauth:grant-type:jwt-bearer`). A short-lived signed JWT is presented as the assertion; the authorization server returns a session/access token.
- **`auth:exchange-user`** — The single scope a trusted issuer must hold to use `/api/hub/auth/exchange`. Other scopes on the issuer record gate other future exchange types.
- **`pod.agentApiKey`** — The slot in `~/.eve/secrets.json` for the `eve` agent's Hub key. Used for the Hub Protocol channel. Sibling slots `agents.openclaw.hubApiKey`, `agents.hermes.hubApiKey`, etc. exist for other agents but are not what this document is about.
