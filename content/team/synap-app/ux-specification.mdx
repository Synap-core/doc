## 1. PHILOSOPHY & ARCHITECTURE

### 1.1 Core Principle: Separation by Purpose

Synap separates apps by **purpose** rather than by entity type or view type. Each app serves a distinct cognitive mode:

```
CANVAS    → Create spatially (visual thinking)
DATA      → Manage entities (database operations)
VIEWS     → Analyze custom (saved queries)
DOCS      → Compose narratives (document writing)
AI CHAT   → Collaborate with intelligence (conversation)
DASHBOARD → Access quickly (entry point)
```

This is different from Notion (everything is a page) or Airtable (everything is a database). We **intentionally separate** entity management (Data), custom queries (Views), and documents (Docs).

### 1.2 The LEGO Brick Metaphor

**Entities are LEGO bricks** - atomic typed data (Task, Person, Company, Project, etc.)

**Apps are ways to work with bricks:**
- Canvas: Arrange bricks spatially on infinite whiteboard
- Data: View brick inventory, modify brick types
- Views: Create custom arrangements of bricks
- Docs: Write about bricks, embed brick arrangements
- AI Chat: Talk with intelligence that operates on bricks

### 1.3 Entity Type Architecture

**Entity types are first-class:**
- Task, Person, Company, Project, Document, Event, etc.
- Each has **typed metadata** (columns/properties)
- Metadata is **flexible** but **typed** (not schema-less chaos)
- Users can define entity types and their metadata (in Data app, modification mode)

**Important distinction from Notion:**
We don't treat everything as pages. A Task is a Task. A Document is a Document. They have different behaviors and different metadata.

### 1.4 Organization Philosophy: No Folders

**We deliberately avoid folder hierarchies.**

**Instead, organization happens through:**

1. **Projects** (hierarchical entities)
   - Can have sub-projects
   - Example: Company → V1 Launch → Feature A
   - Projects ARE entities (have metadata, can be visualized)
   
2. **Tags** (flat, cross-cutting)
   - Multiple tags per entity
   - No nested tags (keeps it simple)
   - Filter by tag combinations

3. **Entity relationships** (typed connections)
   - depends_on, assigned_to, part_of, etc.
   - First-class relationships in event-sourced DB

4. **Search** (full-text, fuzzy)
   - Primary navigation method
   - Command palette (Cmd+K) everywhere

### 1.5 Workspace Model: No Team Separation

**Critical difference from Linear:**

We do **NOT** separate by teams. We have **ONE workspace** with:
- Roles (Owner, Admin, Member, Guest)
- Permissions based on roles
- Optional: Users can create "groups" but they're NOT structural separators

**Views are:**
- Personal (only you see)
- Shared (everyone in workspace sees)
- NO "team views" concept

### 1.6 The Two Inbox Pattern

**Inbox 1: External Items**
- Emails, messages, external data
- Things coming FROM outside

**Inbox 2: Request Approvals**
- AI-proposed changes awaiting approval
- User requests without permission
- Things needing YOUR action

Both inboxes surface on Dashboard and have badge counts.

### 2.1 Layout Structure

```

│  [Header - Optional, contextual]                   │
├────┬───────────────────────────────────────────────┤
│    │                                               │
│ S  │          MAIN CONTENT AREA                    │
│ I  │                                               │
│ D  │                                               │
│ E  │                                               │
│ B  │                                               │
│ A  │                                               │
│ R  │                                               │
│    │                                               │
│    │                                               │
└────┴───────────────────────────────────────────────┘
```

**Sidebar (Icon-only, Notion-style):**
- Position: Fixed left
- Width: 48-56px
- Collapsible: `[` key
- Icons vertically stacked
- Active app highlighted

**Header:**
- NOT always present (app-specific)
- When present: breadcrumbs, context, actions
- Height: 48-64px

**Main content:**
- Full viewport width (minus sidebar)
- Scrollable
- May contain panels

### 2.2 Sidebar App Icons (Top to Bottom)

```
┌────┐
│ 🏠 │ Dashboard (default landing)
├────┤
│ 🔍 │ Search (opens command palette)
├────┤
│ 🎨 │ Canvas
├────┤

├────┤
│ 👁️ │ Views
├────┤
│ 📄 │ Docs
├────┤

├────┤
│ ─── │ (separator)
├────┤
│ ⚙️ │ Settings (bottom)
├────┤
│ 🌙 │ Dark mode toggle (bottom)
└────┘
```

**Interaction:**
- Click icon → Navigate to app
- Active app → Icon highlighted (cyan/accent color)
- Hover → Tooltip with app name + shortcut

### 2.3 Multi-Panel Support

**Apps can split into multiple panels:**

```
┌────┬──────────────┬──────────────┐
│ S  │  Panel 1     │  Panel 2     │
│ I  │  (60%)       │  (40%)       │
│ D  │              │              │
│ E  │              │              │
└────┴──────────────┴──────────────┘
```

**Panel controls:**
- Drag separator to resize (min 30%, max 70%)
- Click X on panel to close
- Click + to add panel (if app supports)
- Panels can show different apps OR same app with different content

**Common panel combinations:**
- Canvas + Data (draw, see list)
- Data + Properties (table, detail view)
- Docs + Canvas (write about visual)
- Data Table + Data Graph (same data, two views)

**Per-app panel support:**
- Dashboard: No (always single)
- Canvas: Yes (split canvas OR canvas + other app)
- Data: Yes (table + graph OR table + detail)
- Views: Yes (multiple saved views side-by-side)
- Docs: Yes (doc + reference material)
- AI Chat: Yes (chat + context)

### 2.4 Command Palette (Global)

**Trigger:** `Cmd+K` (Mac) or `Ctrl+K` (Windows/Linux)

**Functionality:**
- Search all entities
- Navigate to apps
- Create new items
- Execute actions
- Recently used items

**Structure:**
```
┌────────────────────────────────────────┐
│  🔍 Search or command...               │
├────────────────────────────────────────┤

│  + New entity                          │

│                                        │
└────────────────────────────────────────┘
```

**Keyboard shortcuts within palette:**
- `↑` `↓` - Navigate results
- `Enter` - Execute selected
- `Cmd+Enter` - Execute in new panel
- `Esc` - Close palette

### 2.5 Contextual Menus

**Right-click on entities:**
```
┌────────────────────────┐
│ Open                   │
│ Open in new panel      │
│ ───────────────────    │
│ Edit                   │
│ Duplicate              │
│ ───────────────────    │
│ Add to view            │
│ Copy link              │
│ ───────────────────    │
│ Delete                 │
└────────────────────────┘
```

**Consistent across all apps** for entities.

### 2.6 Keyboard Shortcut Philosophy

**Follow Linear's two-key pattern:**
- `G` then letter = **Go to** (navigation)
- `C` then letter = **Create** (new items)
- `V` then letter = **View** (switch views)

**Examples:**
- `G` → `D` = Go to Dashboard
- `G` → `C` = Go to Canvas
- `C` → `T` = Create Task
- `C` → `D` = Create Document
- `V` → `T` = View as Table
- `V` → `G` = View as Graph

**Single-key shortcuts:**
- `[` = Toggle sidebar
- `/` = Focus search (if applicable)
- `?` = Show shortcut help
- `Esc` = Close modals/panels

### 2.7 Empty States (Universal Pattern)

**All empty states must have:**
1. Icon (48×48px, relevant to context)
2. Headline (clear, positive)
3. Description (1-2 sentences explaining benefits)
4. Primary action button
5. Optional: Link to documentation

**Example:**
```
┌────────────────────────────────────┐
│          📊                        │
│                                    │
│    No views created yet            │
│                                    │
│    Create custom dashboards with   │
│    filtered entity lists, graphs,  │
│    and timelines.                  │
│                                    │
│    [Create your first view]        │
│                                    │
│    Learn about views →             │
└────────────────────────────────────┘
```

### 4.4 Entity Creation

**Draw box → Entity created automatically:**
1. User draws rectangle with Draw tool
2. System detects closed shape
3. Entity created with type inference:
   - Rectangle + text inside → Task (if action words)
   - Rectangle + name → Person (if proper noun)
   - Rectangle + date → Event
4. User can change type via property panel

**Draw line → Relationship created:**
1. User draws line from Entity A to Entity B
2. System creates typed relationship
3. Default type: "related_to"
4. User can change type via context menu
   - depends_on, assigned_to, part_of, blocks, etc.

**Flowchart mode (optional enhancement):**
- Toggle: Switches canvas to "structured mode"
- Entities snap to grid
- Only boxes and arrows allowed
- Auto-alignment

**Context menus:**

- Right-click canvas → New entity, Paste

**Multi-panel:**
- Split canvas (two canvases side-by-side)
- Canvas + Data (draw left, see table right)

### 5.4 Progressive Disclosure: Simple vs Modification Mode

**SIMPLE MODE (Default):**
- See entities in table
- Search, filter, sort
- Add new entities
- Edit entity data
- NO schema editing

**MODIFICATION MODE (Owner/Admin only):**
- All simple mode features PLUS:
- Add/remove columns (metadata)
- Change column types
- Create new entity types
- Delete entity types
- Configure relationships

**Toggle between modes:**

```

[Status] [is] [Todo] [×]
[Assigned] [is] [Me] [×]
[Due date] [is before] [Jan 31] [×]
[Priority] [is either of] [High, Urgent] [×]
```

**Available operators:**
- is / is not
- is either of (multi-select)
- contains / does not contain
- is empty / is not empty
- is before / is after (dates)
- is greater than / less than (numbers)

**Filter matching logic:**
- Default: "All must match" (AND)
- Toggle: "Any can match" (OR)

**Shown as:**


### 8.7 Auto-Recreation (Infinite Chat Illusion)

**Behind the scenes:**

Synap auto-recreates chats to maintain performance and context window limits.

**Every N messages (e.g., 20):**
1. System saves conversation state
2. Creates new chat with summarized context
3. User sees seamless "infinite" conversation
4. Old chunks archived but accessible

**Visible to user (optional):**
- Sidebar shows "Recreation points" (collapsed by default)
- Click to see chunks: "Messages 1-20", "Messages 21-40"
- Power users can navigate chunks
- Most users never see this

**Benefits:**
- Performance stays fast
- Context window never overflows
- Complete history preserved
- Transparent to user

## APPENDIX A: DESIGN PRINCIPLES

**1. Progressive Disclosure**
Show simple interface by default, reveal complexity on demand

**2. Keyboard-First**
All actions accessible via keyboard, shortcuts discoverable

**3. Context-Aware**
UI adapts based on what user is doing and what data they have

**4. Consistent Patterns**
Similar actions work the same way across all apps

**5. Instant Feedback**
UI responds immediately, async operations show progress

**6. Forgiving UX**
Undo/redo everywhere, confirmations for destructive actions

**7. Empty States Teach**
Every empty state explains what's possible and how to start

**8. Search Over Navigate**
Command palette and search preferred over deep navigation
