Marketing: A comprehensive map of the synap-app monorepo — 80+ npm packages organized as a shared frontend OS spanning design tokens, data client, cell rendering, view adapters, feature modules, and native OS bridges.

Key User Stories:
- **As a frontend engineer**, I want a complete package map with npm names, roles, and GitHub links so I can find the right package without searching the monorepo.

- **As a feature implementer**, I want to know which packages own view renderers, cell system, entities, channels, AI interfaces, and platform auth so I can extend or create packages correctly.
- **As a new contributor**, I want to understand why packages exist as separate npm modules instead of one app folder (bundle control, non-React contracts, swappable views, separation of concerns).

## Overview

The **`synap-app`** monorepo ([**Synap-core/synap-app** on GitHub](https://github.com/Synap-core/synap-app)) is the **shared frontend OS**: Tamagui UI, tRPC client, hooks, stores, view implementations, cells, and feature apps. **Primary user surface today is the Electron Browser**; `apps/web` and other apps are hosts or legacy shells that **reuse the same packages**.

**Path prefix on GitHub:** `https://github.com/Synap-core/synap-app/tree/main/` + path below.

This space is the canonical package map. For product context about synap-app, see **[Synap App](/team/synap-app)**.

---

## Why packages instead of one app folder

1. **Bundle control** — hosts import only the features they ship (Browser vs extension vs Relay).  
2. **Non-React contracts** — e.g. **`@synap-core/capabilities`** stays free of React so **agents, scripts, and the backend** can read manifests.  
3. **Swappable view engines** — each view type is a package (`view-table`, `@synap-core/kanban-view`, …) behind **`@synap-core/view-renderer`**.  
4. **Separation of concerns** — **entity card** (dense list/grid tile) ≠ **entity views** (inspector / tabs / full layout) ≠ **property-renderer** (typed fields). Each can version independently.

---

## Three-layer data (mandatory)

Components use **`@synap-core/hooks`** → **`@synap-core/stores`** → **`@synap-core/client`** (tRPC + TanStack Query). Do not import the client directly from random UI packages.

## Auth and external-connect boundaries (current rule)

- **`@synap-core/platform-flow`**: pure decisions/state classification only (no transport).
- **`@synap-core/auth`**: canonical CP + pod auth/session primitives, including shared-pod connect.
- **`@synap-core/external-connect-client`**: integration-specific connect helpers that compose auth primitives and backend-owned contracts.
- **`@synap-core/platform-auth`**: UI and host adapters only.
- **`@synap-core/pod-connect`**: compatibility package; new implementations should import from `@synap-core/auth`.

For details and migration guidance, see **[Auth & External Connect Package Boundaries](/team/synap-packages/auth-and-external-connect-boundaries)**.

## How to integrate a new host (checklist)

1. Add **`@synap-core/client`** + **`@synap-core/hooks`** + **`@synap-core/stores`** with your pod URL / auth.  
2. Wrap with **`@synap-core/ui-system`** `TamaguiProvider` + theme.  
3. Pull only the **feature packages** you need (avoid importing all `packages/apps/*`).  
4. Register cells via **`@synap-core/cell-runtime`** if you embed dashboard tiles.  
5. For **view types**, depend on **`@synap-core/view-renderer`** — it pulls concrete view packages.
