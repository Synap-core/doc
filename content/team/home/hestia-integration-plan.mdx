# Hestia ↔ Synap Backend Integration Plan

> How the Hestia Core package connects to the Synap Backend through the Hub Protocol REST API.

## Executive Summary

Hestia is a **composition of packages**. The **Core Package** (`hestia-core`) is Synap Backend. This document outlines:
1. How Hestia packages integrate with Synap Backend
2. What backend extensions are needed
3. How the modular CLI assembles packages
4. The phased implementation approach

**Key Principles from Decisions**:
- **Agnostic Intelligence**: Any IS (Ollama, OpenRouter, Anthropic, custom)
- **Package-Based**: Each component is a self-contained package
- **Script-First**: USB is a wrapper around the script installer
- **CP as Optional**: Control Plane is one connector among many, for redundancy/fleet only

### 2.2 New Hub Protocol Endpoints

#### Intelligence Query Endpoint (Agnostic IS Interface)

**POST /api/hub/intelligence/query**

Standardized interface for any intelligence provider:

```typescript
// Request
{
  "provider": "ollama",  // or "openrouter", "custom"
  "model": "llama3.1:8b",
  "messages": [
    { "role": "system", "content": "You are helpful" },
    { "role": "user", "content": "Hello" }
  ],
  "tools": [...],  // Optional
  "stream": false
}

// Response (OpenAI-compatible)
{
  "id": "chatcmpl-123",
  "object": "chat.completion",
  "model": "llama3.1:8b",
  "choices": [...]
}
```

**Implementation**: Backend proxies to configured provider. Provider config stored in `intelligence_provider` entity.

#### Package Registration Endpoint

**POST /api/hub/packages/register**

Register an installed package with the hearth:

```typescript
// Request
{
  "package_name": "openclaw",
  "version": "2026.3.12",
  "hearth_node_id": "uuid",
  "config": {
    "channels": ["telegram", "discord"]
  },
  "endpoints": {
    "http": "http://localhost:3000",
    "websocket": "ws://localhost:3001"
  }
}

// Response
{
  "package_instance_id": "uuid",
  "entity_id": "uuid",
  "registered_at": "2026-04-12T10:00:00Z",
  "status": "registered"
}
```

#### Hearth Node Endpoints

**POST /api/hub/hearth/register**
Register a new hearth node (replaces `orion/register`):

```typescript
{
  "hostname": "my-hearth",
  "role": "primary",
  "install_mode": "script",
  "packages": ["core", "ollama"]
}
```

**POST /api/hub/hearth/heartbeat**
Node health update:

```typescript
{
  "hearth_node_id": "uuid",
  "packages": [
    { "name": "core", "status": "running", "version": "1.0.0" },
    { "name": "openclaw", "status": "running", "version": "2026.3.12" }
  ],
  "intelligence": {
    "provider": "ollama",
    "status": "healthy",
    "model": "llama3.1:8b"
  }
}
```

## 4. Agnostic Intelligence Service (IS)

### The Intelligence Package Interface

Any package can provide intelligence capabilities:

```typescript
interface IntelligenceProvider {
  name: string;
  version: string;
  
  // Configuration
  configure(config: ProviderConfig): Promise<void>;
  
  // Core capability: Chat completions (OpenAI-compatible)
  chatCompletions(request: ChatRequest): Promise<ChatResponse>;
  
  // Optional: Embeddings
  embeddings?(text: string): Promise<number[]>;
  
  // Optional: Model listing
  listModels?(): Promise<ModelInfo[]>;
  
  // Health
  health(): Promise<HealthStatus>;
}
```

### Provider Implementations

**1. Ollama (Default, Local)**
```yaml
packages:
  intelligence:
    provider: ollama
    config:
      endpoint: http://localhost:11434
      model: llama3.1:8b
```

**2. OpenRouter (Multi-Provider)**
```yaml
packages:
  intelligence:
    provider: openrouter
    config:
      api_key: ${OPENROUTER_KEY}
      model: anthropic/claude-3-opus
      # Can switch models dynamically:
      # model: openai/gpt-4
      # model: google/gemini-pro
```

**3. Anthropic (Cloud)**
```yaml
packages:
  intelligence:
    provider: anthropic
    config:
      api_key: ${ANTHROPIC_KEY}
      model: claude-3-opus-20240229
```

**4. Custom (Any OpenAI-Compatible)**
```yaml
packages:
  intelligence:
    provider: custom
    config:
      endpoint: http://my-llm-server:8000/v1
      api_key: ${CUSTOM_KEY}
      model: default
```

### Backend Proxy

The Synap Backend provides a unified endpoint:

**POST /api/hub/intelligence/query**

```typescript
// Backend receives this:
{
  "hearth_id": "hearth-alpha",
  "messages": [...],
  "tools": [...]
}

// Backend looks up intelligence provider from hearth config
// Backend proxies to appropriate endpoint
// Backend returns standard response
```

This allows:
- Gateway package to call intelligence without knowing provider
- Builder package to use any model
- Easy switching between local and cloud

## 5. Control Plane as Optional Connector

### The CP Connector Package

```yaml
# ~/.hestia/config.yaml (optional CP section)
connectors:
  control-plane:
    enabled: true  # Optional! Only for fleet/redundancy
    url: https://cp.hestia.sh
    role: primary  # primary | backup | fleet-member
    
    # Primary hearth configuration
    backup_hearths:
      - hearth-beta
      - hearth-gamma
    
    # Fleet configuration (future)
    fleet_id: my-fleet
    fleet_role: coordinator
```

### When CP is Used

1. **Redundancy**: Sync data between primary and backup hearths
2. **Fleet Management**: Manage multiple hearths from central view
3. **DNS**: Managed DNS for public domains (optional, can use local DNS)
4. **Monitoring**: Centralized fleet monitoring (optional, can use local monitor)

### When CP is NOT Used

Hestia works completely standalone:
- Local intelligence (Ollama)
- Local channels (OpenClaw)
- Local building (OpenClaude)
- Local DNS (optional dynamic DNS)
- Local backups (restic to S3, etc.)

## 7. Key Design Decisions Applied

| Decision | Implementation |
|----------|----------------|
| **1. CLI Architecture (Option C: Modular with plugins)** | CLI is composed of `packages/` (required) and `plugins/` (optional). CP is a plugin. |
| **2. Entity vs Tables (Option A: Pure entity)** | All Hestia concepts are entities: `hearth_node`, `package_instance`, etc. No new tables. |
| **3. Backend Scope (Agnostic)** | Intelligence proxy works with any provider. Synap Backend doesn't care about IS source. |
| **4. OpenClaw Relationship (Part of triptick)** | Gateway package is OpenClaw. One of three pillars. |
| **5. CP Relationship (Optional connector)** | CP is one connector plugin among many. Used only for redundancy/fleet. |
| **6. Intelligence (Agnostic)** | Any OpenAI-compatible endpoint works. Backend proxies uniformly. |
| **7. Install Method (Script primary, USB wrapper)** | `install/` package is foundation. `usb/` package wraps it. |
| **8. Multi-Node (Single with redundancy path)** | Single hearth now. CP connector enables backup/fleet later. |
