### Graph Queries

No graph DB. Use existing `relations` table with indexed queries.

For 2-hop traversals (sufficient for personal network):
```sql
-- "Who knows Solidity and is in Paris?"
-- Hop 1: person в†’ has_skill в†’ skill(Solidity)
-- Filter: person.properties->>'location' LIKE '%Paris%'

-- "Mutual connections between me and Person X"
-- Find people where: relation(me, knows, Y) AND relation(X, knows, Y)

-- "Connection path to Person Z"
-- BFS through relations table, max 3 hops
```

Backend: new tRPC procedures in a `relay` router (or extend existing entity/relation queries).

## 3. Backend Extensions

### New tRPC Router: `relay`

```typescript
// synap-backend/packages/api/src/routers/relay.ts
export const relayRouter = router({
  // Feed
  getFeed: workspaceProcedure
    .input(z.object({ cursor: z.string().optional(), limit: z.number().default(20) }))
    .query(/* aggregate recent interactions, AI signals, encounters */),

  // People with Relay enrichment
  getPeople: workspaceProcedure
    .input(z.object({ filters: relayFilterSchema, sort: relaySortSchema }))
    .query(/* entities with profileSlug=person, Relay properties */),

  // Graph data
  getGraph: workspaceProcedure
    .input(z.object({ filters: relayFilterSchema, depth: z.number().default(2) }))
    .query(/* nodes + edges from relations table */),

  // Entity resolution
  getMergeCandidates: workspaceProcedure
    .query(/* find potential duplicate persons */),

  mergePersons: workspaceProcedure
    .input(z.object({ sourceId: z.string(), targetId: z.string() }))
    .mutation(/* merge two person entities, preserve event chain */),

  // AI operations
  generatePrepBriefing: workspaceProcedure
    .input(z.object({ eventId: z.string() }))
    .mutation(/* call IS to generate event prep */),

  queryNetwork: workspaceProcedure
    .input(z.object({ query: z.string() }))
    .mutation(/* NL query в†’ graph traversal via IS */),
});
```

### Connector Extensions

**Telegram import worker:**
1. User uploads Telegram export (JSON)
2. Worker parses message history
3. Groups messages into "conversation sessions" (messages within 30min window)
4. For each session: call IS `/api/structure` to extract entities
5. Create/update Person entities, Interaction entities, relations
6. Deduplicate via phone number / Telegram handle

## 8. What's NOT Needed

- **No new database tables** for core entities (entities + properties JSONB)
- **No graph database** (relations table with indexes is sufficient at personal scale)
- **No new auth system** (Better Auth via pod-connect)
- **No new real-time system** (Socket.IO via @synap-core/client)
- **No new search infra** (Typesense + pgvector already available)
- **No new AI pipeline** (Intelligence Service + capture pipeline)
- **No new job system** (pg-boss, add new worker types)
- **No new channel type** for proactive AI (two pod-wide channels per user: `isPersonal` for chat, `isProactiveFeed` for AI posts)

## 9. Feed System (IMPLEMENTED вЂ” 2026-04-07)

See `docs-internal/FEED-SYSTEM.md` for full spec.

**3-surface model:**
- Personal chat (`isPersonal: true`) вЂ” userв†”AI only. Context stays clean.
- Proactive feed (`isProactiveFeed: true`) вЂ” AI posts: briefings, event prep, automation outputs.
- Notifications (DB table) вЂ” ephemeral nudges. "Reconnect with John."

**Relay automation templates (5 total, installed via useRelaySetup):**
| # | Name | Output surface |
|---|------|---------------|
| 1 | New Contact Enrichment | entity_update |
| 2 | Strength Score Computation | entity_update |
| 3 | Daily Reconnection Nudges | notification |
| 4 | Event Prep Briefing | proactive feed |
| 5 | Interaction Summary | entity_update |

**useFeed.ts aggregates 5 sources:** people entities, proactive feed channel, notifCenter.list, campaigns, entity-linked channel messages.

**tRPC router note:** `trpc.notifCenter.*` = automation notifications table. `trpc.notifications.*` = legacy inbox (N8N external items). Feed uses `notifCenter`.
