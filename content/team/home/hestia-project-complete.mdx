### 2. Key Architectural Decisions Made вњ“

| Decision | Chosen | Implication |
|----------|--------|-------------|
| **CLI Architecture** | **Option C** вЂ” Modular with plugins | CP is a plugin, not required |
| **Entity System** | **Option A** вЂ” Pure entities | No new DB tables needed |
| **Backend Scope** | **Agnostic** | Any IS provider works |
| **Intelligence** | **Agnostic** | OpenAI-compatible endpoints |
| **Install Method** | **Script primary, USB wrapper** | Script is foundation |
| **Multi-Node** | **Single в†’ redundancy в†’ fleet** | Progressive scaling |
| **OpenClaw** | **Part of triptick** | One of three pillars |
| **Control Plane** | **Optional connector** | For redundancy only |

### 3. Package-Based Architecture

**Everything is a package:**

```
Hestia = Core Package + Gateway Package + Builder Package + Intelligence Package

Core Package (hestia-core) в†’ Synap Backend
Gateway Package (openclaw)  в†’ Multi-channel communication
Builder Package (openclaude) в†’ AI artifact generation
Intelligence Package в†’ ollama | openrouter | anthropic | custom
```

**Each package handles its own logic.** The CLI assembles them.

### 4. Agnostic Intelligence

Any fire can warm the hearth:

```yaml
# Local (default)
intelligence:
  provider: ollama
  endpoint: http://localhost:11434

# Cloud (optional)
intelligence:
  provider: openrouter
  api_key: ${OPENROUTER_KEY}

# Custom (any OpenAI-compatible)
intelligence:
  provider: custom
  endpoint: http://my-llm:8000/v1
```

Backend provides uniform interface: `POST /api/hub/intelligence/query`

### 5. Entity-First Backend Integration

**No new database tables.** Everything is entities:

- `hearth_node` в†’ Entity with `hearth_node` profile
- `package_instance` в†’ Entity with `package_instance` profile
- `intelligence_provider` в†’ Entity with `intelligence_provider` profile
- `hearth_deployment` в†’ Entity with `hearth_deployment` profile
