**Key design principle:** The Intelligence Hub is **stateless** — it has no database of its own.
All state (threads, entities, documents, memory, skills) lives in the Synap Backend.
The hub is pure reasoning: context in → LLM reasoning + tool calls → structured output out.

## 2. The Two Services

### 2a. Synap Backend (Data Pod)

The backend is the **source of truth**. All clients (including the hub) interact with it through
the Hub REST Adapter at `/api/hub/*`.

**Capabilities:**

- Entity CRUD (tasks, contacts, meetings, ideas, projects, notes) via event pipeline
- Document CRUD with versioning (text, markdown, code, pdf, docx)
- Chat threads + conversation messages (full history, branching, parent-child hierarchy)
- `knowledge_facts` table — long-term AI memory with 1536-dim pgvector embeddings
- Thread ↔ entity/document sidebar linking (idempotent, with relationship types)
- Skills — user-created AI tools stored in DB, executed at runtime
- Proposals — AI-suggested actions pending human approval
- Unified full-text search via Typesense (entities, documents, projects, threads, agents)
- Vector/semantic search via pgvector cosine distance
- Entity relation graph — BFS traversal up to 3 hops
- Cross-thread notifications via event pipeline + Inngest
- Workspace membership + permission validation on all writes
- Event watermark persistence (no replay on restart)

**Open-source components:**
PostgreSQL + pgvector · Inngest (self-hosted) · Typesense · MinIO · Drizzle ORM · Hono

### 2b. Intelligence Hub (AI Layer)

The hub is the **reasoning layer**. It calls tools, assembles context, and streams responses.

**Capabilities:**

- Streaming SSE chat with context-aware pre-prompt injection (conversation summary + personality fingerprint + linked entities/docs + memory)
- 7 agent types: orchestrator, mvp-chat, research, code, writing, knowledge-search, action
- 14 built-in tools (read + write + coordination categories)
- Dynamic user-defined skills loaded at runtime from the backend
- Proposal workflow: AI suggests → human approves → AI executes
- Multi-agent dispatch: orchestrator spawns specialist sub-agents in child threads
- Entity graph traversal tool
- Semantic memory search (pgvector cosine via embedding generation on hub)
- Personality fingerprint caching (6h TTL in thread metadata)
- Rolling thread summary (fire-and-forget after each context compression)
- Context inheritance for child threads (inherits parent messages)
- All LLMs via OpenRouter (Claude, GPT-4, DeepSeek, Gemini, open-source — switchable per agent)

**Proprietary components (your IP):**

- Agent prompting strategy and orchestration logic
- Context compression algorithm + personality fingerprinting
- Tool framework and backend integration mapping
- Pre-prompt assembly pipeline with multi-layer context injection
- Multi-agent hierarchy protocol (parentThreadId + child thread creation)

## 4. Intelligence Hub Internals

### Agent Execution Pipeline

```
AgentContext: { userId, threadId, parentThreadId?, workspaceId, message, history? }
                    │
                    ▼
        assembleContext()  [8-second timeout via Promise.race]
            ├── fetchThreadMessages(threadId)  [3s abort]
            ├── memoryService.recall(query, userId)  → semantic search + keyword fallback
            ├── hubProtocol.getThreadContext(threadId)  → linked entities/docs
            ├── generateConversationSummary(messages)  → cached in chat_threads.contextSummary
            ├── extractPersonalityFingerprint(messages)  → cached in metadata (6h TTL)
            └── fire-and-forget: hubProtocol.updateThreadContext(threadId, { contextSummary, personalityFingerprint })
                    │
                    ▼
        buildSystemPrompt(assembled, context)
            ├── ## Conversation History Summary
            ├── ## Your Communication Style  [personality fingerprint]
            ├── ## Active Context — Linked Entities
            ├── ## Active Context — Linked Documents
            └── ## Long-Term Memory
                    │
                    ▼
        generateText(model, systemPrompt, history + message, tools)
            ├── Tool calls auto-execute (context tools) or propose (action tools)
            ├── AIStep tracking: thinking → tool_call → tool_result
            └── Streaming callbacks: onThinking, onContent, onProposal
                    │
                    ▼
        AgentResponse: { content, aiSteps[], proposals[], branchDecision, requiresApproval }
```

### Context Caching Strategy

| What                    | Where                                          | TTL/Trigger                                          |
| ----------------------- | ---------------------------------------------- | ---------------------------------------------------- |
| Conversation summary    | `chat_threads.context_summary`                 | Re-generated when messages ≥ 6; fire-and-forget save |
| Personality fingerprint | `chat_threads.metadata.personalityFingerprint` | 6-hour TTL; re-extracted only when stale             |
| Memory facts            | `knowledge_facts` + pgvector                   | Recalled per request (semantic search)               |
| Thread context          | Backend API call                               | Per-request (lightweight, just a DB read)            |

### LLM Request Pipeline

```
User message arrives
       │
       ▼
classifyComplexity(message)  →  simple | normal | complex
       │                         │         │         │
       │                     free model  DeepSeek  GPT-4o/Opus
       │
       ▼
llmCircuitBreaker.check()  →  circuit open? → fast-fail with retryAfterSeconds
       │
       ▼
llmSemaphore.acquire(userId, priority)
   ├── Global cap: 20 concurrent (OPENROUTER_CONCURRENCY)
   ├── Per-customer cap: 4 concurrent (LLM_MAX_PER_CUSTOMER)
   ├── Priority: "interactive" served before "background"
   └── Timeout: 45s max queue wait (LLM_ACQUIRE_TIMEOUT_MS)
       │
       ▼
streamText() / generateText()  →  OpenRouter  →  response
       │
       ▼
semaphore.release() + circuitBreaker.onSuccess()
```

- **Dynamic complexity routing**: simple queries hit free models, complex ones get powerful models
- **Circuit breaker**: 5 consecutive failures → 30s open → half_open probe → auto-reset
- **Fair semaphore**: per-customer cap prevents tenant monopolization, priority queue ensures interactive users aren't blocked by background sub-agents

### dispatch_agent — Multi-Agent Hierarchy

When the orchestrator calls `dispatch_agent`:

1. A **child thread** is created with `parentThreadId = orchestratorThreadId`
2. The specialist agent runs with `threadId = childThreadId`
3. Sub-agent result posted back to parent thread as a `system` message
4. `childThreadId` returned in tool output (frontend can link to it)
5. Falls back to parent thread if child creation fails (non-fatal)

## 6. Document Handling

### Supported Document Types

| Type       | MIME                           | AI Readable | Notes                                                          |
| ---------- | ------------------------------ | ----------- | -------------------------------------------------------------- |
| `text`     | text/plain                     |           | Plain UTF-8 text                                               |
| `markdown` | text/markdown                  |           | Markdown, rendered for AI                                      |
| `code`     | application/code               |           | Source code + optional language field                          |
| `pdf`      | application/pdf                |           | Stored in MinIO; content extraction not yet implemented in hub |
| `docx`     | application/vnd.openxmlformats |           | Stored in MinIO; content extraction not yet implemented        |

### Storage Architecture

Documents follow a **dual-storage** model:

- **MinIO/R2 object storage**: raw file bytes (all types)
- **PostgreSQL**: metadata + working content (for text/markdown/code)
- **Yjs CRDT**: `working_state` binary dump for collaborative editing state

Schema fields:

```
documents
  id, userId, workspaceId, entityId?
  type, title, mimeType
  storageUrl, storageKey, size
  content (TEXT — for text/markdown/code)
  currentVersion, lastSavedVersion
  workingState, workingStateUpdatedAt  (Yjs CRDT)
  metadata (JSONB)
```

### AI Access to Documents

- **Reading**: `get_document` tool fetches full `content` field via hub-protocol REST
- **Writing**: `create_document` / `update_document` tools POST to backend
- **Proposals**: Document edits are proposed (diff-based) before applying, requiring user approval
- **Full-text search**: All documents indexed in Typesense at write time
- **Semantic search**: `vector_search` searches `entity_vectors` table (documents with entities)

### What's Missing for Full Document AI Support

| Gap                                                  | Effort | Priority |
| ---------------------------------------------------- | ------ | -------- |
| PDF text extraction (pdfjs-dist or pdf-parse)        | Medium | High     |
| DOCX text extraction (mammoth or docx2txt)           | Small  | High     |
| Image understanding (vision models)                  | Large  | Low      |
| Slide/presentation extraction (LibreOffice headless) | Large  | Medium   |
| Chunked reading for large docs (>100k tokens)        | Medium | Medium   |

**Recommended immediate fix** (2 files, no migration):

1. Add `pdf-parse` to hub package
2. In `get_document` tool: if `type === "pdf"` and no `content`, fetch from `storageUrl`, extract text, cache back to content field
3. Same pattern for `docx` via `mammoth`

## 8. Frontend Capabilities & Demo User Stories

### What the Frontend Receives

#### SSE Stream Events

```typescript
// Content chunk (stream text token by token)
{ type: "content", content: "partial text chunk" }

// AI thinking step (tool call, search, reasoning)
{ type: "step", step: { id: "step-1", type: "thinking"|"tool_call"|"tool_result", content: "...", timestamp: "..." } }

// Proposal (action pending user approval)
{ type: "proposal", data: { id: "uuid", toolName: "create_entity", description: "Create task: ...", args: { type: "task", title: "..." } } }

// Branch suggestion (AI recommends spawning a sub-agent)
{ type: "branch", decision: { shouldBranch: true, agentType: "research", purpose: "...", reasoning: "..." } }
```

#### Non-Streaming Response

```typescript
{
  content: string;           // Full text response
  aiSteps: AIStep[];         // Array of thinking/tool steps
  proposals: Proposal[];     // Pending actions awaiting approval
  branchDecision?: {         // AI's branch recommendation
    shouldBranch: boolean;
    agentType: string;
    purpose: string;
    reasoning: string;
  };
  requiresApproval: boolean;
  threadId: string;
}
```

#### US-01: Live Streaming Chat with Thinking Steps

1. A " Assembling context..." spinner (thinking step)
2. Tool call indicators: " Searching for related entities..."

**What to render:**

## 9. External Agent Integration

The hub is designed as an **open intelligence bus**. Any external agent can be integrated via two patterns:

### Pattern A: External Agent as a Tool

Expose any external service as a custom tool in the `tools/actions/` directory. The orchestrator can then call it like any other tool.

**Example: Perplexity as a tool**

```typescript
// tools/actions/perplexity-search.ts
export const perplexitySearchTool = defineTool({
  name: "perplexity_search",
  category: "context",
  description:
    "Real-time web search with AI synthesis. Use for current events, live data, or information not in the knowledge base.",
  parameters: z.object({
    query: z.string().describe("Web search query"),
    focus: z.enum(["internet", "news", "academic"]).default("internet"),
  }),
  async execute(args) {
    const response = await fetch("https://api.perplexity.ai/chat/completions", {
      method: "POST",
      headers: { Authorization: `Bearer ${process.env.PERPLEXITY_API_KEY}` },
      body: JSON.stringify({
        model: "sonar-medium-online",
        messages: [{ role: "user", content: args.query }],
      }),
    });
    const data = await response.json();
    return {
      answer: data.choices[0].message.content,
      citations: data.citations,
    };
  },
});
```

Then register in `tool-registry.ts` and add to the orchestrator's tool list.

**Other tools to build this way:**

- `brave_search` — Brave Search API (privacy-focused web search)
- `calculator` — Wolfram Alpha for computation
- `github_search` — GitHub API for code search
- `notion_read` — Read Notion pages
- `slack_message` — Send Slack messages
- `calendar_check` — Google/Microsoft Calendar

### Pattern B: External Agent as a Specialist (dispatch_agent)

Add a new agent type to the `agent-registry.ts` that wraps an external service. The orchestrator's `dispatch_agent` tool can then delegate to it.

**Example: Perplexity as a research sub-agent**

```typescript
// agents/specialist/perplexity-agent.ts
export class PerplexityAgent {
  async execute(context: AgentContext): Promise<AgentResponse> {
    // Call Perplexity API directly
    const response = await perplexityClient.chat({
      model: "sonar-large-online",
      messages: [{ role: "user", content: context.message }],
    });

    // Post result to child thread (for audit trail)
    await hubProtocol.postMessage({
      threadId: context.threadId,
      role: "assistant",
      content: response.content,
      userId: context.userId,
    });

    return { content: response.content, aiSteps: [] };
  }
}
```

Register in `agent-registry.ts`:

```typescript
case "perplexity": return new PerplexityAgent();
```

Add `"perplexity"` to `dispatchAgentTool`'s `agentTypeEnum`.

### Pattern C: External Agent as a Backend Trigger (Webhook/Event)

For agents that operate asynchronously (webhooks, long-running processes, domain-specific services):

1. External agent calls `POST /api/hub/threads/:id/messages` to inject findings into a thread
2. Frontend sees the new system message in real-time
3. Orchestrator can reference it in the next turn

**Use cases:**

- Code analysis bots (run on every commit, post findings to a "code review" thread)
- Monitoring agents (alert on anomaly → post to "infrastructure" thread)
- Domain-specific AI (medical, legal, financial) → post analysis to dedicated thread

### External AI Models via OpenRouter

The hub already routes ALL LLM calls through OpenRouter. Any model available on OpenRouter
can be used by simply adding it to `models.ts`:

```typescript
// models.ts — add any OpenRouter model
```

## 12. Graph Database Analysis: FalkorDB + Graphiti

### What Graphiti does

Graphiti (by Zep) is a **temporal knowledge graph framework** built on graph databases
(Neo4j, FalkorDB, Kuzu). It stores:

- **Episodes**: discrete interactions, timestamped
- **Entities**: extracted nodes (people, projects, concepts)
- **Relations**: edges with timestamps + invalidation (newer fact supersedes older)

Key capability: **bi-temporal model** — tracks when something happened AND when it was
ingested. Enables queries like "what did the AI know about project X as of last Tuesday?"

Graphiti uses **hybrid retrieval**: BM25 full-text + semantic embeddings + graph traversal.

### Current State vs Graphiti

| Our Current System                    | With Graphiti + FalkorDB                      |
| ------------------------------------- | --------------------------------------------- |
| `knowledge_facts` flat table          | Temporal knowledge graph                      |
| Cosine distance search (pgvector)     | BM25 + embeddings + graph traversal           |
| No contradiction handling             | Temporal edge invalidation                    |
| Entity relations in `relations` table | Full entity → relation → entity graph         |
| No episodic memory                    | Episodic memory (each conversation = episode) |

### Recommendation: Incremental Path

**Now (done):** pgvector cosine distance on `knowledge_facts`. Good enough for MVP.

**Phase D (short-term):** Skills workspace scoping, message queue per thread.

**Phase E (when you have product-market fit):**

- Run Graphiti as a Python sidecar (Docker Compose, REST API)
- FalkorDB as backing store (lighter than Neo4j, sub-ms graph traversal)
- Replace `MemoryService` to call Graphiti REST
- `HubProtocolClient.rememberFact()` interface unchanged — swap implementation only
