# Eve Channels — Design Spec

**Date:** 2026-05-05
**Status:** Design specification. Implementation pending Phase 3 of [Eve OS Vision](./eve-os-vision.mdx).

This document is concrete enough that a developer can implement the channels page end-to-end after one read. Sister doc to [Eve OS Home — Design Spec](./eve-os-home-design.mdx); shares its visual language and tokens.

---

## 1. Design intent

The channels page exists to make **the Agentic Triangle self-documenting**. The operator should glance at this page and immediately understand, without explanation:

> "WhatsApp messages arrive at OpenClaw, become Synap channel messages, become Hermes tasks, get executed, results flow back."

Three feelings to deliver:

1. **Aliveness** — the system is breathing. Edges pulse with real traffic. The OS feels awake.
2. **Legibility** — topology is clear at a glance. Nothing hidden, nothing ambiguous.
3. **Configurability without complexity** — clicking a node opens its panel. No dropdown trees, no settings labyrinth.

What it is NOT: a workflow editor. The topology is **fixed by the architecture**. Users observe and configure; they do not redraw the triangle.

## 3. Topology (fixed by architecture)

The graph is FIXED. Five conceptual zones, left to right:

| Zone | Contents | Type |
|---|---|---|
| **External** | Telegram, WhatsApp, Discord, Signal, Matrix, Slack, Voice | One node per connected channel + one `+ Add` placeholder |
| **OpenClaw** | Single node | Always present |
| **Synap Channels** | Personal channel + active branches | One node per channel (cluster if many) |
| **Hermes** | Single node | Always present |
| **Tasks** | Task queue (aggregate node) | Always present, shows queue depth |

Edges are also fixed:

- `External ↔ OpenClaw` — one per external channel
- `OpenClaw ↔ Synap` — single edge to channel cluster
- `Synap → Hermes` — task pickup
- `Hermes → Tasks` — execution + writeback to Synap (curves back to Synap)

User cannot add or remove edges. They can:
- Add new external channels (creates a new node + edge)
- Remove disconnected external channels
- Click any node to configure or inspect

## 4. Component breakdown

### 4.1 `TopBar` (`/app/(channels)/components/top-bar.tsx`)

Same height (64px) and structure as the OS Home TopBar, with one addition: a **filter chip row** below the search box.

Filter chips: `All` (default), `External`, `Synap`, `Hermes`, `AI Events`. Clicking a chip dims unrelated edges + nodes (opacity 0.3) so the user can focus.

### 4.2 `FlowCanvas` (`/app/(channels)/components/flow-canvas.tsx`)

Wraps `ReactFlow` from the `reactflow` package. Configuration:

```tsx
<ReactFlow
  nodes={nodes}
  edges={edges}
  nodeTypes={nodeTypes}
  edgeTypes={edgeTypes}
  fitView
  fitViewOptions={{ padding: 0.2 }}
  panOnDrag
  zoomOnScroll
  zoomOnPinch
  selectionOnDrag={false}
  nodesDraggable={false}        /* topology is fixed */
  nodesConnectable={false}       /* no manual edges */
  edgesUpdatable={false}
  elementsSelectable={true}
  minZoom={0.5}
  maxZoom={1.5}
  proOptions={{ hideAttribution: true }}
>
  <Background variant="dots" gap={24} size={1} color="rgba(255,255,255,0.04)" />
  <Controls position="bottom-right" showInteractive={false} />
</ReactFlow>
```

The canvas fills the viewport between TopBar and ActivityTape. Background uses a dotted grid at 4% opacity (whisper, not loud) — gives spatial reference without competing with content.

### 4.4 Edge types

```typescript
const edgeTypes = {
  pulseEdge: PulseEdge,
};
```

#### `PulseEdge` (`/app/(channels)/components/pulse-edge.tsx`)

Custom React Flow edge. Two visual layers:

1. **Static rail** — thin (1.5px), neutral-700 in dark mode / neutral-200 in light mode. The "track" the pulse travels on.
2. **Pulse layer** — when an event arrives, a small dot of light (4px diameter, AI-emerald gradient) is appended to the edge and animated from source to target over 1200ms with ease-out, fading to 0 at the end. Uses CSS animations (GPU-accelerated transforms only).

Multiple pulses can coexist on the same edge (busy traffic). Cap at 5 simultaneous pulses per edge to avoid visual overload — older pulses fade earlier when limit reached.

```css
@keyframes pulse-travel {
  0%   { offset-distance: 0%; opacity: 0; }
  10%  { opacity: 1; }
  90%  { opacity: 1; }
  100% { offset-distance: 100%; opacity: 0; }
}

.pulse-dot {
  offset-path: path('<the SVG path of the edge>');
  animation: pulse-travel 1200ms cubic-bezier(0.4, 0, 0.2, 1) forwards;
  width: 4px;
  height: 4px;
  border-radius: 50%;
  background: radial-gradient(circle, #6EE7B7 0%, #34D399 50%, transparent 100%);
  box-shadow: 0 0 6px rgba(110, 231, 183, 0.6);  /* glow only on pulses, not nodes */
}
```

Note: this is the only place we use `box-shadow` in the entire OS — for the traveling-light glow effect. Static depth is still done with borders.

## 5. Data model

### 5.1 Page state

```typescript
type ChannelNodeType =
  | 'external' | 'openclaw' | 'synapChannel' | 'hermes' | 'tasks';

type ChannelNode = {
  id: string;
  type: ChannelNodeType;
  label: string;
  position: { x: number; y: number };  // computed by layout, not user-editable
  data: ExternalNodeData | OpenClawNodeData | SynapNodeData | HermesNodeData | TasksNodeData;
};

type Edge = {
  id: string;
  source: string;
  target: string;
  type: 'pulseEdge';
  data: { active: boolean };
};

type FlowEvent = {
  id: string;            // unique per event
  timestamp: number;
  type: 'openclaw:message:received'
      | 'channel:message:created'
      | 'hermes:task:picked'
      | 'hermes:task:completed'
      | 'channel:reply:sent';
  source: string;        // node id
  target: string;        // node id
  excerpt?: string;      // for activity tape
  payload?: Record<string, unknown>;
};

type PageState = {
  nodes: ChannelNode[];
  edges: Edge[];
  recentEvents: FlowEvent[];     // capped at 100, last 10 in tape
  selectedNodeId: string | null;
  filterChip: 'all' | 'external' | 'synap' | 'hermes' | 'ai';
  isOnboarding: boolean;
  socketStatus: 'connected' | 'disconnected' | 'connecting';
};
```

### 5.2 Real-time event flow

```
Synap Realtime (Socket.IO /presence)
        │
        ▼
useChannelSocket() hook
        │
        ▼
events stream → reducer
        │
        ├── append to recentEvents (slice 100)
        ├── derive node statuses (was the sender online recently?)
        └── trigger edge pulse animation (push pulse to edge.data.activePulses[])
        │
        ▼
ReactFlow re-renders nodes/edges
ActivityTape re-renders top-10
```

Per the substrate audit, two events need to be added to Synap before this works:

| Event | Where to emit | Status |
|---|---|---|
| `openclaw:message:received` | `synap-backend/packages/api/src/routers/hub-protocol/rest/channels.ts` after POST | Registry + schema landed 2026-05-05; emit-site wires next |
| `hermes:task:queued` / `:started` / `:completed` / `:failed` | `background-tasks-service` on status transition | Registry + schemas landed 2026-05-05; emit-site wires next |

Both follow the canonical `{actor}:{entity}:{action}` realtime naming convention —
see [Event Chain → Realtime event naming](./event-chain.mdx#realtime-event-naming--actorentityaction)
for the registry, schema policy, and `emitTyped` usage.

The other events (`chat:stream`, `route_to_channel`, `ai:proposal`) already fire today.

### 5.3 Layout algorithm

Topology is fixed → positions can be deterministic. Don't use auto-layout (`dagre`, `elkjs`) — they introduce visual instability when nodes are added/removed. Use a simple **manual zone layout**:

```typescript
function computeLayout(nodes: ChannelNode[]): ChannelNode[] {
  const ZONE_X = {
    external: 80,
    openclaw: 360,
    synap: 640,
    hermes: 920,
    tasks: 1180,
  };
  const NODE_GAP_Y = 96;

  return nodes.map((node, i) => {
    const zoneNodes = nodes.filter(n => n.zone === node.zone);
    const indexInZone = zoneNodes.indexOf(node);
    const totalInZone = zoneNodes.length;
    const startY = -(totalInZone - 1) * NODE_GAP_Y / 2;

    return {
      ...node,
      position: {
        x: ZONE_X[node.zone],
        y: startY + indexInZone * NODE_GAP_Y,
      },
    };
  });
}
```

Adding a new external channel just appends to its zone — vertical positions auto-recompute, smooth animation via React Flow's built-in transitions.

## 6. Interaction flows

### 6.1 First load

1. Page mounts → fetch initial state in parallel:
   - `GET /api/channels/topology` (Eve dashboard API) → list of connected external channels + Synap workspace channels
   - Open Socket.IO connection to Synap with API key (Phase 3A prerequisite)
2. Render canvas with initial nodes + edges (positions computed by `computeLayout`)
3. Wire socket event handlers
4. ActivityTape shows "Listening for events..." until first event arrives

### 6.2 Event arrives

1. Socket receives event (e.g., `openclaw:message:received`)
2. Reducer:
   - Pushes pulse to relevant edge's `activePulses[]` (with id + timestamp)
   - Prepends event to `recentEvents`
3. Edge re-renders with new pulse → CSS animation kicks off → 1200ms travel + fade
4. ActivityTape prepends new item → existing items shift right
5. After pulse animation completes, removes itself from `activePulses[]` (cleanup)

### 6.3 Click node

1. User clicks node → `setSelectedNodeId(id)`
2. Node renders with `.selected` style (border-color shifts to primary)
3. SidePanel slides in from right (280ms)
4. Panel renders type-specific content

### 6.4 Add a channel

1. User clicks `+ Add` placeholder OR a disconnected external channel node
2. `ChannelOnboardingModal` opens, Step 1
3. User flows through 3 steps
4. On `Connect`, modal shows progress
5. Eve API:
   - Updates OpenClaw config in `secrets.json`
   - Triggers `recreate` lifecycle action on OpenClaw component
   - Waits for OpenClaw to come back up (health check polling)
6. New external channel node appears in canvas with fade-in (200ms)
7. Modal closes
8. First inbound message lights up the edge

### 6.5 Filter chips

1. User clicks filter chip (e.g., "AI Events")
2. State updates `filterChip`
3. Edges + nodes that don't match dim to opacity 0.3 (220ms transition)
4. ActivityTape filters its list
5. Click `All` to reset

### 6.6 Activity tape interaction

1. User hovers an item in tape
2. The item's `source` and `target` node IDs map to the corresponding edge
3. Edge highlights: width 1.5px → 3px, brightness +20%, for 2 seconds
4. Hovering off ends the highlight

## 9. Performance considerations

The page is event-heavy. Things to get right:

1. **Don't re-render the canvas on every event.** Pulses live in edge `data.activePulses[]` — the edge component subscribes to its own data, doesn't ripple up to ReactFlow root.
2. **Cap activity tape at 100 events** — older drop. The DOM only renders the visible 10.
3. **Memoize node and edge components** — `React.memo` everywhere with explicit equality checks.
4. **Throttle socket bursts** — if 50 events arrive in one tick (rare but possible during a sync storm), batch into a single state update via `requestAnimationFrame`.
5. **Reduce motion mode** — `prefers-reduced-motion: reduce` disables pulse animations entirely; events appear as a brief border flash on the edge instead.
6. **Pause animations when tab is backgrounded** — listen for `document.visibilitychange`, freeze pulses until visible again. Also pauses the activity tape time-ago tickers.

## 10. Accessibility

- All nodes are keyboard-focusable. Tab cycles in zone order (external → openclaw → synap → hermes → tasks). Enter opens panel, Escape closes.
- Pulse animations are decorative — `aria-hidden`. The activity tape is the screen-reader narration of events: a live region (`aria-live="polite"`) that announces new events textually.
- Filter chips: arrow keys navigate, Enter toggles.
- Side panel: focus trap when open, Escape closes, focus returns to triggering node.
- Color is never the only signal: status dots also have hover labels ("connected", "offline"), edges have widths that change with activity.
- Onboarding modal: form fields properly labeled, error messages associated with inputs via `aria-describedby`.

## 12. Open design questions (deferrals)

1. **Multi-workspace support.** v1 shows one workspace's channels. If user has multiple workspaces, defer multi-workspace selector to v2.
2. **Aggregating busy edges.** If 100 events fire on one edge in a second, do we show 100 pulses (visual chaos) or aggregate to a thicker pulsing edge? v1: cap at 5 simultaneous, drop overflow. Revisit if it looks bad in prod.
3. **Historical replay.** "Show me what happened in the last hour" — scrub bar at bottom that replays events visually. Cool but not v1.
4. **Per-channel routing rules.** Currently any external channel routes to a single Synap default. Future: rule-based routing ("messages with #urgent go to branch X"). Out of scope.
5. **Voice channel visualization.** Voice has more dimensions (audio quality, transcription latency). v1 treats it like any other channel; richer visualization later.
6. **Cross-workspace edges.** If branches span workspaces (rare), how do we show that? Defer.

## 13. Reference: what this is NOT

- ❌ Not a workflow editor (topology fixed)
- ❌ Not a message reader (panels show recent activity, full reading happens in Synap)
- ❌ Not a settings labyrinth (config lives in panels, not buried in tabs)
- ❌ Not a debugging tool (it's the operator's window into their OS, not a developer log viewer — Doctor page is for that)
- ❌ Not a chat UI (that's OpenWebUI, accessed from the OS Home)

The channels page does ONE thing: make the agentic triangle visible and configurable. Anything else is in `/settings`.
