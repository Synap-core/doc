## Part 1: Making OpenClaw Use Synap's Intelligence Service

This is the killer play. If you pull this off, Synap becomes not just the knowledge layer but also the **intelligence layer** for OpenClaw agents. Users route their LLM calls through Synap, which means you control both the brain and the memory.

**How OpenClaw model routing works technically:**

OpenClaw supports any OpenAI-compatible API endpoint. From the config and docs I've seen, the model provider setup looks like this:

```json
{
  "models": {
    "providers": {
      "custom": {
        "baseUrl": "https://your-pod.synap.live/v1",
        "api": "openai-completions",
        "apiKey": "synap_user_...",
        "models": [
          {
            "id": "synap/auto",
            "name": "Synap Intelligence (Auto-routed)",
            "contextWindow": 200000,
            "maxTokens": 8192
          },
          {
            "id": "synap/balanced",
            "name": "Synap Balanced (Kimi K2.5)",
            "contextWindow": 131072,
            "maxTokens": 8192
          },
          {
            "id": "synap/advanced",
            "name": "Synap Advanced (Claude Sonnet)",
            "contextWindow": 200000,
            "maxTokens": 8192
          }
        ]
      }
    }
  }
}
```

**What you need to build on the Synap side:**

An OpenAI-compatible `/v1/chat/completions` endpoint on each pod (or on a shared Synap IS gateway) that accepts standard OpenAI-format requests and routes them through your four-tier system internally. The user sends a request to `synap/auto`, and your IS decides whether it goes to DeepSeek V3 (free tier), Kimi K2.5 (balanced), Claude Sonnet (advanced), or Claude Opus (complex). The response comes back in standard OpenAI format. OpenClaw never knows the difference.

This is architecturally simple because you already have the four-tier routing logic built. You just need to wrap it in an OpenAI-compatible HTTP interface. There are open-source proxies that do this (LiteLLM is the most common), but you'd want your own because the routing intelligence IS the product.

**The strategic value is enormous:**

Every API call flowing through Synap means you see usage patterns, you can optimize costs (90% of traffic on cheap/free models), you can inject entity context from the pod into the system prompt before it hits the model, and you have a natural upsell from free (self-hosted, BYOK) to paid (managed IS with smart routing). The user's OpenClaw agent gets better responses because it has structured context from the pod AND optimized model selection. That's something no other provider offers.

**The pricing angle:** Anthropic just killed subscription credits for third-party tools. OpenClaw users are now paying full API prices. If your IS routes 90% of traffic to DeepSeek V3 and Kimi K2.5 (near-free), you can undercut direct API pricing significantly while making margin on the 10% that hits Claude. The pitch: "your agent costs $300/month on Claude API directly. Through Synap IS, it costs $40/month with the same quality because we route intelligently."

## Part 2: The Synap CLI for OpenClaw — "synap init"

This doesn't exist yet and it absolutely should. This is your **distribution weapon**. Here's why:

Every single best-practices guide I read says the same things: harden security, bind to localhost, enable token auth, limit tool access, be careful with skills, set up memory properly, configure model routing. And every single user has to do all of this manually, reading five different blog posts and patching config files by hand.

You should build a CLI that does all of this in one command. Not an OpenClaw fork. A companion tool that configures an existing OpenClaw install with security best practices AND connects it to Synap.

**What `synap` CLI should do:**

## Additional CLI Commands

Beyond `synap init`, the CLI should have ongoing utility:

```bash
synap status          # Pod health, agent count, entity stats, IS usage
synap dash            # Open the Agent OS dashboard in browser
synap security audit  # Re-check security config, flag issues
synap backup          # Export pod (SQL + JSON + files)
synap update          # Update synap-memory skill + CLI
synap pod create      # Provision a new managed pod
synap pod export      # Full data export
synap is stats        # Intelligence Service usage: cost saved, tier distribution
synap skill publish   # Publish a skill to ClawHub (wraps clawhub CLI)
```

The `synap security audit` command alone is worth the install. Nobody else in the ecosystem offers a one-command security assessment. Every time someone posts about OpenClaw security concerns on HN or Reddit, you can say "run `synap security audit` — it checks everything automatically."

## The Architecture of the CLI

Build it as a standalone npm package:

```bash
npm i -g @synap/cli
```

Or bundled with the docker-compose for self-hosted:

```bash
curl -fsSL https://synap.live/install.sh | bash
```

Tech stack: Node.js (same ecosystem as OpenClaw), TypeScript, uses your existing @synap/sdk internally for pod operations. The CLI reads OpenClaw's config at `~/.openclaw/openclaw.json`, modifies it with the hardening rules and Synap provider config, and manages the synap-memory skill lifecycle.

Key design principle: the CLI is **non-destructive and reversible**. It never overwrites OpenClaw config without showing a diff first. Every change can be rolled back with `synap reset`. This builds trust — people are (rightly) paranoid about tools modifying their agent config.

## What This Means Strategically

You're not just building a skill anymore. You're building a **distribution tool that wraps OpenClaw in Synap infrastructure**. The funnel becomes:

```
User hears about OpenClaw
  → Searches "how to install openclaw securely"
  → Finds synap init (blog post, README, ClawHub)
  → Runs one command
  → Gets: hardened OpenClaw + structured memory + smart routing + dashboard
  → They're on Synap without even deciding to "switch" to anything
```

This is the Homebrew strategy. Nobody "decided" to use Homebrew. They installed something else and Homebrew was the tool that got it set up properly. `synap init` becomes the way smart people set up OpenClaw.
