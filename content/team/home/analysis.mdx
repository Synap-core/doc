# Analysis: IS as OpenClaw Provider, CLI, and Repo Structure

**Date:** 2026-04-04

---

## 1. Synap IS as OpenClaw AI Provider

### What exists
- Four-tier model routing: FREE (DeepSeek) → BALANCED (Kimi K2.5) → ADVANCED (Claude Sonnet) → COMPLEX (Claude Opus)
- Cost tracking with Prometheus metrics
- OpenRouter as unified backend
- All routing logic in `synap-intelligence-service/apps/intelligence-hub/src/config/models.ts`
- External chat API at `POST /api/external/chat/stream` (custom format, NOT OpenAI)

### What's needed
**One thing: an OpenAI-compatible endpoint.** OpenClaw supports any OpenAI-compatible API via its provider config. We need:

```
POST {pod-url}/v1/chat/completions
```

That accepts standard OpenAI format:
```json
{
  "model": "synap/auto",
  "messages": [{"role": "user", "content": "..."}],
  "stream": true
}
```

And returns standard OpenAI SSE deltas. Internally it routes through our four-tier system.

### Where to build it
**On the backend pod**, not on the IS directly. Reason: the pod already has auth (API keys), rate limiting, and the IS proxy. The flow:

```
OpenClaw → POST pod/v1/chat/completions (Bearer: API key)
  → Backend validates key, resolves IS endpoint
  → Backend forwards to IS /api/chat/stream (converting format)
  → IS routes to correct model tier
  → Response streams back in OpenAI delta format
```

This is a **single new route** in `synap-backend/packages/api/src/routers/external/` — maybe `openai-compat.ts`. ~200 lines. It's a format adapter, not a new system.

### What it implies

**For us:** One new route. Format conversion (OpenAI ↔ Synap). The routing intelligence stays in IS unchanged.

**For OpenClaw:** User adds Synap as a "custom provider" in their config:
```json
{
  "providers": {
    "synap": {
      "baseUrl": "https://pod.synap.live/v1",
      "apiKey": "synap_hub_live_xxx",
      "models": ["synap/auto", "synap/balanced", "synap/advanced"]
    }
  }
}
```

**For the user:** They get smart routing (90% on cheap models, 10% on Claude) without thinking about it. Their agent works the same but costs 5-10x less. AND every request can be context-enriched from the pod's entity graph (we can inject relevant entities into the system prompt before routing to the model).

### Effort
Small. 1-2 days for the OpenAI-compat endpoint + testing with OpenClaw.

## 2. The CLI Question

### What exists
**A CLI already exists:** `@synap/cli` in `synap-intelligence-service/apps/cli/`

Commands: `chat`, `threads`, `skills`, `install`, `commands`, `memory`, `entities`, `history`, `search`, `status`, `config`, `infra`.

Framework: Commander.js + Ink (React TUI). It's a full interactive terminal client for Synap.

### The question: Separate CLI or extend existing?

**Option A: Extend `@synap/cli`** — Add `synap init`, `synap connect`, `synap security audit` commands to the existing CLI.

**Option B: New `@synap/openclaw` package** — Separate CLI specifically for OpenClaw integration.

**Option C: New `synap-openclaw` repo** — Full separate repo with CLI, skills, and OpenClaw-specific tooling.

### Recommendation: Option A (extend existing) + move to its own repo later

The existing CLI is already a Node.js package with Commander.js. Adding `synap init` and `synap connect` is straightforward — they're just new command files.

The North Star doc describes `synap init` as the entry point. That command should:
1. Detect OpenClaw (check `~/.openclaw/`)
2. Apply security hardening
3. Connect to Synap pod (self-hosted or managed)
4. Install the `synap` skill
5. Optionally configure IS as AI provider

This fits naturally in the existing CLI. No need for a separate repo yet.

**When to split:** If the CLI grows to >20 commands and the OpenClaw-specific code dominates, extract to `synap-openclaw` repo. Not now.

### Where the CLI should live

Currently in `synap-intelligence-service/apps/cli/`. This is wrong — the CLI talks to the pod (backend), not the IS directly. It should be:

**Option 1:** Move to `synap-backend/apps/cli/` — co-located with the API it calls
**Option 2:** Standalone repo `synap-cli` — published to npm as `@synap/cli`
**Option 3:** Keep in IS but rename — it's a monorepo, location matters less

For now: keep where it is. The IS monorepo is its home. When we publish to npm, the package name (`@synap/cli`) matters more than the directory.

## 3. Should we create a `synap-openclaw` repo?

### What would go in it?
- The unified `synap` skill (SKILL.md + README.md)
- OpenClaw-specific CLI commands (`synap init`, `synap connect`, `synap security audit`)
- OpenClaw provider config templates
- Security hardening rules
- Integration tests

### Pros of separate repo
- Clean GitHub presence for OpenClaw community (they see "synap-openclaw", not "synap-backend")
- Independent versioning (skill updates don't need backend deploy)
- Can be listed on ClawHub as a standalone project
- README/docs focused on OpenClaw users, not Synap internals

### Cons of separate repo
- Version drift risk (skill references API endpoints that might change)
- More repos to maintain (already have 6)
- The skill is literally 1 file — a whole repo for 1 file is overhead
- The CLI commands depend on `@synap/sdk` anyway

### Recommendation: **Don't create a separate repo yet**

The skill lives in `synap-backend/skills/synap/`. It's published via raw GitHub URL. That's enough for ClawHub listing.

The CLI lives in `synap-intelligence-service/apps/cli/`. Adding OpenClaw commands there is simpler than a new repo.

**When to create synap-openclaw:** When we have 5+ OpenClaw-specific files (skill + CLI commands + provider configs + tests + docs) and the co-location creates confusion. Not today.

## 5. The Context Injection Opportunity

The north star doc mentions this but it deserves emphasis. When we're the AI provider:

```
OpenClaw sends: "What's the status of Project X?"
  ↓
Pod receives /v1/chat/completions
  ↓
BEFORE routing to model:
  → Search pod for "Project X" entities
  → Find: Project entity + 3 linked tasks + 2 related people
  → Inject into system prompt: "Context from knowledge graph: Project X has..."
  ↓
Model sees: query + structured context
  ↓
Response is BETTER than any other provider because we have the data
```

This is the moat. No other AI provider has access to the user's structured knowledge graph. Claude API doesn't know about your projects. OpenRouter doesn't know about your contacts. Synap does.

This turns the IS from a "cheaper routing proxy" into a "smarter AI because it has your context." That's the real value prop.
