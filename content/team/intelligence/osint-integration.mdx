# OSINT Dashboard + Synap Integration

## Vision

The OSINT Dashboard becomes a **first-class Synap app** — a standalone product that uses Synap pods as its backend, while its blocks (Map, Telegram, Polymarket, etc.) become installable widgets available across any Synap workspace dashboard.

Two integration surfaces:
1. **Standalone App** — OSINT Dashboard connects to a Synap pod for persistence, auth, and AI
2. **Widget Package** — OSINT blocks ship as a Synap marketplace package, installable on any dashboard

## Integration Path 1: OSINT Blocks as Synap Widgets

### How Synap Widgets Work

Synap has two widget integration models:

#### Option A: Native Cells (React components)

Register directly into the cell runtime. Full access to Synap hooks, stores, and navigation.

```typescript
// @osint/widgets/src/map-cell.tsx
import { registerCell } from "@synap/cell-runtime";
import { MapBlock } from "./blocks/map/MapBlock";

registerCell({
  key: "osint-map",
  component: OSINTMapCell,
  meta: {
    name: "OSINT Map",
    description: "Live geopolitical intelligence map with 8 data layers",
    icon: "Globe",
    category: "productivity",
    displayModes: ["compact", "medium", "full"],
    defaultSize: { w: 6, h: 4 },
    minSize: { w: 3, h: 3 },
  },
  settingsComponent: OSINTMapSettings, // Config form for bento edit mode
});
```

Cell receives these props:
```typescript
interface CellProps {
  cellProps?: Record<string, unknown>;  // e.g. { visibleLayers: ["conflicts", "fires"] }
  config?: Record<string, unknown>;     // Bento block config
  workspaceId?: string;
  blockId?: string;
  isEditing?: boolean;
}
```

And can use the runtime:
```typescript
const runtime = useCellRuntime();
// runtime.navigate({ type: 'entity', id: '...' })  — open entity detail
// runtime.openPanel('entity-detail', { entityId })  — open side panel
// runtime.toast('Intel captured', 'success')        — show notification
```

#### Option B: Iframe Widgets (Sandboxed, any tech stack)

For blocks that are hard to port to React or need isolation. Uses postMessage SDK:

```html
<!-- Widget rendered in sandboxed iframe -->
<script>
  window.SynapWidget.onInit((config, context) => {
    // config = { visibleLayers: ["conflicts", "fires"], ... }
    // context = { workspaceId: "ws_123" }
    renderMap(config);
  });

  window.SynapWidget.onUpdate((config) => {
    updateMapLayers(config);
  });

  // Tell host app to resize
  window.SynapWidget.resize(600);

  // Navigate host app
  window.SynapWidget.navigate({ type: "entity", id: "ent_123" });

  // Show notification in host
  window.SynapWidget.toast("New earthquake detected", "info");
</script>
```

**SDK Protocol** (postMessage):
- Host → Widget: `synap:init`, `synap:update`
- Widget → Host: `synap:ready`, `synap:resize`, `synap:navigate`, `synap:toast`, `synap:error`

### Which OSINT Blocks Map to Which Model

| Block | Integration Model | Why |
|-------|------------------|-----|
| Map | **Iframe** | Complex (Leaflet, 8 APIs, 6k+ lines). Isolate it. |
| Telegram | **Native cell** | DOM scraping → can use React + fetch proxy |
| Polymarket | **Native cell** | Pure API data → React renders |
| Stock | **Native cell** | API + chart → React + Chart.js |
| Feed (RSS) | **Native cell** | Fetch proxy + render |
| Subdomain | **Native cell** | API call + table render |
| Discord | **Native cell** | API + message list |
| Browser | **Skip** | Synap already has a full browser |
| Chat | **Skip** | Synap has channels |
| Clock/Countdown | **Skip** | Synap has clock widget |
| TradingView | **Iframe** | Already an embed |
| Twitter | **Native cell** | Browserless API + render |
| Notion | **Iframe** | Complex editor, isolate |

## Integration Path 2: Standalone App with Pod Backend

### What Changes in the OSINT Dashboard

Replace JSON file storage with Synap pod:

| Current (JSON files) | Synap Pod |
|----------------------|-----------|
| `server/data/dashboards.json` | Entities (profile: `dashboard`) |
| `server/data/users.json` | Kratos sessions (pod auth) |
| `server/data/chats.json` | Channels (pod messaging) |
| `server/data/webhooks.json` | Automations (triggers) |
| Discord OAuth | Pod auth (Kratos) or CP auth (Better Auth) |
| Express file CRUD | `@synap/client` tRPC calls |

### Connection Pattern

```javascript
// osint-dashboard/src/synap-client.js
import { createTRPCClient } from "@synap/client";

const podUrl = localStorage.getItem("synap_pod_url");
const client = createTRPCClient({ url: `${podUrl}/trpc` });

// Save dashboard as entity
await client.entities.create.mutate({
  profileSlug: "osint-dashboard",
  title: "Ukraine Monitoring",
  properties: {
    blocks: [...],
    gridConfig: { columns: 24, rows: 16 },
  },
});

// Load dashboards
const dashboards = await client.entities.list.query({
  profileSlug: "osint-dashboard",
});
```

### Auth Flow

1. User opens OSINT Dashboard standalone app
2. App checks for stored pod session (Kratos cookie)
3. If no session → redirect to pod login (same Kratos UI as Synap browser)
4. Authenticated → load dashboards from pod entities
5. All CRUD goes through `@synap/client` → pod tRPC

## API Surface for OSINT Proxy

The OSINT Dashboard's Express proxy server aggregates 8+ external APIs. This can either:

**Option A: Keep as standalone proxy** — OSINT widgets call the proxy directly (simplest)
```
Widget → https://osint-proxy.example.com/api/map/earthquakes → USGS API
```

**Option B: Pod-side automation** — Pod fetches data on schedule, stores as entities
```
Automation trigger (cron: every 5min) → fetch USGS → create/update entities → widgets query pod
```

Option A is recommended for Phase 1 (live data doesn't need persistence). Option B for Phase 3 (historical analysis, AI processing).
