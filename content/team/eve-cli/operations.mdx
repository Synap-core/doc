## Two trees on the server (do not confuse)

| Path (typical) | What it is |
|----------------|------------|
| **`/opt/eve`** | **Eve / hestia-cli** clone — where you run **`pnpm … eve`**. **`.eve/`** state is relative to **this cwd** (or whatever directory you `cd` into when running `eve`). |
| **`/opt/synap-backend`** | **Synap backend** checkout — **`synap install`** uses **`cwd` = repo root** and **`SYNAP_DEPLOY_DIR`** = `…/deploy`. **`docker compose`** for Synap must run from **`…/deploy`**, not from `/opt/eve`. |

Running **`docker compose logs realtime`** from **`/opt/eve`** hits the **wrong compose file** (or no `realtime` service). Always **`cd` to the Synap deploy directory** for Data Pod logs.

## Docker inside LXC / Proxmox CT

Nested containers often need:

- **Nesting** enabled on the CT (`features: nesting=1` in Proxmox).
- Sometimes **keyctl** or **privileged** CT — see host docs.

**Typical error:** `open sysctl net.ipv4.ip_unprivileged_port_start … permission denied` during **`runc` create** — this is **host/CT capability**, not a missing Synap `.env` value. See **[Stack & tools — Docker & LXC](/team/technologies/docker-and-containers)**.

## Synap install: tarball vs git

If the backend tree was extracted from a **tarball** (no **`.git`**), **`synap install`** must still recognize **`deploy/docker-compose.yml`** at the repo root — otherwise it may try to clone into **`/opt/synap-backend/synap-backend`**. Backend **`synap`** script encodes this; Eve sets **`SYNAP_REPO_ROOT`** and **`SYNAP_DEPLOY_DIR`** so installs anchor correctly. Details: [Synap integration](/team/eve-cli/synap-integration).

## Logs cheat sheet

| Goal | Where |
|------|--------|
| Synap **realtime** build failure | Build output of **`docker compose build realtime`** from **`synap-backend/deploy`** — container may not exist yet; **`docker compose logs realtime`** only after **`up`**. |
| Synap stack | `cd /opt/synap-backend/deploy && docker compose logs <service>` |
| Eve-side services | Compose files under Eve repo / organ commands — check **`eve`/`README`** for service names |

| Hermes Agent (Builder) | `docker logs eve-builder-hermes` (or corresponding container name) |

| Ollama (Brain) | `docker logs eve-brain-ollama` |

## Agent operational concerns

### OpenClaw (Arms)

- **Startup**: OpenClaw container pulls latest image and starts. On first boot, it reads `SYNAP_API_URL` and `SYNAP_API_KEY` from its environment (injected from `.eve/secrets/secrets.json`).
- **Health check**: Docker healthcheck pings OpenClaw's `/health` endpoint. If unhealthy, container restarts.
- **Logging**: OpenClaw logs to stdout/stderr; Docker collects them via `docker logs`.
- **Messaging devices**: WhatsApp/iMessage/Telegram connections are managed externally. OpenClaw does not create them — the operator (or Synap platform) provisions them and points OpenClaw at them.

### Hermes Agent (Builder)

- **Startup**: Hermes Agent container starts with Synap API credentials. It does NOT connect to messaging channels.
- **Cron polling**: Hermes uses its built-in cron to periodically poll Synap REST API for queued tasks. Default interval is configurable (documented in Hermes Agent config).
- **Subprocess management**: When spawning the codegen tool (opencode or OpenClaude), Hermes runs it as a child process. If the codegen tool crashes or hangs, Hermes's process manager handles timeout and error recovery.
- **Resource isolation**: Codegen tool runs with limited CPU/memory (via Docker cgroups or `ulimit` when invoked outside Docker) to prevent runaway resource consumption.

### Codegen tool (opencode / OpenClaude)

- **Lifespan**: Short-lived. Spawns, executes task, exits. Does not persist between invocations.
- **Statelessness**: Does not maintain its own database. All state comes from command-line arguments and environment variables.
- **Output capture**: stdout/stderr are captured by the calling process (Hermes). No separate codegen tool log files.

## Dashboard sidecar

The Eve dashboard (Next.js, port **7979**) ships with a small WebSocket sidecar process for terminal traffic, listening on port **3041**. Source: `hestia-cli/packages/eve-dashboard/server/terminal-sidecar.ts`.

- **Auth:** JWT on `eve-session` cookie or `?token=<jwt>` query param. HS256, secret = `secrets.dashboard.secret` (shared with Next).
- **Why a separate process:** Next.js App Router can't host WebSocket upgrades inside Route Handlers without a custom server. Splitting the WS surface keeps Next boring.
- **Logs:** stdout/stderr — captured by whatever supervisor starts it (`pm2`, `systemd`, `docker logs <dashboard container>`).
- **Dev:** `pnpm --filter eve-dashboard dev:sidecar` runs alongside `pnpm --filter eve-dashboard dev`.
- **Bind:** `localhost` by default. Exposing requires explicit operator action (Traefik route, port-forward, etc.).

See [Dashboard](./dashboard.mdx) for the full terminal-kind taxonomy (Subprocess REPL / Log Tail / Recipe Runner).

## Dokploy operations (optional)

When Dokploy is enabled:

- **Docker management**: Dokploy's Docker socket is mounted into the Dokploy container. Eve's Legs layer (Traefik) may or may not route through Dokploy depending on configuration.
- **Service discovery**: deployed services (via codegen tool → Dokploy) are discoverable by their service URLs in Synap file entities.
- **Scaling**: Dokploy manages container lifecycle for deployed services. Eve does not directly manage these containers.
- **Logs**: Dokploy service logs are managed through Dokploy's own interface, not Docker logs directly.
- **Shutdown**: `dokploy stop` halts all services. `dokploy start` restarts them. No separate `eve builder stop` — this is a Dokploy lifecycle command.
