# 02 — Entity Profile Extensibility

> **Status 2026-03-25**: Profile packs Phase 1 complete. CP serves `profile-pack` packages; browser fetches + installs with local fallback; installed packs tracked in workspace settings; `composesFrom` dependency merging wired. Phase 2 (automation-driven behaviors) and Phase 3 (property enhancements) are next.

### Profile Creation Paths

1. **System profiles** — `ensure-system-profiles.ts` on workspace init (backend job)
2. **Template profiles** — `createFromDefinition` API, creates profiles from `WorkspaceProposal`
3. **AI-created profiles** — `propose_workspace` tool or `profile.create` via Hub Protocol
4. **User-created profiles** — Profiles app in browser (manual creation)

## Limitations

### 1. Profile Discovery is Poor
- No marketplace or gallery of profile types
- Users don't know what profiles are available or what they can create
- No "profile pack" concept — you either get system profiles or define them in a template
- No way to import a single profile type from another workspace or template

### 2. Profile Properties Are Flat
- Properties are key-value JSONB — no nested objects, no computed fields, no formulas
- No property inheritance (parent profiles share slug namespace but not properties in practice)
- No property validation beyond basic types (no regex, no range, no conditional required)
- Property `constraints` field exists in schema but is not enforced in the UI

### 3. No Profile Behaviors
- Profiles define data shape but not behavior
- No "when task status changes to done, set completedAt" triggers
- No default views per profile type
- No profile-specific AI skills (e.g., "when creating a book entity, ask for author and genre")

### 4. No Profile Composition
- Can't say "a book IS a note WITH extra properties (author, genre, rating)"
- Parent profile inheritance exists in schema but is limited in practice
- No mixin concept — can't add "taggable" or "rateable" to any profile

### 5. No Profile Marketplace
- Control plane marketplace exists for workspace templates but not individual profiles
- No way to share a "book" profile type across users
- No versioning of profile definitions

### 6. Property Rendering Gaps
- Property renderer handles ~20 value types but some are basic
- No custom renderer per property (e.g., star rating for "rating" field)
- No conditional visibility (show field X only if field Y = value)

## What Needs to Change

### Near-term: Profile Packs
Instead of monolithic workspace templates, support composable pieces:

```typescript
interface ProfilePack {
  id: string;
  name: string;              // "Knowledge Management"
  description: string;
  profiles: ProfileDefinition[];
  views?: ViewDefinition[];  // Optional suggested views
  relations?: RelationDefinition[]; // Optional suggested relations
  displayTemplates?: DisplayTemplate[];
}
```

A workspace template becomes a composition of profile packs:
```typescript
const knowledgeWorkspace = compose(
  systemProfiles,           // note, task, project (always included)
  profilePack('books'),     // book, author, genre
  profilePack('music'),     // album, artist, playlist, dj-session
  profilePack('learning'),  // course, lesson, flashcard
);
```

### Near-term: Profile Behaviors (via Automation Triggers)
Connect profiles to the automation system:

```typescript
// When entity of profile "book" is created, trigger skill "book-reflection"
{
  trigger: { type: 'entity.created', profileSlug: 'book' },
  action: { type: 'skill.execute', skillId: 'book-reflection' }
}
```

This bridges to the **event-triggered skills** gap (Bridge #1).

### Medium-term: Profile Property Enhancements
- Computed fields: `fullName = firstName + ' ' + lastName`
- Conditional required: `if status === 'closed' then closedAt is required`
- Custom renderers: `rating` property → star rating component
- Property groups: organize properties into sections in the entity detail view

### Long-term: Profile Marketplace
- Share profile packs on control plane marketplace
- Version profiles (migration when schema changes)
- Community-contributed profile packs

- [x] `createFromDefinition` already accepted packs alongside templates (existing `workspaceId` path); no change needed

- [x] Installed packs tracked in `workspace.settings.installedPacks` JSONB array
- [x] `composesFrom` dependency merging implemented in `ProfilePackBrowserCell.handleInstall`

### Phase 2: Automation-Driven Profile Behaviors
- [ ] Connect automation triggers to profile events (entity.created, entity.updated)
- [ ] Define profile-specific AI skills that auto-attach when profile is added
- [ ] Create "profile setup wizard" — AI asks questions to configure properties

### Phase 3: Property Enhancements
- [ ] Enforce `constraints` field in UI (currently ignored)
- [ ] Add computed field support
- [ ] Add conditional visibility for properties
- [ ] Custom property renderers (star rating, color picker, progress bar)

### Research Questions
- [ ] How does Notion handle custom property types? (relation, rollup, formula)
- [ ] How does Airtable handle field types and computed fields?
- [ ] How does Linear handle custom fields vs built-in fields?
- [ ] Should profile inheritance be reworked? (current parent system is underused)
- [ ] How to handle profile schema migrations when a pack is updated?
