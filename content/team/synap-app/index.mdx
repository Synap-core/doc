Target Users: Frontend and full-stack engineers working on Synap UI, platform contributors who touch shared packages, product and design teammates needing package-level orientation.

Marketing: The shared frontend package monorepo that powers every Synap host surface — from the Electron Browser desktop app to mobile and web — through a composable system of UI tokens, data client, view adapters, and feature apps.

Key User Stories:
- **As a frontend engineer**, I want a single monorepo with all shared packages (UI system, data client, hooks, stores, cells, views) so I can build any feature without cross-repo dependency dance.
- **As a platform contributor**, I want clear package boundaries and a three-layer data pattern (hooks → stores → client) so I can modify UI without breaking tRPC contracts or introducing upward imports.
- **As a full-stack engineer**, I want to understand how frontend integrates with the Intelligence Service (tRPC + Socket.IO bridge) so I can wire AI chat, proposals, and real-time events end-to-end.
- **As a product designer**, I want the ui-system package to be the source of visual truth (tokens, components, spacing) so I can prototype using the same design language the codebase ships.
- **As a new contributor**, I want a status-and-scope doc that clearly marks what is active vs deprecated in the repo so I don't waste time on dead code paths like Telegram or legacy web.

Current Status: The shared frontend packages in `synap-app` are fully active. The Electron Browser is the primary user surface; `apps/web`, `apps/web-desktop`, `apps/telegram`, and extensions are hosts that consume the same packages. Telegram app in this repo is deprecated.

## Overview

`synap-app` is the frontend package monorepo for Synap. It contains no monolithic app — instead it ships ~80+ npm packages: a design system (`ui-system`), data client (`client`, `hooks`, `stores`), cell system (`cell-runtime`, `capabilities`), view adapters (`table`, `kanban`, `calendar`, `masonry`, `flow`, `graph`, `map`), feature modules (`ai-chat`, `channels`, `proposals`, `connectors`, `capture`), and apps (`chat`, `intelligence`, `whiteboard`, `relay`, etc.). All host surfaces (Browser Electron, web, extensions, Relay) consume the same packages.

This team space holds product context about the repo: status, package boundaries, and frontend ↔ Intelligence Service integration. The canonical package map lives in **[Synap Packages](/team/synap-packages)**.

## Architecture

```
Host (Browser Electron / web / extension / Relay)
  │
  ├─ @synap-core/ui-system        (Tamagui provider, tokens, base components)
  ├─ @synap-core/client           (tRPC + TanStack Query transport to pod)
  ├─ @synap-core/hooks            (queries + stores + UX rules)
  ├─ @synap-core/stores           (Zustand ephemeral state)
  ├─ @synap-core/cell-runtime     (CellRenderer dispatcher)
  ├─ @synap-core/view-renderer    (view type dispatcher + shared chrome)
  ├─ @synap-core/{entity-card, entity-views, property-renderer}
  ├─ @synap-core/channels, chat-interface, notifications
  ├─ @synap-core/ai-chat          (full AI chat app module)
  └─ @synap-core/{platform, auth, pod-connect, pod-failover}
```

**Tech stack**: React 19, Tamagui, TanStack Query, Zustand, tRPC 11, Socket.IO client, Electron (as host).

**Three-layer data pattern** is mandatory: Components use `@synap-core/hooks` → `@synap-core/stores` → `@synap-core/client`. Never import client directly from UI packages. Never skip a layer.

**Cell system**: `@synap-core/capabilities` is the manifest (zero React) consumed by agents, scripts, and backend. `@synap-core/cell-runtime` is the registry + dispatcher. Cells render via hosts (BentoCellHost, PanelCellHost, PageCellHost).

## Package Map / Structure

The canonical package map is in **[Synap Packages](/team/synap-packages)**. This repo's top-level structure:

- `packages/core/*` — Foundational packages: UI system, data client, cells, views, profiles, channels, notifications, auth, platform, native-OS bridges.
- `packages/features/*` — Feature modules: AI chat, sidebar, floating panels, capture, proposals, connectors, document, entity-views, agent-service, platform-auth.
- `packages/views/*` — View adapters behind `view-renderer`: table, kanban, list, grid, masonry, calendar, timeline, flow, graph, map.
- `packages/apps/*` — Embeddable feature apps: chat, intelligence, whiteboard, relay, telegram, extension, agent-os, zeroclaw.
- `packages/sdk`, `packages/react`, `packages/core/database`, `packages/core/data-utils` — SDK surface, React providers, client-side DB helpers, data transforms.
- `packages/ipc-contract`, `packages/core/deep-link-constants` — Cross-process contracts.
- `apps/web`, `apps/telegram` — Host shells (web and telegram — legacy).

See **[Synap Packages → package map](/team/synap-packages)** for the full package table with npm names and roles.
