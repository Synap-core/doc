## 1. What We're Building

A complete personal AI agent system for a single user on their own data pod. The user gets:

- A **personal agent** that knows them deeply, improves over time, and can express multiple specialist personalities — one continuous relationship
- An **optional agent workspace** for staging — the agent can work freely without cluttering the user's space, but this is a convenience, not a requirement
- A **proposal system** where agent writes in the user's workspace go through approval by default — same RBAC as users, one extra default
- **Branching** — the personal agent spawns specialist branches for parallel work; branches communicate via A2AI channels
- **Independent agents** — OpenClaw, custom agents, or marketplace agents connected via Hub Protocol, each with their own permission scope
- **The UX** — session continuity, tool disclosures, branch cards, memory inspector, proposal cards, agent workspace panel

---

## 2. Architecture Principles (Non-Negotiable)

See `docs/AGENT_MODEL.md` for the full spec. Key principles:

1. **Three strict layers.** Backend = data + permissions. Intelligence service = AI logic. Frontend = UI. Nothing crosses.
2. **Agent = user with proposal default.** Agents use the same RBAC as humans. The only difference: `writesRequireProposal: true` by default.
3. **Agent workspace is optional staging, not fundamental.** Agents operate in the permission model. A dedicated workspace is a convenience for isolation.
4. **One personal agent, multiple personalities.** The user has one AI. Personalities (OCto, CMO, etc.) are role overlays on the same agent — same memory, same relationship.
5. **Data scope is first-class.** System (pod-wide) / shared (multi-workspace) / workspace / user — visible and configurable, not just a backend detail.
6. **A2AI channels for agent-to-agent communication.** Agents don't share workspaces to communicate — they use channels.

## 4. Agent Workspace Pair (Revised)

### 4.1 The Optional Staging Pattern

The personal agent's dedicated workspace is **one useful default** — not an architectural requirement. The system should support:

- **No staging workspace:** Agent proposes everything in the user's workspace. Simple, no extra navigation.
- **Dedicated staging workspace:** Agent has owner role. Direct writes within that workspace. User browses and "promotes" items via entity transfer. This is what Phase 1 creates.
- **Future:** Project workspaces where agents are granted member/admin role for scoped work.

The FirstTimeSetupFlow already offers the staging workspace as an opt-in (not default). That's correct.

### 4.2 Auto-Creation Flow

When the user opts in, or when the personal agent needs to stage work for the first time:

1. Check if a workspace with `workspaceType: "agent"` and `linkedAgentId: personalAgentId` already exists
2. If not: create it with `workspaceName: "[User's agent name]'s Workspace"`, type `"agent"`, linked to the agent
3. Add the agent as `owner` role member of that workspace
4. Add the user as `admin` role member (oversight without being the primary owner)
5. Cache the `agentWorkspaceId` in the agent's session context

### 4.3 Permission Model

| Actor | In agent's staging WS | In user's personal WS |
|-------|----------------------|-----------------------|
| Agent | Full CRUD (owner role) | Read + propose (default) |
| User | Full CRUD (admin role) | Full CRUD (owner role) |

This falls out of the RBAC system naturally — no special cases needed.

### 4.4 Entity Transfer (Staging → Personal)

```
POST /entities/:id/transfer-workspace
Body: { targetWorkspaceId: string }
```

Permission: user must have write access to both workspaces. No proposal needed (user is doing it directly). Entity's `workspaceId` is updated; no data copy.

## 5. The Personal Agent

### 5.1 One Agent, Multiple Personalities

The user's personal agent is **one entity** with a single compacted state. When the user invokes a specialist role (or the agent decides to shift mode), a personality overlay is applied at the prompt assembly stage:

```
Base identity (from compacted state):
  "[Agent name]. Direct communication style. Deep context about the user."

Personality overlay (applied at session start or on mode switch):
  "You are currently in [OCto/CMO/Analyst] mode.
   Focus: [role-specific instructions].
   Tools: [role-specific tool set].
   Everything else about you and the user remains the same."
```

Personalities are **not separate agents** in the database. They are configurations stored in `agent_configs` under the personal agent's ID, applied dynamically.

### 5.2 Agent Configuration

```typescript
interface PersonalAgentConfig {
  name: string;
  basePersonality: string;
  primaryModel: string;
  branchModel: string;
  compactionModel: string;

  // Workspace (optional staging)
  agentWorkspaceId?: string;
  defaultTargetWorkspaceId: string;

  // Permissions
  autoApproveFor: string[];          // default: ["search.*", "memory.*", "entity.read", "document.read"]

  // Branching
  enableBranching: boolean;
  maxBranchDepth: number;            // default: 3
  maxParallelBranches: number;       // default: 5

  // Personalities
  personalities: PersonalityConfig[];
}

interface PersonalityConfig {
  id: string;                        // e.g. "octo", "cmo", "analyst"
  name: string;                      // display name
  description: string;               // when to use this mode
  personalityOverride: string;       // prompt overlay
  toolOverrides?: string[];          // restrict/expand tools for this mode
  modelOverride?: string;            // use different model for this mode
}
```

### 5.3 Tool Context

The intelligence service assembles tool context per-session. Required fields for Phase 1:

```typescript
interface ToolContext {
  userId: string;
  agentUserId?: string;
  channelId: string;
  sessionId: string;
  workspaceId: string;               // channel's workspace
  agentWorkspaceId?: string;         // staging workspace if configured
  personalWorkspaceId: string;       // user's primary workspace
  sourceMessageId?: string;
  dataPodUrl?: string;
  dataPodApiKey?: string;
  currentContext?: {                 // ← MISSING, needed for Phase 1
    currentPage?: string;
    deviceType?: "desktop" | "mobile";
  };
}
```

## 6. Independent Agents

Independent agents use the same RBAC + proposal default. They connect via Hub Protocol using API keys scoped to the pod.

### 6.1 Registration

```
POST /intelligence/register
Body: { agentType, name, capabilities, defaultScope }
Returns: { agentUserId, apiKey }
```

The agent is created as a user with `userType: "agent"`, `isPersonalAgent: false`, added as a workspace member at the configured scope.

### 6.2 Scope Options

- `workspace` — added as member to a specific workspace only
- `user` — can access all workspaces the user owns (useful for assistants)
- `pod` — deprecated/not recommended for Phase 1; too broad

### 6.3 Communication via A2AI Channels

Agents communicate through channels, not through shared workspace access. The A2AI channel type (`channel_type: "a2ai"`) is already in the schema. Agent-to-agent messages go there.

## 7. Branching System

Branches are channels (`channel_type: "BRANCH"`) with `parentChannelId` set. No separate branch table.

### 7.1 Spawn Flow

```
1. Orchestrator calls create_branch tool
2. Intelligence service creates channel: { type: BRANCH, parentChannelId, branchPurpose, agentType }
3. Assemble branch bootstrap:
   - Identity: parent compacted state identity + specialist overlay
   - User model: full (same user)
   - Continuity: task-specific context from parent's recent messages
   - Goals: the specific branch task
   - Entity context: entities relevant to the task
4. Start session on branch channel
5. Post branch_spawn system message to parent (for UX)
6. Branch agent runs ReAct loop independently
```

### 7.2 Branch Communication

When multiple branches run in parallel, an A2AI channel links them:

```
1. Orchestrator spawns Branch A and Branch B
2. Creates A2AI channel linking both
3. Both branches get context: "You are working in parallel with [other branch]"
4. Branches use send_to_branch / check_branch_messages tools
5. Cross-communication indicator shown in parent UX
```

### 7.3 Completion and Compaction

```
1. Branch finishes → writes resultSummary to channel
2. Sets channel.status = "MERGED"
3. Sends branch_result system message to parent channel
4. Orchestrator synthesizes for user
5. On next parent session compaction: branch summaries included in continuity block
```

## 13. Open Questions

1. **Agent workspace discoverability:** Show in main workspace switcher, or only via chat interface? (Current: opt-in in onboarding, accessible as normal workspace)

2. **Personality persistence across sessions:** Does switching to "OCto mode" persist until explicitly switched back, or reset per session? (Recommendation: persist until reset)

3. **Branch agent models:** Should branches always use a cheaper model, or should critical branches use the primary model? (Recommendation: configurable per personality/specialist type)

4. **Agent workspace cleanup:** Auto-archive old items after N days, or let the agent manage its own cleanup? (Recommendation: agent manages, with a user-accessible "archive old items" action)

5. **`isPersonalAgent` auto-provisioning:** When is the personal agent created? On pod setup? On first message? On opt-in? (Recommendation: on first message that would benefit from AI, lazy creation)
