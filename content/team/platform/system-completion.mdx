## Completion Tracks

### Track 1 ‚Äî PLAN-03: Event-Triggered Skills
**Priority: #1 ‚Äî Highest remaining product impact**

The "magic moment": AI automatically reacts when data changes. Infrastructure is 90% there (automation system, triggers, commands, skills all exist). The gap is the bridge connecting them.

**What's missing:**
- `skill_triggers` DB table (or reuse existing `automation_triggers` with `actionType: 'skill'`)
- Backend: when trigger fires, resolve skill ‚Üí run IS endpoint
- IS: skill execution endpoint `/skills/trigger` (Hub Protocol route)
- Frontend: skill trigger configuration UI in automation editor
- Frontend: `SkillTriggerWidget` bento cell (shows active skill triggers)

**What's already there:**
- `automation_triggers` table with event type matching
- `pg-boss` workers that process trigger events
- 13 system skills loadable by IS
- Hub Protocol: IS ‚Üî Backend REST fully wired
- `skill-loader.ts` in IS handles skill routing

**Effort:** ~3-4 days
**Unlocks:** The entire "AI reacts to your data" narrative. Demo moment: create a task ‚Üí AI automatically enriches it via skill.

### Track 2 ‚Äî Onboarding Flow Redesign
**Priority: #2 ‚Äî Blocks user acquisition**

Current `FirstTimeSetupFlow` in browser is a series of modals. Target: `@synap/onboarding-flow` unified package with 4 steps: Hook ‚Üí FirstCapture ‚Üí Connect ‚Üí Done.

**What's missing:**
- `@synap/onboarding-flow` package (new)
- Hook screen: value prop + 1-minute promise
- FirstCapture: capture anything (text/URL/file) in 30 seconds
- Connect screen: install 1 profile pack (from CP pack browser)
- Done screen: first bento dashboard seeded
- Profile Pack Phase D: pack selection wired into onboarding step 3
- Replace `FirstTimeSetupFlow` in browser with the new package

**What's already there:**
- Profile pack browser cell (CP fetch + offline fallback) ‚Äî PLAN-02 done
- `createDefaultHomeBentoConfig()` seeds dashboard
- Capture flow (tab saving, selection capture) exists
- Settings WsPacksTab shows installed packs

**Effort:** ~4-5 days
**Unlocks:** Clean entry into the product. Retention cliff fix.

### Track 4 ‚Äî Vault AI: Secret Request Flow
**Priority: #4 ‚Äî Completes vault integration (~85% done)**

The vault is built. AI can read secrets it has been pre-authorized for. The missing piece: AI asks for a secret it doesn't have access to yet, creating a reviewable request.

**What's missing:**
- `request_secret` tool in IS tool registry
- Hub Protocol route: `POST /vault/request-secret` (creates pending vault request)
- `vault_requests` table (or reuse proposals with `proposalType: 'vault_access'`)
- Frontend: `AIRequestCard` component (banner variant) ‚Äî shows "AI needs GITHUB_TOKEN" with [Provide] [Deny]
- Notification type `ai_request.vault_access` in registry (requires Track 3)
- Vault side panel integration: [Provide] opens vault with secret pre-selected

**What's already there:**
- Vault built, secret storage, per-secret permissions
- `checkPermissionOrPropose()` pattern available for reuse
- PLAN-05 architecture already defines `AIRequestCard` component
- `navigatePanel('vault', { secretId })` navigation is available

**Effort:** ~2 days
**Unlocks:** Full AI ‚Üî vault interaction loop. AI becomes self-sufficient with user approval.

### Track 5 ‚Äî Terminal Bridge (PLAN-04: Remote Execution)
**Priority: #5 ‚Äî Required for DevOps workspace**

AI can run commands. The gap: no Hub Protocol route for terminal operations, no terminal widget that AI can target, no security policy for command authorization.

**What's missing:**
- Hub Protocol: `POST /terminal/execute` (creates session + runs command)
- Hub Protocol: `GET /terminal/sessions/{id}` (poll output)
- `ai_request.terminal_exec` proposal type ‚Äî every command goes through `checkPermissionOrPropose()`
- IS tool: `execute_command(command, cwd, workspaceId)` ‚Üí Hub Protocol ‚Üí pod terminal
- `TerminalOutputCell` bento widget ‚Äî streams output from a session
- DEFAULT_AUTO_APPROVE: opt-in terminal commands (read-only shell ops only)

**What's already there:**
- Terminal service in Electron main (pty, sessions, IPC)
- `terminal:create`, `terminal:write`, `terminal:kill` preload APIs
- IS Hub Protocol router wired for 20 sub-routers (easy to add)
- `checkPermissionOrPropose()` reusable for command approval

**Effort:** ~3-4 days
**Unlocks:** DevOps workspace (git status, npm run, docker ps). AI debugging loop.

## Definition of "System Complete"

The system is complete (v1 shippable) when:

1.  A user can capture anything (browser, Telegram, manual) and it lands in Library
2.  AI can organize captures into entities, grouped by profile
3.  AI mutations are reviewable via real-time bell with inline approve/reject
4. ‚ùå **AI reacts to data changes** (event-triggered skills ‚Äî Track 1)
5. ‚ùå **New users get in smoothly** (onboarding redesign ‚Äî Track 2)
6.  External data flows in via connectors (code done, needs deploy ‚Äî Track 6)
7. ‚ùå **AI can request vault secrets with user approval** (Track 4)
8. ‚ùå **AI can run terminal commands with user approval** (Track 5)
9.  Notifications reach OS (Electron IPC fixed)
10.  **All notification types delivered** (DB table + registry + NotificationService + toasts + banners + prefs + proactive integration ‚Äî Track 3 DONE 2026-03-28)

**Current score: 8.5/10.** Remaining: onboarding (2), connectors deploy (6), vault grant dispatch (4), terminal bridge (5).

**Launch readiness: ~1 day.** As of 2026-03-26, all transactional routes are implemented, security is correct, billing redirects are proper 302s. Only blocking items: 2 CP API invite endpoints + invite email + Browser first release (v0.5.0).
