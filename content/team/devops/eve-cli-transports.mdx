# Eve CLI Transports

> **TL;DR** — `eve auth provision`, `eve doctor`, the lifecycle hooks
> and every other CLI command that touches Synap pick their HTTP base
> URL based on **where the CLI runs**, not what's stored in
> `secrets.json`. On the pod host it goes through a loopback-only port
> Eve published in `docker-compose.override.yml`; off-host it goes
> through the public Traefik URL. The runner is plain `fetch` either
> way. There is no `docker exec` in the happy path.

## Why this exists

For a long time the on-host CLI tried to reach synap-backend through
its **public** URL — `https://pod.<domain>/api/hub/...`. That worked
in production when DNS, TLS and Traefik routing were all healthy, but
during initial setup or on disturbed deployments it broke in
spectacular ways:

- DNS not configured yet → `ENOTFOUND`.
- Cert not minted yet → `unable to verify the first certificate`.
- Split-DNS routing the FQDN back through the public IP → host firewall
  drops the SYN, fetch hangs to timeout.
- Traefik mid-restart → 502s with no useful body.

We chased a particularly memorable phantom 404 for two days through
that stack — the symptom was `POST /api/hub/setup/agent` returning
404, the suspicion fell on the backend image being too old, but the
real problem was the BusyBox `wget` we were using inside the container
exec was silently dropping `--header=` arg-form. Two layers of
unrelated brokenness stacked into one inscrutable error.

The fix is the standard pattern every comparable system uses
(Vault, Consul, Caddy admin, K8s apiserver, Postgres, Docker daemon
itself): **bind the admin/API on the host's loopback interface**.
Anyone with access to the box can reach it directly, no certificate
or DNS needed; nothing on the network sees a thing. The CLI just
calls it with plain `fetch`. The same `fetch` works off-host against
the public URL when run from a laptop.

## The two transports

| Where you run `eve` | URL the CLI hits | Source of that URL |
|---|---|---|
| **On the pod host**, container detected, loopback port reachable | `http://127.0.0.1:14000` | Eve's `docker-compose.override.yml` publishes `127.0.0.1:14000:4000` on the synap-backend service. |
| **Off the pod host** (laptop, CI, remote ops) | `https://pod.<domain>` | `domain.primary` in `~/.eve/secrets/secrets.json`, derived in `resolveSynapUrl(secrets)`. |

Selection happens in `resolveSynapUrlOnHost(secrets)` (in
`@eve/dna/src/loopback-probe.ts`). It does a 200ms TCP connect to
`127.0.0.1:14000`; if anything answers, that's our URL. Otherwise it
delegates to `resolveSynapUrl` and gets the public URL. The probe
result is cached for the life of the CLI process, so subsequent calls
are free.

```ts
// Pure derivation — no I/O, no probes. Used for embedding URLs into
// other containers' env files (openwebui-pipelines, sandbox, etc.).
const url = resolveSynapUrl(secrets);

// I/O — probes 127.0.0.1:14000, falls back. Use this from CLI runtime
// code (auth.ts, doctor.ts, lifecycle hooks).
const url = await resolveSynapUrlOnHost(secrets);
```

Container-side code paths use the in-network DNS instead:
`SYNAP_BACKEND_INTERNAL_URL = http://synap-backend-backend-1:4000`.
That's what's written into pipelines / openclaw `.env` files — those
containers ride the `eve-network` bridge and resolve the hostname via
Docker's internal DNS, no public round-trip needed.

## How the override file works

The synap-backend `docker-compose.yml` is owned by the synap-backend
repo and we don't edit it directly — that file is regenerated by
`installSynapFromImage` from a bundled template. To add a host port
without touching upstream YAML, Eve writes a sibling
`docker-compose.override.yml` in the same directory. Compose v2
auto-merges any `*.override.yml` next to the main file on every
invocation; lists like `ports` are concatenated.

```yaml
# Eve-managed docker-compose override — DO NOT edit by hand.
#
# Marker: eve-managed:synap-loopback-override:v1
#
# Adds a loopback-only host port mapping so the on-host Eve CLI can
# reach the synap-backend HTTP API directly via 127.0.0.1:14000…

services:
  backend:
    ports:
      - "127.0.0.1:14000:4000"
```

The `127.0.0.1:` prefix in Docker port mapping is the safety: Docker
binds the listening socket only on the loopback interface, never on
`0.0.0.0`. The host's external network interfaces, the Docker bridge,
and the public internet see nothing on port 14000. Anyone with shell
access to the box (already root-equivalent over `/opt/synap-backend`)
can `curl 127.0.0.1:14000` — that's exactly the surface we want.

The override file is regenerated by:

- `installSynapFromImage` (`@eve/brain`) — first install.
- `runUpdatePlan` for synap (`@eve/lifecycle`) — every `eve update synap`.

Both flows go through `ensureSynapLoopbackOverride(deployDir)` in
`@eve/dna/src/synap-overrides.ts`, which is idempotent:

| Disk state | Action | Side effect |
|---|---|---|
| File missing | Write our content. | Compose picks it up on next `up -d`. |
| File present, our magic marker found | Refresh content. | No-op if bytes match. |
| File present, no marker | Leave alone, log "kept user-owned override". | CLI falls back to public URL transparently. |

The marker is a single literal string —
`eve-managed:synap-loopback-override:v1` — chosen long enough that an
accidental collision with hand-written content is implausible. To opt
out and keep your own overrides, just delete the marker line; Eve
will treat the file as user-owned forever after.

## Why `127.0.0.1:14000` and not `127.0.0.1:4000`

Two reasons:

1. **The synap-app dev server binds 4000 in development.** A developer
   running `pnpm dev` on the same host that runs a synap pod (rare but
   it happens) would have the dev server and the pod fighting for
   port 4000.
2. **The loopback port is an admin/CLI implementation detail.** The
   value just needs to be stable and uncommon. 14000 is empty in our
   deployments and high enough that nothing in our stack collides
   with it.

Mirrored in two places (`@eve/dna/src/components.ts` and
`@eve/dna/src/synap-overrides.ts`) via the `SYNAP_HOST_LOOPBACK_PORT`
constant — keep them in lockstep if either ever changes.

## What about `docker exec`?

We had a `DockerExecRunner` that ran `docker exec node -e fetch(...)`
inside the synap-backend container as the on-host happy path. It worked
but cost an exec spawn (~50–100ms) per call, hand-rolled HTTP framing in
JSON envelopes, and added "what does BusyBox do today" as a permanent
debug surface.

Now `DockerExecRunner` exists only as an `eve doctor` break-glass
diagnostic — a "can the synap container respond on its own loopback
even when the host loopback is broken?" probe. It's never wired into
`auth provision`, normal probes, or any caller that just needs a
working HTTP transport. Don't reintroduce it as the primary path
without a written design doc.

The reasoning is in
[`packages/eve-cli/src/lib/doctor-runners.ts`](https://github.com/Synap-core/hestia-cli/blob/main/packages/eve-cli/src/lib/doctor-runners.ts) —
look for the `buildPodRunner` block.

## Image pruning on update

Same neighborhood, different problem: every `compose pull` accumulates
image layers. After a year of weekly updates a single repo can hold
50+ tags, several GB each. `docker image prune` only collects
*dangling* (untagged) images — it never removes old tags from the
same repo. We've seen pod hosts fill their root disk, OOM-kill
postgres, and lose data because nobody ran `docker rmi` manually.

After every successful `compose up -d` Eve now runs
`pruneOldImagesForRepo(repo, keep=3)` for each image repository in
the component's `pruneImages` list. **Generation-based**, not
time-based: keep the latest 3 tags, drop the rest. In-use images are
auto-protected — `docker rmi` refuses to remove an image backing a
running container, which we surface as "skipped" in the log, never as
an error.

For synap, the prune list is:

```ts
pruneImages: {
  repositories: [
    "ghcr.io/synap-core/backend",
    "ghcr.io/synap-core/pod-agent",
  ],
  keep: 3,
}
```

3 keeps the live image plus 2 rollback steps. Bump the count by
editing the `UpdatePlan` for the component in
`packages/@eve/lifecycle/src/index.ts`. Always scope by repository
prefix (the explicit list above) — never wildcard-prune the whole
host, an over-eager `docker rmi` would yank images from sibling
projects unrelated to Eve.

## Debugging

| Symptom | Likely cause | Check |
|---|---|---|
| `eve auth provision` fails with 4xx pointing at the public URL | Loopback override missing or marker missing | `cat /opt/synap-backend/deploy/docker-compose.override.yml` — looking for the `eve-managed:synap-loopback-override:v1` marker. Run `eve update synap` to regenerate. |
| `eve auth provision` fails with `ECONNREFUSED 127.0.0.1:14000` | synap-backend container not running, or override file present but compose hasn't been up'd since | `docker ps --filter name=synap-backend-backend-1`. If running, `docker compose up -d backend` from `/opt/synap-backend/deploy`. |
| `eve doctor` reports "no synap container detected" on the pod host | synap-backend container isn't visible to docker (different daemon? user not in `docker` group?) | `docker ps -a`. If the container isn't there at all, run `eve add synap` or check who owns the existing install. |
| Disk filling up | Image prune disabled, scoped wrong, or repos changed | `docker images --filter "reference=ghcr.io/synap-core/*" --format "{{.Repository}}:{{.Tag}} {{.CreatedAt}}"` to see what's still around. The next `eve update synap` will reclaim space; manual GC: `docker rmi <ref>` (refused on in-use is normal). |
| `EVE_DEBUG_RUNNER=1 eve <cmd>` shows `using docker-exec` | Stale code (pre-2026-05-04 doctor-runners.ts) | `pnpm -F @eve/cli build && pnpm -F @eve/dna build`. Then re-check. The current code never picks docker-exec for a normal HTTP call. |

If a probe is just *slow*, the loopback path is sub-millisecond by
design — anything slower is the fallback to the public URL kicking
in, which means the override isn't published or the port is shadowed.
Same diagnostic loop.
