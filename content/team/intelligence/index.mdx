# Intelligence

Target Users: Intelligence service engineers, platform/backend engineers integrating AI workflows, operations team handling runtime reliability
Marketing: The AI Agent Hub — OrchestratorAgent, PersonaAgents, skills, and the Hub Protocol that connects AI to your data with governance
Key User Stories:
- "I want an AI co-founder that can read my knowledge, take actions, and propose changes I can review"
- "I need domain-expert agents (researcher, writer, analyst) that work within my workspace's governance model"
- "I want AI to post proactive insights, morning briefings, and health checks into my personal channel"
- "I need my AI agents to talk to my data pod through a stable, documented, permission-controlled API"

## Overview

The Intelligence Service is the **AI execution layer** for Synap — a distributed agent system with two architecture patterns:

1. **Personal AI** — OrchestratorAgent (co-founder, full tools, pod-wide context) + session memory (always on, 4h personal)
2. **Domain expertise** — PersonaAgent (domain expert, branch-only scope, specialized skills like "researcher", "writer", "analyst")

All AI ↔ Pod communication goes through the **Hub Protocol REST** (`/api/hub/*`) — never direct DB access. Permissions are enforced via `checkPermissionOrPropose()` which creates reviewable, reversible proposals for AI mutations.

The Intelligence Service runs on Docker Swarm, autoscales based on load, and connects to the pod via Bearer token + Hub routes. It integrates with LiteLLM for model routing and has a full monitoring stack for agent performance, memory usage, skill health, and execution traces.

```
│                  Intelligence Service                     │

│  │ Orchestrator    │  │ Persona Agents   │  │ Skills  │  │
│  │ Agent           │  │ (researcher,     │  │ Registry│  │
│  │ (co-founder)    │  │ writer, analyst) │  │        │  │
│  └────────┬────────┘  └────────┬─────────┘  └────────┘  │
│           │                    │                           │
│  ┌────────┴────────────────────┴────────────────────────┐ │
│  │              Hub Protocol (21 sub-routers)           │ │
│  │  entities · views · channels · documents · skills   │ │
│  │  automations · monitoring · runtime management       │ │
│  └────────────────────┬─────────────────────────────────┘ │
│                       │ (Bearer token, governance)        │
└───────────────────────┼───────────────────────────────────┘
                        ▼
┌───────────────────────────────────────────────────────────┐
│                   Data Pod (Platform)                     │
│  tRPC · Drizzle · PostgreSQL · event chain · proposals   │
└───────────────────────────────────────────────────────────┘
```

**Deployment:** Docker Swarm with Traefik routing, autoscaling, LiteLLm model routing
