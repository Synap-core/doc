# Eve CLI

Eve is Synap's **sovereign stack orchestrator** — a **TypeScript CLI** (`eve`) and monorepo (**hestia-cli**) that installs and manages a self-hosted stack around the **Synap Data Pod**.

> **The Golden Rule:** Synap is the exclusive source of long-term truth. Every agent reads from and writes to Synap via REST API. No agent maintains its own long-term state (SQLite, Vector DBs, local files). This is non-negotiable.

---

## What Eve Does

1. **Guided installation** — `eve setup` installs a complete stack: Synap Data Pod, inference services, routing, agents, and developer tooling. Profiles: `inference_only`, `data_pod`, `full`.
2. **Organ commands** — `eve brain`, `eve arms`, `eve builder`, `eve eyes`, `eve legs` — each manages a domain of the stack via the organ metaphor.
  3. **Agent wiring** — Configures the **5 Eve-managed agents** (`eve`, `openclaw`, `hermes`, `openwebui-pipelines`, `coder`) to communicate exclusively through Synap's REST API. No agent speaks directly to another. The `coder` agent is one Hub identity that drives one of three engine binaries (`claudecode` / `opencode` / `openclaude`) — engine choice is config (`secrets.builder.codeEngine`), not identity. See [coder](./coder.mdx).
4. **Provider routing** — Stores local tooling preferences (OpenRouter, Anthropic, Ollama) in `.eve/secrets/secrets.json`; optionally syncs non-secret policy via `eve ai sync`.

## Architecture Overview (v2.0)

| Organ | Package | Technology | Responsibility |
|-------|---------|------------|----------------|
| **Brain** | `@eve/brain` | Synap Backend, PostgreSQL, Redis, Ollama | Central memory, API, governance, task queue |
| **Arms** | `@eve/arms` | OpenClaw | Messaging ingress (WhatsApp, iMessage, Telegram), voice memos, reasoning. Fetches memory from Synap, writes back to Synap. Does NOT do coding. |
| **Builder** | `@eve/builder` | Hermes Agent, `coder` engine binary (claudecode / opencode / openclaude), Dokploy (optional) | Headless dev orchestrator. Polls Synap for queued tasks + background intents, spawns the active `coder` engine via subprocess, writes results to Synap. Builder Workspace is seeded automatically on install — see [Builder Workspace](../platform/builder-workspace.mdx). |
| **Eyes** | `@eve/eyes` | RSSHub, Outerbase | Information intake (RSS feeds), database visualization |
| **Legs** | `@eve/legs` | Traefik, Pangolin | Reverse proxy, routing, sovereign tunneling |
| **Add-ons** | compose template | Open WebUI | Optional chat UI wired to Synap IS as OpenAI-compat endpoint. No external DB (SQLite). Includes MCP support via Synap Hub. |

## Data Flow (v2.0)

```
User message → OpenClaw (Arms)
    │
    ▼
Write to Synap via REST API (new message entity or channel update)
    │
    ▼
Synap queues task for Hermes
    │
    ▼
Hermes (Builder) polls Synap cron → reads task → spawns codegen tool via subprocess
    │
    ▼
Codegen tool runs → stdout/stderr captured → written as result back to Synap
    │
    ▼
Synap notifies OpenClaw → OpenClaw replies to user
```

**Key property:** OpenClaw and Hermes never communicate directly. They both talk exclusively through Synap. They are loosely coupled services connected by event-driven queue.

## Golden Rule: Synap as Central Switchboard

Every Eve-managed agent (`eve`, `openclaw`, `hermes`, `openwebui-pipelines`, `coder`) treats Synap as their sole source of truth:

| Layer | What | Where |
|-------|------|-------|
| **Memory** | User state, conversation history, device profiles | Synap PostgreSQL (agent state tables) |
| **State** | Agent sessions, tool results, task queue | Synap PostgreSQL, Redis |
| **Knowledge** | Skills, device mappings, provider routing | Synap workspace settings |
| **Communication** | Agents communicate via Synap task entities and channels | Synap entities + channels |
| **Auth** | API keys per agent | Synap API key system |

No agent runs its own SQLite, Vector DB, or external state store. This applies to OpenClaw (`@eve/arms`) and Hermes (`@eve/builder`).

| `eve setup` three-path flow | Shipped | `inference_only`, `data_pod`, `full` — guided wizard with dry-run |
