# Search

Two-tier search: Typesense for full-text relevance ranking and pgvector for semantic/neural similarity search. Both are composited into a single search experience across all surfaces.

## Architecture

| Engine | Description |
|--------|-------------|
| **Typesense (Full-text)** | Primary search engine. Handles keyword matching, fuzzy search, faceted filtering, and relevance scoring. Documents are synced from PostgreSQL via async workers. **Key tables:** TypeSense index synced from entities, profiles, and custom indexes. **Triggers:** entity create/update/delete triggers reindexing. |
| **pgvector (Semantic)** | Vector similarity search in PostgreSQL. Entities are embedded via the intelligence service and stored in the `entities` table vector column. Used for "find similar" and semantic queries. **Embedding model:** Configurable (supports multiple providers via intelligence service) |

## Search flow

1. **Query** — User types search query in any surface (browser search, Raycast command, CLI search)
2. **Routing** — Frontend determines query intent (keyword vs semantic) and routes accordingly
3. **Typesense query** — Full-text search with faceted filtering by entity type, date, tags
4. **pgvector query** — Semantic similarity search (if enabled) for conceptual matches
5. **Result composition** — Results from both engines are deduplicated, reranked, and merged
6. **Presentation** — Results shown as entity cards in the search UI, filtered by workspace access

## Key files

- Search API: `/api/search` in synap-team-docs (Next.js route for docs search)
- Typesense sync: backend search indexing workers
- pgvector: PostgreSQL vector extension, embedding generation in intelligence service
- Frontend: search cell in `@synap/core/search`

## Configuration

- **VECTOR_SEARCH_ENABLED** env var — set `false` on shared pods to disable pgvector (Typesense-only mode)
- **Typesense configuration** — Managed via docker-compose; index creation handled on startup

## Typesense Configuration

The Typesense index schema maps directly from the `entities` Drizzle table with these field types:

| Typesense Field | Source Column | Field Type | Tokenization |
|-----------------|---------------|------------|--------------|
| `id` | `entities.id` | `string` | No (primary key) |
| `entity_type` | `entities.profile_slug` | `string` | No (filterable enum) |
| `title` | `entities.name` | `string` | Yes (faceted, sortable) |
| `description` | `entities.profile_data` | `string` | Yes (full-text) |
| `content` | `entities.profile_data` | `string` | Yes (full-text) |
| `created_at` | `entities.created_at` | `date` | No (filterable) |
| `updated_at` | `entities.updated_at` | `date` | No (filterable) |
| `workspace_id` | `entities.workspace_id` | `string` | No (filterable) |
| `tags` | `entities.profile_data.tags` | `string[]` | Yes (faceted) |
| `status` | entities.status fields | `string` | No (filterable) |
| `score` | computed at query time | `float` | No |

**Fuzzy matching** is enabled with `max_typos` set to 2 for short queries (under 6 chars) and 1 for longer queries. `prefix` matching is set to `false` for exact term matching on entity types but `true` for free-text queries.

**Synonyms** are not statically configured — Typesense leverages its built-in stemming and phonetic matching. Custom synonym lists can be injected via the Typesense API for domain-specific terms (e.g., medical or legal terminology).

**Faceting** is enabled on `entity_type` and `workspace_id` for faceted navigation in the UI sidebar.

## Typesense-PG Sync

Entities are synced to the Typesense index asynchronously through the pg-boss job system (`packages/jobs/`). The flow is:

1. Entity create/update/delete triggers a job enqueued via Drizzle's `afterCommit` hook
2. The `search-index` worker (pg-boss job) picks up the job and performs an upsert into the Typesense index
3. On delete, the entity ID is removed from the Typesense index

**Sync latency** is typically under 500ms for individual operations under normal load. Under high throughput (>100 writes/min), batching kicks in and latency can increase to ~2 seconds. There is no real-time sync — a stale read of up to 5 seconds is acceptable by design.

Large-scale bulk operations (workspace imports, template seeding) use a separate `search-bulk-index` worker that batch-inserts entities into Typesense in pages of 100.

## pgvector Embedding Pipeline

Embeddings are generated by the intelligence service and stored in a pgvector `vector(768)` column on the `entities` table (dimensionality configurable, 768 is the default for the primary model).

**Flow:**
1. When an entity is created or updated, the backend sends the entity's text content (name, description, profile_data extraction) to the intelligence service's `/embed` endpoint
2. The intelligence service runs the content through a configurable embedding model (default: a multilingual model for broad language support) and returns the vector
3. The vector is stored in the pgvector column via Drizzle's vector type support

**Recalculation** happens automatically on any entity profile_data change that includes modifiable text fields (name, content, description, tags). The intelligence service does not re-embed entities on unrelated profile changes (status, metadata, etc.) to save compute costs.

Admins can trigger a full re-embedding pass via the `search-reembed` pg-boss job, which iterates all entities and regenerates vectors using the current model.

## Search Ranking Algorithm

Search results are composited in two phases:

**Phase 1 — Parallel Retrieval:** The frontend (or proxy service) queries both Typesense and pgvector in parallel. Typesense returns results ranked by BM25-style text relevance with optional field-weight boost (title weighted higher than content). The pgvector query uses cosine similarity to find conceptually related entities.

**Phase 2 — Reranking & Merge:** Results are deduplicated by `entity.id`. A composite score is computed:
```
composite_score = (0.65 * normalized_typesense_score) + (0.35 * normalized_pgvector_score)
```
The weights are tunable per-tenant and default to 65/35 in favor of full-text relevance. Semantic similarity acts as a recency and conceptual boost for borderline matches.

When pgvector is disabled (shared pod mode), composite_score falls back to Typesense score alone.

## Query Parsing & Intent Detection

The frontend search heuristic classifies user intent as either **keyword** or **semantic** based on:

- Query length: queries >20 characters are biased toward semantic routing
- Query patterns: questions ("how to...", "what is...") route to semantic
- Known entity type prefixes (`type:task`, `type:person`) suppress semantic search
- Tag filters (`tag:important`) suppress semantic search

Additional query expansion strategies:
- **Stemming**: Applied automatically by Typesense for morphological variants
- **Synonym expansion**: Built-in stemming handles common variants; custom synonym injection available
- **Typo tolerance**: Up to 1-2 character typos depending on query length (see Typesense section above)

## Filter Capabilities

The search layer supports these filters (all applied in the Typesense API call and/or pgvector WHERE clauses):

| Filter | Source | Typesense | pgvector |
|--------|--------|-----------|----------|
| Entity type | `type:` prefix or facet selection | `entity_type` term filter | Not applicable |
| Workspace | Auto-injected from user's access | `workspace_id` term filter | Not applicable |
| Date range | Date picker UI | `created_at` / `updated_at` range | Not applicable |
| Tags | Tag chips or `tag:` prefix | `tags` array contains any | Not applicable |
| Status | Status filter UI | Enum field term filter | Not applicable |
| Property values | Custom profile field filters | JSONB field extraction | Not applicable |

All filters use the user's workspace access context (via `workspace_members`) — users can only filter/search within workspaces they belong to.

## Performance Benchmarks

Typical response times (measured from query to composed results):

| Scale | Typesense latency | pgvector latency | Composite total |
|-------|-------------------|-------------------|-----------------|
| 10K entities | ~15ms | ~20ms | ~45ms |
| 100K entities | ~30ms | ~80ms | ~110ms |
| 1M entities | ~60ms | ~250ms | ~310ms |

pgvector latency scales with index size but benefits significantly from HNSW indexing (configured with `m=16`, `ef_construction=64`, `ef_search=128`). The HNSW index makes pgvector queries practical even at the million-entity mark.

Bulk index operations are non-blocking — the Typesense index can be rebuilt in the background while searches continue against the existing index.

## Shared Pod Restrictions

On shared pods, search functionality is intentionally simplified:

- **VECTOR_SEARCH_ENABLED=false** — pgvector is completely disabled. The `search-reembed` job is a no-op. The pgvector query in the composition phase is skipped entirely.
- **Typesense-only mode** — All search goes through Typesense with full-text and faceted filtering only. No semantic discovery of related entities.
- **Reduced sync** — The Typesense-PG sync still runs but is lighter-weight since no embedding generation is involved.
- **Why**: Shared pods have constrained compute. Embedding generation via the intelligence service is an expensive operation per entity.

Users who upgrade to a dedicated pod automatically get pgvector enabled — no migration needed. The existing Typesense index continues to serve as the primary search engine; pgvector is simply layered on top.

## Search Edge Cases

Special case handling is built into both search engines:

- **Emoji**: Fully supported in Typesense text fields (stored as UTF-8, tokenized normally). Emoji in entity names or content are searchable.
- **International characters**: Both Typesense and the embedding model support UTF-8 / multilingual content. The embedding model is explicitly multilingual, so semantic search works across English, French, Spanish, Japanese, etc.
- **Special characters**: Normalized by Typesense's text processor. Accented characters (é, ö, ñ) are handled via ICU normalization — searching `resumé` will match `resume` and vice versa.
- **Empty/whitespace-only queries**: Frontend suppresses the search call if the trimmed query is empty. Results page shows default view (recent entities or workspace defaults).
- **Very long queries (>500 chars)**: Truncated to 500 characters to protect backend query performance. A warning indicator is not shown — the user may not notice.
