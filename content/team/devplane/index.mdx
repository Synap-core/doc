DevPlane is the developer intelligence layer built on a Synap data pod. Everything you build, every package you publish, every session you run, every decision you make — stored as connected entities. AI that knows your full stack. One-button automations for the flows you repeat. The mental overhead disappears because the system handles it.

Live at **[devplane.synap.live](https://devplane.synap.live)**.

See [North Star](./north-star) for the full vision, pillars, and implementation roadmap.

## What it is

DevPlane is a Next.js web app (`synap-app/apps/devplane/`, port 3032) that sits on top of a Synap data pod. There is no separate database or backend — every piece of data is a typed Synap entity. The pod stores it; DevPlane renders it.

It is the first example of a **vertical Synap app**: derived from `apps/base/`, with its own workspace definition, entity profiles, and purpose-built UI — while sharing all core infrastructure (auth, tRPC, stores, cell runtime, command palette).

## What it ships today

| Section | What you can do |
|---------|-----------------|
| **Apps** | Track repositories, tech stacks, ports, deploy URLs. Vault refs for NPM and GitHub tokens. |
| **Features** | Kanban board (Backlog → In Progress → In Review → Done). Per-feature AI channel with context injection — the agent knows the linked app, packages, recent deployments, and open incidents. |
| **Services** | Table of backend services with type, host:port, health check URL, and linked app. |
| **Packages** | Grid with health grades (A–F), stale status, dependency graph edges, publish history. Impact analysis before publishing. |
| **Servers** | SSH-accessible server nodes with Kamal role (web/worker/db/cache), provider metadata, vault-backed SSH keys. Full xterm.js terminal. |
| **Deployments** | Timeline with status, version, commit SHA, linked server. Filter by status. |
| **Recipes** | SSH command sequences with live streaming output per step. Templates: Kamal, Shell, Custom. |
| **Incidents** | Severity triage, status tracking, root cause, linked to server and recipe run that caused it. |
| **Crons** | Scheduled job registry: cron expression, status, last/next run, linked app. |
| **Knowledge** | Two-tab page: Decision Records (context + rationale + alternatives) and Snippets (reusable AI prompts, optionally executable as skills). |

## How it relates to Synap

DevPlane uses Synap as pure infrastructure:

- **Data pod** — all entities are stored on `antoine-perso.synap.live` under a workspace named "DevPlane"
- **tRPC** — all reads/writes go through `trpc.entities.*` and `trpc.workspaces.*`, proxied through the Next.js `/trpc` route
- **Auth** — same CP sign-in → pod handshake flow as all other Synap apps
- **Workspace definition** — profiles and views are defined entirely in `lib/devplaneWorkspaceDefinition.ts`. The pod/backend never knows about DevPlane-specific profiles. This is intentional.

## Workspace bootstrap

DevPlane self-provisions on first login:

1. `useDevplaneSetup` creates the "DevPlane" workspace and all 13 entity profiles via `trpc.workspaces.createFromDefinition`
2. `useWorkspaceReadiness` (runs on every login) verifies all 13 profiles exist. If any are missing (e.g. after a definition update), it calls `createFromDefinition` with the existing workspace ID — idempotent, adds only what's missing.

The "Verifying workspace profiles" setup screen users see on first launch is this readiness gate in action.
