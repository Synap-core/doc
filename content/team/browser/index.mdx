## Why this application exists

1. **Workspace shell** — Synap apps (Home, Library, Chat, Canvas, etc.) run inside a **trusted** embedded view with session access to the user’s pod — same graph, AI, and governance as the backend, without asking users to live in a generic browser tab.  
2. **Built-in web** — Tabs + address bar for **arbitrary sites** in **sandboxed** webviews, separate from the trusted Synap UI. Enables research → **Save to Pod** (URL, metadata, optional screenshot/readability/clip) as first-class entities and events.  
3. **Sovereign capture story** — “Web → graph” with **provenance** (who, when, which URL, which action). AI-assisted extraction stays **proposal-based** where writes are involved.  
4. **OS integration** — Deep links from **[synap.live](https://synap.live)** (auth, billing, invites, verify-email), downloads, notifications, optional in-app **[documentation](/docs)**.  
5. **Shared UI stack** — Tamagui, hooks, stores, and many packages aligned with **`synap-app`**; see **[Synap App](/team/synap-app)** for package boundaries.

**Non-goals for the MVP-class scope:** full browser replacement (extension store, full sync), unconstrained background scraping, or a separate “signup app” — account flows stay on the **marketing / account web** (see **[Landing audit](/team/home/landing-audit)**).

## User journeys

### Connect or create a pod (signed-in path)

1. Launch app → onboarding (**`OnboardingFlow`**, `onboarding.completed` gate).  
2. **Connect my Data Pod** → Control Plane sign-in and/or pod URL (self-host).  
3. Persist **`podBaseUrl`**, tokens (secure storage), **`workspaceId`** → land in **AppShell** with real pod data.

**Success target:** time-to-first productive workspace on the order of minutes, not hours.

### Explore first (anonymous / local-first path)

Single **`OnboardingFlow`**: welcome → **Connect** vs **Explore first**. Explore path skips pod setup, enters **anonymous mode**, may show **WorkspaceCreationPopup** and **anonymous onboarding chat** (CP anonymous token → IS → **WorkspaceProposal** JSON, stored locally e.g. IndexedDB, **no pod writes** until the user connects a pod). Budget UX drives **Connect a pod** when limits are low.

**Principle:** one flow, multiple outcomes; IS can author workspace **proposals** before a pod exists.

### Day-to-day: Synap + web

- Sidebar **apps** (manifest-driven) for product surfaces.  
- **Browser** app: navigate, then **Save to Pod** / clip / (later) AI extract → proposals.  
- **Workspace / pod gates** — clear blocking UI when the pod is offline or workspace not selected (`PodGate`, `WorkspaceGate`).

### Updates

**`electron-updater`** checks GitHub Releases when not in dev; **`electron-builder.yml`** `publish` must match the repo that publishes artifacts. See **Operations** below.

## Architecture (trust boundaries)

```text
Main process (Node)     — windows, auto-update, secure storage, IPC router, capture orchestration
     ↑ typed IPC
Renderer (React/Tamagui) — Synap shell + browser chrome + modals
     ↑ postMessage (strict)
Webviews (untrusted)     — real internet; sandbox, no Node, minimal preload (selection/metadata only)
```

**Rules:** `contextIsolation`, no Node in webviews, **allowlisted IPC** with validated payloads; prefer **pod calls from main** so tokens never touch untrusted web content. Multi-pod: **no cache bleed** across pods.

Embedded **Synap-app** workspace view may use a **trusted-origin** configuration (e.g. session/cookies) — document any `sandbox: false` trade-off in release/security notes.

## Security (summary)

Align with Electron hardening: CSP where feasible, block dangerous schemes (`file://`, `javascript:`), gate permissions, redact logs, signed updates. Full checklist-style content lives in team docs here; **release gates** before broad distribution: webview sandbox verified, IPC allowlist + schema validation, HTTPS + manifest checks for pod URL, secrets only in OS secure store.

Future **wallet** posture (if ever): keys never in webviews; dedicated signing UI; capability-style grants — see historical **Document 3/3** security notes merged into this section conceptually.

## Operations (release, signing, CI)

**Canonical repo files:**

- **`browser/electron-builder.yml`** — `appId`, targets, `publish` (`owner` / `repo` for updater).  
- **`browser/package.json`** — `dist`, `dist:mac`, `dist:win`, `dist:linux`, `dist:all`.  
- **`.github/workflows/browser-release.yml`** (repo root) — tag `v*` → build Linux/macOS/Windows → `electron-builder --publish always`.  
- **`browser/INSTALL.md`** — download URLs for end users.  
- **`browser/build/`** — `icon.icns`, `icon.ico`, `icon.png` (required for packaging).

**Checklist highlights:** icons present; **code signing** (Apple Developer ID + notarization; Windows Authenticode); **GITHUB_TOKEN** / PAT for CI publishing; confirm updater **owner/repo** matches where Releases live.
