# Native OS — UX integration plan

How the `@synap-core/native-os` capabilities become features the user
actually sees and uses in Relay. Companion to `NATIVE-OS-GUIDE.md`
(implementation) and `NEXT-GEN-SHIP-PLAN.md` (sprint ordering).

**Date:** 2026-04-11.
**Scope:** Relay first. Browser mobile + future Expo apps reuse the same
plan.

---

## Design principles

Before the capability list, the rules every integration follows:

1. **OS events reuse the deep-link pipeline.** The capture modal, entity
   detail, space switcher — all of them already handle `relay://` URLs.
   Every new OS entry point produces a URL and hands it to the router. No
   new "share mode" or "intent mode" screens.

2. **Zero per-platform branching at the UI layer.** If Relay needs to
   check `Platform.OS` to render something, the abstraction failed. The
   bridge reports availability; the UI hides/shows based on that, not on
   the platform name.

3. **Graceful degradation is invisible.** A capability that isn't
   available doesn't produce an error state — it just doesn't exist in
   the UI. The user never sees "sorry, not supported." They see the
   affordances that work and nothing else.

4. **Every surface earns its pixels.** A home-screen widget that shows
   stale data is worse than no widget. A Dynamic Island pill that doesn't
   update is worse than no pill. Each surface must have a continuous data
   pipe — if we can't keep it live, we don't ship it.

5. **Capture is the north star.** Every OS integration funnels back to
   one of three actions: capture something new, find something existing,
   or switch context. Nothing else.

## 1. Share Extension

### What the user sees

- Opens Safari, reads an article, taps Share → **Relay** appears in the
  share sheet alongside Messages, Mail, Notes.
- Taps Relay → app cold-starts (if closed) → capture modal opens with
  the article URL prefilled → AI structuring kicks in → user hits Send
  → entity created, modal closes, back to Safari.
- Same flow from: Mail (forward an email), Photos (share an image),
  Twitter (save a tweet), Chrome / Gmail / Samsung Notes on Android.

### Screens involved

- **No new screens.** Reuses existing `CaptureModal` with `initialText`.
- Modal needs a subtle "from [source app]" attribution badge above the
  input so the user remembers the context. 1-line addition to
  `CaptureModal.tsx` — read `source` query param from the deep link.

### Data flow

```
Safari → Share → iOS Share Extension (expo-share-intent)
                        │
                        ▼
            sharedStorage['share-queue'] = {...}
                        │
                        ▼
            Relay foregrounds → NativeOSBridge useShareIntent fires
                        │
                        ▼
            Linking.openURL('relay://capture?text=...&source=safari')
                        │
                        ▼
            expo-router → Home tab → CaptureGestureButton opens modal
```

### What we need to build

- [x] Bridge wiring (`NativeOSBridge.tsx` already subscribes to share)
- [ ] Install `expo-share-intent` in `relay-app` + rebuild dev client
- [ ] Attribution badge in `CaptureModal` reading the `source` query param
- [ ] Handle image/file share payloads — wire into `@synap-core/capture-pipeline`'s
      attachment path
- [ ] App Store / Play Store asset: Share Extension icon (matches app icon)

### Power upgrade (W9)

Swap `expo-share-intent` → `expo-share-extension` for a custom SwiftUI
share sheet with:
- In-sheet space picker (choose which space to capture into)
- AI tag preview (runs structuring in the extension process)
- Instant save without app foreground

Done entirely inside `native-os-share` — consumer code untouched.

## 2. Spotlight (iOS) + AppSearch (Android)

### What the user sees

- Pulls down on the iOS home screen → search "Alice" → **Relay** result
  appears under Mail and Messages → taps → Relay opens to Alice's
  person detail.
- Same for notes, tasks, projects. Works from anywhere in iOS (Today
  View, Home Screen search, Shortcuts, Lookup).
- Android: Pixel Launcher search shows the same results; AppSearch
  bridge ships in a later sprint.

### Screens involved

- **No new screens on the Relay side.** Reuses existing entity detail
  screens (`app/person/[id]`, `app/note/[id]`, `app/campaign/[id]`, ...).
- One new route: `app/entity/[id].tsx` — a generic resolver that looks
  up the entity kind by id (via shared storage map) and redirects to
  the kind-specific screen. Needed because Spotlight only hands us the
  id on tap, not the kind.

### Data flow

```
useFeed() in root provider
        │
        ▼
useSpotlightSync(entities)  ← reindexes on every feed mutation
        │
        ▼
Core Spotlight index (iOS system)
        │
        ▼
User taps result → iOS opens relay://entity/:id
        │
        ▼
app/entity/[id].tsx resolves kind → redirects to kind-specific screen
```

### What we need to build

- [x] Bridge wiring (`useSpotlightSync` + `useSpotlightTap`)
- [ ] Install `react-native-spotlight-search` + rebuild dev client
- [ ] Root provider call to `useSpotlightSync(indexableEntities)` —
      derives from the existing feed hook
- [ ] `app/entity/[id].tsx` generic resolver route
- [ ] Entity-id-to-kind map written to shared storage on every feed
      mutation (same write path as widgets)
- [ ] Decide: do we index vault entities? **Default no.** Add a
      `searchable: boolean` flag to the entity → exclude vault items
- [ ] Exclusion list in user settings: "Don't show [X] in system search"

### Android follow-up

Custom Kotlin AppSearch module. Tracked in `NEXT-GEN-ANDROID-PLAN.md`.
Not blocking the iOS ship.

## 3. Voice Live Activity (Android via Notifee)

### What the user sees

- Starts a voice recording via the gesture button (drag to chip, hold).
- A **persistent notification** appears in the Android status bar:
  "Recording… 00:12" with a cancel button.
- User swipes away from Relay to check a message in WhatsApp → comes
  back → recording is still going, transcript is still accumulating.
- Taps the notification → Relay foregrounds, gesture button is in
  detached mode showing the Send affordance.

### Screens involved

- **No new screens.** The gesture button already has the detached-mode
  UI. The live activity is a mirror of that state, shown outside the app.

### Data flow

```yaml

CaptureGestureButton starts voice → voice.isListening = true
        │
        ▼
useLiveActivity({ kind: 'voice-recording', title: 'Recording…', elapsedSec })
        │
        ▼
Notifee displayNotification({ ongoing: true, asForegroundService: true })
        │
        ▼
User backgrounds app → notification stays, progress keeps updating
        │
        ▼
voice.stop() → useLiveActivity(null) → Notifee cancelNotification

```

### What we need to build

- [x] Bridge wiring (`useLiveActivity` hook exists)
- [ ] Install `@notifee/react-native` + rebuild dev client
- [ ] Notifee channel created at app start (`synap-live-updates`)
- [ ] Elapsed time state in `CaptureGestureButton` (setInterval during
      recording — 5 lines)
- [ ] Hook the live activity state into the button's phase state
      machine
- [ ] Tap handler: `Notifee.onForegroundEvent()` → `Linking.openURL('relay://capture')`

### iOS counterpart (W6)

Same JS code. Dynamic Island pill via the Swift `RelayActivity` target.
Ship plan Sprint 2.

## 4. Vision — Still image OCR

### What the user sees

- Opens capture modal → taps a new **"Scan"** button in the action bar
  (between mic and camera) → camera opens → points at a whiteboard /
  book page / business card → tap shutter → camera closes → text
  appears in the capture input.
- For business cards specifically: text is structured into a person
  proposal automatically (name, email, phone, company).

### Screens involved

- **No new screens.** `expo-image-picker`'s camera is the interface.
- Capture modal action bar gets a new button. Icon: `Scan` from
  `lucide-react-native`.
- Power upgrade (W8): dedicated `app/scanner.tsx` route with a live
  VisionCamera overlay showing tap-to-select text regions.

### Data flow

```typescript

User taps Scan button in CaptureModal
        │
        ▼
vision.scanText() → expo-image-picker camera
        │
        ▼
User snaps photo → ML Kit / Vision recognize
        │
        ▼
Returns { text } → setText((prev) => prev + '\n' + text)
        │
        ▼
User edits if needed → Send → AI structuring

```

### What we need to build

- [x] Bridge wiring (`vision.scanText`)
- [ ] Install `@react-native-ml-kit/text-recognition` + rebuild
- [ ] `ScanButton` component in `CaptureModal` — hide if
      `!vision.availability.available`
- [ ] Business card detection heuristic: if OCR result contains both an
      email pattern and a phone pattern, route through
      `@synap-core/capture-pipeline` with `intent: 'person'`
- [ ] Capture pipeline: accept `source: 'vision'` in the structuring
      request so the AI knows the text came from OCR (may contain
      errors, should be more forgiving)

## 5. Widgets — Quick Capture (first widget to ship)

### What the user sees

- Long-press home screen → Add Widget → Relay → **Quick Capture** →
  drag onto home screen.
- Widget shows a big "+" icon with "Capture" label and the active space
  color as background.
- One tap → Relay opens directly to the capture modal (empty).
- Power mode (future): long-press the widget → a voice capture variant
  appears, one tap starts a voice recording and a Live Activity.

### Screens involved

- **New native targets:** `RelayWidgets.swift` (iOS) and `QuickCaptureWidget.kt`
  (Android). These are Swift/Kotlin, not JS.
- **No changes to Relay JS.** The widget button target is a deep link
  `relay://capture` which the existing router handles.

### Data flow

```
Widget tap → intent = relay://capture
        │
        ▼
iOS / Android open Relay with the URL
        │
        ▼
expo-linking resolves → CaptureModal opens
```

For state display (Today's Captures widget — ships after Quick Capture):

```yaml

Relay feed mutates (any capture, sync, delete)
        │
        ▼
widgets.setState('todays-captures', { count, items: [...] })
        │
        ▼
sharedStorage['widget:todays-captures'] = JSON
        │
        ▼
Widget timeline reload → widget reads shared storage → renders

```

### What we need to build

- [ ] iOS: `RelayWidgets` extension target via `@bacons/apple-targets`.
      SwiftUI widget view reads App Group UserDefaults for state.
- [ ] Android: Glance widget under `android/app/src/main/java/.../widgets/`.
      Custom Expo config plugin to inject the widget provider.
- [ ] State writes from JS (already possible via `widgets.setState`)
- [ ] Widget reload trigger after each state write — needs the native
      `WidgetCenter.reloadAllTimelines()` call wired in
- [ ] Design: widget preview images for the gallery picker

## 7. App Intents — Create Note

### What the user sees

- User holds the Home button / says "Hey Siri, new Relay note about
  the design review meeting" → Siri confirms "Note added to Relay" →
  note appears in the feed next time Relay foregrounds (or immediately
  if it's already open).
- Same on Android: "Hey Google, new Relay note…"
- Also appears in Spotlight Actions, Shortcuts suggestions, iOS 18
  AppShortcutsProvider.

### Screens involved

- **No Relay screen changes.** Intent handlers run in native code, write
  to the backend via `@synap-core/capture-pipeline` called from the
  bridge's JS handler, and the feed refreshes on next foreground.
- **New:** a lightweight "intent receipt" toast shown when Relay opens
  with a recently-fired intent. "Note created from Siri" — 3-second
  fade. Builds user trust in the zero-UI flow.

### Data flow

```yaml

"Hey Siri, new Relay note about..." → iOS App Intent target
        │
        ▼
RelayIntents.swift CreateNoteIntent.perform()
        │
        ▼
sharedStorage['intent-queue'] = [{kind: 'create-note', params: {text: '...'}}]
        │
        ▼
Writes through to Relay's local entity store (via a shared native
helper that reuses the capture-pipeline types)
        │
        ▼
Intent completes, Siri confirms "Note added"
        │
        ▼
Next time Relay foregrounds: useIntentHandler fires, shows receipt toast

```

### What we need to build

- [ ] iOS: `RelayIntents.swift` with `CreateNoteIntent` conforming to
      `AppIntent` + `AppEntity` for note references
- [ ] Android: `shortcuts.xml` + `CreateNoteActivity.kt`
- [ ] Shared native helper for intent→entity persistence (so we don't
      duplicate the capture pipeline in Swift/Kotlin)
- [ ] Toast component for intent receipts
- [ ] Intent donation on every in-app capture (already implemented at
      facade level, just need to call it from the capture pipeline's
      success handler)

## Metrics to track per capability

Not currently wired but should be added to the existing analytics layer:

- Share Extension: shares received per day, conversion rate (queued →
  captured → saved), drop-off at each step.
- Spotlight: queries that hit Relay results, tap-through rate.
- Intents: intents fired per day, intents donated per day, Siri
  suggestion surface rate.
- Widgets: impressions (from timeline reload events), taps.
- Live Activity: activation rate (recording started → activity shown),
  cancellation source (tap inside vs outside app).
- Vision: scan attempts per day, average OCR text length, correction
  rate (user edited the extracted text before sending).

## Open questions

1. **Vault exclusion UX.** Default is "never index vault items." Does
   the user get a toggle per entity, per profile, or just a global
   settings switch?

2. **Entity id → kind resolver.** Three options:
   (a) Encode the kind into the id as a prefix (`note:abc123`).
       Minimal code but requires a DB migration on every existing id.
   (b) Maintain an id→kind map in shared storage, updated on every
       feed mutation. No migration, adds ~50KB of RAM per 1000
       entities.
   (c) Route all taps through a generic `/entity/:id` resolver that
       looks up the entity in the existing store. Cleanest code, adds
       one redirect hop on every tap.
   **Recommendation:** (c). The redirect is invisible (< 50ms) and we
   avoid DB migration.

3. **Intent receipts — always show or only first-time?** Always risks
   toast fatigue. First-time may confuse users who don't understand
   what happened. **Recommendation:** show for the first 3 intents
   across the app's lifetime, then silent unless the user re-enables.

4. **Widget refresh cadence.** Widgets can't pull — they reload on a
   timeline budget that iOS decides. Today's Captures widget wants
   sub-second refresh after a capture. This requires us to call
   `widgets.reloadKey('todays-captures')` on every feed mutation, which
   needs the native `WidgetCenter.reloadAllTimelines()` bridge wired up.
   **Blocker:** Sprint 2 work.

5. **Siri voice capture vs existing voice flow.** When the user says
   "Hey Siri, new Relay note," should Siri capture the audio and hand
   us the text, or should we open Relay's own voice capture flow? The
   Siri-native path is lower latency but gives us no AI structuring
   preview. **Recommendation:** Siri-native path for the MVP,
   one-tap-to-refine in the intent receipt toast.
