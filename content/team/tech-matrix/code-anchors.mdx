## Browser (Electron App)

**Repo:** `browser/`
**Port:** 3000 (dev)
**Surface:** Primary desktop experience

### Architecture

```
Main process (Node) — windows, secure storage, updater, IPC router
Renderer (React/Tamagui) — shell UI and product apps
Webviews (untrusted) — external websites in sandboxed contexts
```

## Synap App (Web Monorepo)

**Repo:** `synap-app/`
**Port:** 3000 (dev)
**Structure:** pnpm monorepo with workspace packages

### Architecture

**Three-layer data pattern:**
```
React Component (synap-app/packages/)
  → @synap/hooks (useEntity, useView, useChannel...)
    → @synap/stores (Zustand workspace/UI state)
      → @synap/client (tRPC + TanStack Query)
```

**Cell system layers:**
- `@synap/capabilities` — manifest of cell types, widget types, and contracts (source of truth)
- `@synap/cell-runtime` — registry singleton + CellRenderer dispatcher
- Hosts: `BentoCellHost`, `PanelCellHost`, `PageCellHost`

**Key directories:**
- `packages/core/` — ui-system, capabilities, cell-runtime, view-renderer, entity-card, proposal-types, notifications
- `packages/features/` — ai-chat, connectors, sidebar, onboarding
- `packages/views/` — table, kanban, bento, calendar, timeline, graph, flow, etc.
- `packages/apps/` — app compositions
- `packages/synap-client/` — tRPC proxy, TanStack Query setup
- `packages/synap-stores/` — Zustand workspace and UI state
- `packages/synap-hooks/` — React hooks layer (useEntity, useView, etc.)

### Key Files

### [synap-app] packages/core/capabilities/src/manifest.ts — Cell manifest (source of truth)

## Synap Backend (API Pod)

**Repo:** `synap-backend/`
**Port:** 4000 (dev)
**Stack:** Hono, tRPC 11, Drizzle ORM, PostgreSQL/TimescaleDB

### Architecture

```
tRPC router (Hub Protocol + tRPC) → services → repositories → Drizzle → PostgreSQL
```

**Three communication channels:**
1. **tRPC** — Direct client-to-backend (300ms target, used by all frontend apps)
2. **Hub Protocol REST** — Agent-to-backend REST endpoints (POST /api/hub/*, used by Intelligence Service and OpenClaw)
3. **Socket.IO** — Realtime subscriptions (used by all apps for live updates)

tRPC and Hub Protocol REST share the same tRPC routers but are routed differently: tRPC via `/trpc/*`, Hub Protocol via `/api/hub/*` (directly mounted, not proxying tRPC).

### Key Files

### [synap-backend] packages/api/src/routers/hub-protocol-rest.ts — Hub Protocol REST router
21 sub-routers mounted at `/api/hub/*`. Handles ALL IS ↔ Backend communication. NOT `/trpc/hubProtocol.*`.

### [synap-backend] packages/api/src/routers/ — tRPC routers directory
All tRPC router definitions. Key routers:
- `profiles.ts` — Profile CRUD and effective property resolution
- `entities.ts` — Entity CRUD with `checkPermissionOrPropose()` gating
- `views.ts` — View filter compilation + default columns (threads `workspaceId`)
- `channels.ts` — Channel management and message operations
- `proposals.ts` — Proposal lifecycle
- `ai.ts` — AI integration endpoints (proactive posts, agent actions)

### [synap-backend] packages/api/src/utils/permission-check.ts — Permission check
`checkPermissionOrPropose()` — RBAC + DEFAULT_AUTO_APPROVE whitelist. Used for AI mutations and external connector syncs.

### [synap-backend] packages/database/src/schema/ — Drizzle schema
All database table definitions. Key schemas:
- `entities` — Core entity table (JSONB properties, profile_id, workspace_id, documentId)
- `profiles` — Entity profiles (14 system profiles + custom profiles)
- `views` — View definitions (type, filters, config JSONB)
- `view_filters` — View filter specifications
- `channels` — Channel definitions with types
- `channel_messages` — Messages within channels
- `proposals` — Proposal lifecycle and governance
- `property_defs` — Property definition schema with three-level workspace scoping
- `workspace_members` — User↔workspace membership
- `entities_relations` — Entity relationship edges
- `documents` — Document content and associations
- `ai_memory` — Agent session memory (session-scoped)

### [synap-backend] packages/database/src/services/profile-resolution-service.ts — Profile resolution service
`getEffectiveProperties(profileId, workspaceId)` — single source of truth for overlay property resolution. 60s TTL cache with `invalidateEntityScopeCache(slug)`.

### [synap-backend] packages/database/src/utils/ensure-system-profiles.ts — System profile seeding
Creates 14 default entity types: note, task, project, event, person, contact, company, deal, bookmark, website, article, capture, file, anchor.

### [synap-backend] packages/jobs/ — pg-boss job workers
27 workers + 3 proactive cron jobs. Key workers:
- `ai-workers` — AI classification, proactive posts
- `materializer` — Entity view materialization (create/update)
- `proactive-cron` — Morning briefing, weekly digest, health check

### [synap-backend] packages/realtime/ — Socket.IO + Yjs realtime
WebSocket bridge, presence, collaborative editing sync.

### [synap-backend] skills/synap/ — OpenClaw skill
Unified OpenClaw skill for Synap integration. Install via `openclaw skill install synap`.

### Other key files

### [synap-backend] deploy/docker-compose.yml — Docker Compose (production)
Full stack: backend, realtime, postgres, redis, minio, typesense, kratos, hydra, caddy, openclaw (profile: openclaw), rsshub/browserless (profile: rsshub), monitoring (profile: monitoring), pod-agent, updater.

### [synap-backend] deploy/setup-openclaw.sh — OpenClaw setup script

### [synap-backend] deploy/docker-compose.standalone.yml — Standalone compose
Optional `--profile openclaw` profile for self-hosted OpenClaw.

### [synap-backend] skills/synap-os/SKILL.md — OpenClaw OS skill

## Intelligence Service

**Repo:** `synap-intelligence-service/`
**Port:** 3001 (dev)
**Stack:** OrchestratorAgent + PersonaAgent, Hono, OpenRouter

### Architecture

Two-layer agent model:
- **OrchestratorAgent** — Co-founder persona with full tool access
- **PersonaAgent** — Domain expert personas with branch-only access

Session-scoped memory: always on, 4h personal sessions / 30min other sessions, compaction engine.

### Key Files

### [intelligence-service] apps/intelligence-hub/src/agents/orchestrator-agent.ts — Orchestrator agent
Full-tool-access agent. Core of the AI conversation flow.

### [intelligence-service] apps/intelligence-hub/src/agents/persona-agent.ts — Persona agent

### [intelligence-service] apps/intelligence-hub/src/agents/ — Agent directory

### [intelligence-service] apps/intelligence-hub/src/skills/ — Agent skills
Skill definitions for agents: entity management, view creation, document operations, workflow automation.

### [intelligence-service] apps/intelligence-hub/src/ — Hub directory
Hub protocol implementation, event sourcing, proposal integration.

### [intelligence-service] apps/intelligence-hub/src/agents/ — Agents directory

### Proactive intelligence
