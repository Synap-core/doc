## Current State

### Template Architecture

Templates are `WorkspaceProposal` JSON objects containing everything needed to create a workspace:
- `profiles[]` — Entity type definitions
- `views[]` — View configurations
- `bentoLayout[]` — Dashboard widget arrangement
- `suggestedEntities[]` — Seed data
- `suggestedRelations[]` — Links between seed entities
- `displayTemplates[]` — How entities render in cards/detail views
- `layoutConfig` — Sidebar items, pinned apps, default view

### Template Sources

| Source | Location | Status |
|--------|----------|--------|
| **Local fallback** | `synap-app/packages/features/onboarding/src/templates/` | 3 templates (Project Mgmt, Personal, Blank) |
| **Control Plane registry** | `synap-control-plane-api/templates/` | Flagship templates (CRM, Second Brain, Content OS, Agent OS) |
| **Dev Dashboard** | `synap-dev-dashboard/src/template/` | Internal DevOps workspace |
| **AI-generated** | Via `propose_workspace` tool | AI creates full proposals on-the-fly |

### Creation Pipeline

```
Template JSON → createFromDefinition API → Backend:
  1. Create workspace
  2. Create profiles (entity types)
  3. Create views with configs
  4. Create bento dashboard
  5. Create seed entities
  6. Create relations
  7. Create display templates
  8. Set layout config
  → Return workspaceId
```

### Quality Assessment

**GOOD:**
- Template format is comprehensive and well-documented (53KB guide)
- `createFromDefinition` handles the full pipeline in one API call
- DevOps template is a good reference implementation
- AI can generate templates via `propose_workspace`

**RESOLVED (2026-03-25):**
-  Templates are no longer all-or-nothing — profile packs add to existing workspaces
-  Local fallback vs CP registry — ProfilePackBrowserCell has proper online/offline logic

**STILL OPEN:**
- Type duplication — `WorkspaceProposal` defined in 3+ places (no `@synap/template-types` package yet)
- No versioning — templates can't be updated after installation
- No template preview beyond pack list cards
- `composesFrom` merge logic not yet implemented in browser

## Limitations

### 1. Monolithic Templates
A template creates an entire workspace. You can't:
- Add a profile pack to an existing workspace
- Import just the views from a template
- Merge two templates together
- Update a workspace when its template gets updated

### 2. No Composability
Templates are flat JSON blobs. There's no:
- Inheritance (template B extends template A)
- Mixins (add "CRM profiles" to any template)
- Conditional sections (include X only if user selects option Y)

### 3. No Post-Install Updates
Once a template is installed:
- No way to push updates to existing workspaces
- No migration path when profile definitions change
- User has to manually recreate or modify

### 4. Type Fragmentation
`WorkspaceProposal` is defined in:
- `synap-dev-dashboard/src/types.ts`
- `synap-app/packages/features/onboarding/src/templates/types.ts`
- Possibly in control plane code
- Each copy may drift

### 5. Template Discovery
- No browseable template gallery in the browser
- No template preview (screenshots, sample data)
- No search/filter by category
- CP marketplace exists but is focused on "packages" not "templates"

## Proposed Architecture: Composable Templates

### Profile Packs (see 02-entity-profiles.md)
The smallest unit of template composition. Contains profiles + optional views + relations.

### Template = Composition of Packs + Config

```typescript
interface WorkspaceTemplate {
  id: string;
  name: string;
  description: string;
  icon: string;
  category: 'productivity' | 'knowledge' | 'development' | 'business' | 'personal' | 'custom';

  // Composition
  profilePacks: string[];        // IDs of profile packs to include
  additionalProfiles?: ProfileDefinition[];  // Template-specific profiles

  // Layout
  bentoLayout: BentoWidget[];
  bentoViewBlocks?: BentoViewBlock[];
  views: ViewDefinition[];
  layoutConfig: LayoutConfig;

  // Seed data (optional)
  seedEntities?: EntitySeed[];
  seedRelations?: RelationSeed[];

  // AI behavior
  skills?: SkillDefinition[];    // AI skills activated with this template
  automations?: AutomationDef[]; // Event triggers
}
```

### Template Installation Flow

```
User selects template
  → Preview (show profiles, views, sample dashboard)
  → Customize (toggle profile packs, rename workspace)
  → Install:
    1. Resolve profile packs (fetch from CP or local)
    2. Merge profiles (system + packs + template-specific)
    3. Create workspace via createFromDefinition
    4. Register skills
    5. Set up automations
    6. Optionally: AI walks user through initial data entry
```

### Template Update Flow

```
Template v2 published on CP
  → User notified "Update available for Knowledge Base template"
  → Preview changes (new profiles, new views, changed properties)
  → Apply:
    1. Add new profiles (non-destructive)
    2. Add new views (non-destructive)
    3. Update bento layout (if user hasn't customized)
    4. Migrate properties (if schema changed)
  → Rollback available for 30 days
```
