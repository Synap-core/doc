# Intelligence Service вҖ” Architecture & Integration Guide

**Date:** 2026-04-08
**Status:** Implemented вҖ” covers provisioning, AI gating, and per-surface integration.

---

## Overview

The Intelligence Service (IS) provides AI capabilities (chat, entity extraction, proactive insights, proposals) to Synap pods. Access is gated by subscription tier and provisioned per-pod via the Control Plane.

This document covers:
1. How IS provisioning works end-to-end
2. The `@synap-core/intelligence-connect` shared package
3. How each surface (Browser, Relay, CLI) gates AI features
4. Token pricing and quota enforcement

---

## 1. Provisioning Flow

```
User subscribes (Stripe)
  вҶ’ CP creates: subscription + data_pod + intelligenceAccess
  вҶ’ Pod reaches "ready" state

User activates AI (Browser Settings / Relay Settings / CLI)
  вҶ’ POST /intelligence/provision/{podId} (to CP)
  вҶ’ CP verifies subscription entitlement
  вҶ’ CP builds provision JWT (ES256, 10min TTL)
  вҶ’ CP calls IS Hub: POST /admin/provision-pod
  вҶ’ IS returns serviceApiKey
  вҶ’ CP stores encrypted key in DB
  вҶ’ CP relays key to pod: POST /api/provision/register-intelligence
  вҶ’ Pod stores encrypted (AES-256-GCM) in intelligence_services table
  вҶ’ Pod starts using IS for all AI operations
```

### Re-provisioning (Frontend-Driven)

**Architecture principle:** The pod is agnostic вҖ” it does NOT call the Control Plane. Re-provisioning is the frontend's responsibility.

When IS credentials fail:
1. Pod detects 401 from IS вҶ’ `markServiceCredentialError()` (marks DB status only)
2. `GET /api/provision/status` reports `credentialsValid: false`
3. Frontend detects via `useIntelligenceStatus` вҶ’ status = `out_of_sync`
4. Frontend calls CP: `POST /intelligence/provision/{podId}`
5. CP generates fresh key, pushes to pod via `/api/provision/register-intelligence`
6. Pod marks service as `active`
7. Next AI request succeeds

**Auto-provision triggers:**
- **App launch** (`useRelaySetup` Step 5): checks if user has subscription but IS not provisioned вҶ’ provisions automatically
- **Capture fallback**: when `capture.structure` returns a single low-confidence note (IS unavailable), the capture client auto-provisions via CP and retries the call
- **Settings**: user can manually provision/reconnect from AI & Automations page

### Health Monitoring

- pg-boss job pings IS every 2 minutes via `GET /health`
- Tracks `lastHealthStatus`: healthy / degraded / unhealthy
- Unhealthy services are skipped during routing

**Location:** `synap-app/packages/core/intelligence-connect/`

### Architecture

```

```

### AI Gate Pattern

Every AI-calling surface should check `useAIGate()` before making AI requests:

```typescript
import { useAIGate } from '@synap-core/intelligence-connect/react';

function MyAIFeature() {
  const gate = useAIGate(options);

  if (gate.isLoading) return <Spinner />;

  if (!gate.available) {
    return (
      <Banner
        message={gate.message}
        actionLabel={gate.actionLabel}
        actionHref={gate.actionHref}
      />
    );
  }

  // Proceed with AI call
}
```

`AIGateResult` provides:
- `available: boolean` вҖ” can AI calls proceed?
- `statusKind` вҖ” the raw 10-state kind
- `message` вҖ” human-readable reason when unavailable
- `actionLabel` вҖ” button text ("Subscribe", "Activate", "Reconnect", etc.)
- `actionHref` вҖ” where to navigate (`synap://settings/intelligence`, URL, etc.)
- `isLoading` вҖ” still checking (don't show error)
- `refresh()` вҖ” manually re-check

### Platform Adapters

Each app provides its own config via an adapter hook:

**Browser:**
```typescript
import { useBrowserAIGate } from '../hooks/useBrowserAIGate';
const gate = useBrowserAIGate();
```

**Relay:**
```typescript
import { useRelayAIGate } from '../hooks/useIntelligenceConnect';
const gate = useRelayAIGate();
```

**CLI / Scripts (no React):**
```typescript
import { checkAIAvailability } from '@synap-core/intelligence-connect';
const result = await checkAIAvailability(config);
if (!result.available) console.log(result.message);
```

### Configuration

Each adapter builds an `IntelligenceConnectConfig`:

```typescript
interface IntelligenceConnectConfig {
  cpFetch: CpFetchFn;      // Authenticated CP REST calls
  podUrl: string | null;    // Pod base URL
  podId: string | null;     // Pod ID from CP
  timeout?: number;         // Fetch timeout (default 5000ms)
}
```

- **Browser**: `cpFetch` from `platformClient.ts` (IPC-backed for Electron cookie auth)
- **Relay**: Module-scoped fetch with Bearer token from AsyncStorage
- **CLI**: Direct fetch with stored credentials from `~/.synap/credentials.json`

## 4. Per-Surface Integration

### Browser

**Gating philosophy:** Implicit degradation. UI is always clickable вҖ” AI features fail gracefully at runtime. The IntelligenceProvisionSection in Settings is the explicit gating surface.

| Component | Gate | Behavior when unavailable |
|-----------|------|---------------------------|
| IntelligenceApp | `intelligenceActive` prop | Shows "Activate" CTA |
| Chat (channels.sendMessage) | Pod marks `credential_error` | Frontend detects and re-provisions via CP |
| Capture (structure) | Fallback detection + auto-provision | Returns fallback вҶ’ client provisions вҶ’ retries вҶ’ shows AI results |
| Proposals | None | Query returns empty |
| Settings | `useIntelligenceStatus` | Full 10-state provision UI |

**Available hooks:**
- `useBrowserAIGate()` вҖ” for new AI features that want explicit gating
- `useIntelligenceStatus()` вҖ” for the settings/provision UI

### Relay

**Gating philosophy:** Explicit + graceful. AI surfaces check the gate and degrade with helpful messages.

| Component | Gate | Behavior when unavailable |
|-----------|------|---------------------------|
| AI Chat (search) | `useRelayAIGate` | Returns message: "Subscribe to..." |
| Quick Capture | `useRelayAIGate` | Falls back to offline structure |
| Event Capture | `useRelayAIGate` via useCapture | Falls back to offline structure |
| AI Settings | `useRelayIntelligenceStatus` | Status card + action buttons |
| Feed (proactive) | None | No proactive messages if IS off |

**Available hooks:**
- `useRelayAIGate()` вҖ” check before AI calls
- `useRelayIntelligenceStatus()` вҖ” full status for settings
- `useRelayProvisionIntelligence()` вҖ” trigger provisioning
- `useRelayIntelligenceUsage()` вҖ” token usage tracking

### CLI

**Gating philosophy:** Check and offer to provision during setup.

| Command | Gate | Behavior |
|---------|------|----------|
| `synap init` | `isStep()` | Checks pod IS status, provisions via CP if subscribed |
| `synap status` | `/api/provision/status` | Shows IS health, credentials validity |

## 7. Adding AI Gate to a New Feature

1. Import the gate hook for your platform:
   ```typescript
   // Browser
   import { useBrowserAIGate } from '../hooks/useBrowserAIGate';
   // Relay
   import { useRelayAIGate } from '../hooks/useIntelligenceConnect';
   ```

2. Call it at the top of your component/hook:
   ```typescript
   const gate = useBrowserAIGate(); // or useRelayAIGate()
   ```

3. Check before AI calls:
   ```typescript
   if (!gate.available) {
     // Show gate.message to user
     // Optionally render a button with gate.actionLabel вҶ’ gate.actionHref
     return;
   }
   // Proceed with AI call
   ```

4. For non-React contexts (CLI, scripts):
   ```typescript
   import { checkAIAvailability } from '@synap-core/intelligence-connect';

   const result = await checkAIAvailability({
     cpFetch: myFetchFn,
     podUrl: 'https://pod.example.com',
     podId: 'pod-uuid',
   });
   if (!result.available) {
     console.error(result.message);
   }
   ```
