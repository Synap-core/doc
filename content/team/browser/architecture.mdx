# Browser Architecture

Browser is an Electron application with two primary processes (main + renderer), multiple sub-windows (app OS windows, PIP modes, quick-capture overlay), and a system tray. This page documents all process boundaries, IPC channels, data flows, and security controls.

## Window Model

Browser supports 3 window types, managed by `WindowManager`:

### MainWindow

Single primary window. Config:
- 800×600 minimum, resizes with user
- State persisted to `userData/window-state.json`
- `contextIsolation: true`, `nodeIntegration: false`, `sandbox: true`, `webviewTag: true`
- Auto-hides menu bar, default title bar
- OAuth popups allowed (Google, GitHub, Microsoft, etc. — see `isOAuthPopup()`)
- `window.open()` → in-app browser tabs or OS handler for non-HTTP schemes

### AppWindows (App OS)

Secondary windows, created per-app. Two modes:

| Mode | Use Case | Frame | Resizable | On Top |
|------|----------|-------|-----------|--------|
| `normal` | Whiteboard in separate window | Yes | Yes | No |
| `pip` | Floating widgets (stat cards, mini chat) | No | No | Yes |

Created via `windowManager.createAppWindow({ id, appId, mode })`. Each has its own preload, loads renderer with URL params (`?windowId=X&appId=Y&mode=Z`). On close, main window receives `app-window:closed` IPC.

### Quick Capture Overlay

Always-on-top floating window. Created lazily on first hotkey press.
- Resizable via `quick-capture:set-height` IPC
- Hides on save/escape via `quick-capture:hide` IPC
- Reads pod URL from `SYNAP_INSTANCE_URL` cookie directly
- Tracks workspace ID pushed from main window renderer

## Browser Webview System

The in-app browser uses a **dedicated Electron session** (`persist:browser`) that isolates webview state from the renderer's default session:

1. **Custom preload**: `webview-preload.js` masks Electron fingerprints (`navigator.webdriver`, `navigator.plugins`)
2. **User-Agent strategy**: Chrome UA by default; auto-switches to Firefox UA for `.google.com`, `.microsoft.com`, `.apple.com` domains (bypasses embedded-browser blocks used by Google Sign-In, Microsoft 365, etc.)
3. **Client Hints stripping**: Removes `sec-ch-ua` headers for Firefox UA sessions
4. **DRM support**: Grants `protectedMediaIdentifier` permission for streaming sites
5. **Certificate verification**: Strict in production; permissive in dev/localhost
6. **HTTP Basic Auth**: Handled via main-process `login` event → renderer IPC dialog (60s timeout)

## IPC Channel Registry

All IPC channels are registered by `IpcRouter`. Key categories:

| Category | Channels | Direction |
|----------|----------|-----------|
| Window | `window:minimize`, `window:maximize`, `window:close`, `window:fullscreen` | Renderer → Main |
| App Windows | `app-window:create`, `app-window:close`, `app-window:bounds`, `app-window:focus`, `app-window:close-all` | Bidirectional |
| Pod | `pod:connect`, `pod:disconnect`, `pod:ping`, `pod:status` | Bidirectional |
| Capture | `capture:start`, `capture:save`, `capture:cancel`, `capture:analyze` | Renderer → Main |
| Terminal | `terminal:spawn`, `terminal:write`, `terminal:resize`, `terminal:close` | Renderer → Main |
| Extension | `extension:install`, `extension:uninstall`, `extension:list` | Renderer → Main |
| Quick Capture | `quick-capture:hide`, `quick-capture:set-height`, `quick-capture:get-session`, `quick-capture:set-workspace` | Renderer → Main |
| Updater | `updater:install` | Renderer → Main |
| Notifications | `app:deep-link` (renderer ← main) | Main → Renderer |
| Auth | `webview-auth-request`, `webview-auth-response` | Bidirectional |
| App Events | `synap-app:open-external` (renderer ← main) | Main → Renderer |

## Session & Auth Flow

1. **Connection**: Renderer connects to pod via tRPC using `SYNAP_INSTANCE_URL` cookie
2. **Session token**: Pod returns tRPC session token, stored in renderer Zustand store
3. **Workspace selection**: User picks workspace → pushed to quick-capture window via IPC
4. **OAuth connectors**: Nango Connect UI opens in popup window (detected by `isOAuthPopup()`)
5. **Deep links**: `synap://` protocol → main process handles → forwards to renderer via IPC

## Data Flow Summary

```
User action (renderer)
  ↓ (IPC → Main)
IpcRouter dispatches to specific handler (Window/Capture/Terminal/Extension)
  ↓ (Main process logic)
CaptureService: webview extraction → entity analysis → pod write via tRPC
WindowManager: window creation/bounds → native OS window
  ↓ (IPC ← Renderer)
Renderer receives response, updates UI state
```

## Package Dependencies

Browser is a **host**, not a feature package consumer. It imports ~55 `@synap-core/*` workspace packages from `synap-app`, including:

**Core**: `ui-system`, `types`, `client`, `hooks`, `stores`, `cell-runtime`, `view-renderer`, `capabilities`
**Feature modules**: `ai-chat`, `channels`, `proposals`, `connectors`, `capture-pipeline`, `imports-core`, `imports-parsers`, `bento-view`, `masonry-view`
**Apps consumed as packages**: `data-app`, `data-structure`, `docs-app`, `document`, `entity-card`, `entity-views`, `intelligence`, `whiteboard`, `workflows`, `relay`, `relay-app`, `bento-view`, `notification`, `search`, `sidebar`, `onboarding`, `marketplace`, `platform-auth`, `billing`, `pricing`, `workspace`
**Platform**: `auth`, `pod-client`, `pod-failover`, `external-connect-client`, `platform-flow`, `control-plane-types`, `deep-link-constants`, `ipc-contract`, `react`

## Design Principles

1. **contextIsolation is non-negotiable** — all Node.js API access goes through explicit IPC channels
2. **No singleton window** — WindowManager manages N app windows, each with independent lifecycle
3. **One entry, many routes** — renderer loads `index.html` for every window; window params (id, appId, mode) are URL search params
4. **Session token lives in renderer** — main process never holds auth tokens; pod URL comes from cookies
5. **Deep links are first-class** — `synap://` protocol is registered at app lifecycle; all platforms supported (macOS `open-url`, Windows/Linux `second-instance`)
