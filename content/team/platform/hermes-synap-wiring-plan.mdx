## Part 1 — Skills: Two Systems, One Decision

### The two skill systems

There are two completely separate skill systems in Synap. They have nothing in common except the word "skill."

#### System A — IS Internal Skills (proprietary, never expose externally)

**Location:** `synap-intelligence-service/apps/intelligence-hub/src/skills/`

These are markdown instruction files injected into the OrchestratorAgent / PersonaAgent system prompt at runtime. They're loaded by `skill-loader.ts` based on which tools the agent has access to (`SKILL_TRIGGERS` map).

| File | Loaded when |
|------|-------------|
| `hub-protocol.md` | Any agent with `search_unified` |
| `entity-management.md` | Agent has `create_entity` |
| `document-management.md` | Agent has `create_document` |
| `entity-schema.md` | Agent has `create_profile` |
| `workspace-views.md` | Agent has `create_view` |
| `automations.md` | Agent has `create_automation` |
| `inline-references.md` | Any tool available |
| `cli-execution.md` | Agent has `run_command` |
| `native-widget.md` | Agent has `create_native_widget` |
| `view-types/*.md` (4) | Batch-loaded with `create_view` |

**Decision: Hermes does NOT use these.** They teach tool call patterns for tools that only exist inside the IS (`create_entity`, `search_unified`, etc.). Hermes has an entirely different tool registry and calling convention.

#### System B — Hub Protocol Skills (user-created, DB-stored, externally callable)

**Location:** `synap-backend/packages/api/src/routers/skills.ts`

Skills stored in the DB with `kind: "instruction" | "code"`, scoped to pod/user/workspace. Listed via `GET /api/hub/skills/getSkills`, executed via `POST /api/hub/skills/:id/execute` → IS `/api/skills/execute`.

**Decision: Hermes CAN and SHOULD call these** — they're exactly what external agents need. ✅ `POST /api/hub/skills/:id/execute` now exists.

#### System C — OpenClaw/Eve Skills (SKILL.md files, distribution format)

**Location:** `synap-backend/skills/` (three packages: synap, synap-schema, synap-ui) and `~/.eve/skills/synap/SKILL.md`

These are markdown documents in the Anthropic Agent Skills spec format. They document the Hub Protocol REST API surface. The `synap/SKILL.md` at `~/.eve/skills/synap/` is exactly what Hermes needs as its primary skill.

**Decision: Volume-mount into Hermes container.** ✅ Done — one `-v ~/.eve/skills:/opt/data/skills:ro` flag in `installHermes()`.

### Summary decision table

| Skill type | Should Hermes use? | How | Status |
|---|---|---|---|
| IS internal skills (hub-protocol.md, entity-management.md, …) | **No** | N/A | N/A |
| DB user-created skills | **Yes** | `POST /api/hub/skills/:id/execute` | ✅ shipped |
| `~/.eve/skills/synap/SKILL.md` | **Yes** | Volume mount | ✅ shipped |

## Part 2 — MCP: Who Serves, Who Connects, Why

### What already exists

The MCP infrastructure is more built than it appears:

- **`mcp-servers` DB table** — workspace-level registry
- **`mcp-client-manager.ts`** in IS — manages connection lifecycle
- **`request-mcp-tool.ts`** — IS tool for requesting MCP tool access (requires user approval)
- **`intelligence-services.mcpEndpoint` + `mcpApproved`** — IS can register itself as MCP server
- **Eve Dashboard OpenClaw config panel** — already has MCP preset install/remove UI (`openclaw-config.tsx`)
- **Hermes MCP server** — `:9120`, `MCP_SERVER_ENABLED=true` in docker run ✅
- **Claude Code wiring** — `.claude/mcp-servers/hermes.json` auto-written by `writeBuilderProjectEnv()` ✅

### The three MCP roles

```
┌──────────────────────────────────────────────────────────────┐
│  MCP Servers (provide tools)                                 │
│                                                              │
```

### Phase 2 — Identity Linking ✅ (done, lifecycle gap remains)

- Migration `0021_api_keys_linked_user.sql` — `linked_user_id TEXT` on `api_keys`
- `api-key-repository.ts` — `CreateApiKeyInput.linkedUserId` stored on create
- `POST /setup/agent` accepts `linkedUserId` in body
- `HubVariables.linkedUserId?: string` in `_shared.ts`
- Hub middleware sets `c.set("linkedUserId", keyRecord.linkedUserId)` when present
- `rest/memory.ts` dual-write: after agent fact write, writes same fact for linked human (non-fatal)

**Remaining:** lifecycle `installHermes()` doesn't yet fetch pod owner userId and pass it as `linkedUserId` on key provisioning.

## Decisions recorded

| Question | Decision | Reason |
|---|---|---|
| Use IS internal skills in Hermes? | No | They reference IS-specific tool names that don't exist in Hermes |
| Expose user-created skills to Hermes? | Yes, via new Hub REST execute endpoint | Skills are the user's automation surface — agents should all reach them |
| Which MCP server does Hermes run? | `hermes mcp serve` on :9120, wired by Eve | Hermes native, zero custom code |
| Where do external MCPs get added? | Eve Dashboard → Intelligence → MCP tab per agent | Centralised, no file editing |
| Memory ownership after identity linking? | Fact stored with both `agentUserId` and `linkedUserId` | Queryable by Hermes (agent) and the Synap UI (human) |
| Channel credentials: per-agent vs centralised? | Centralised in `secrets.channels`, routing in `secrets.channelRouting` | Already done — just needs GUI in Phase 4 |
| Webhook schema: draft vs Hub Protocol canonical? | Dropped old draft, recreated with Hub schema | `subscriber_id` = apiKeyId ties subscriptions to key identity |
