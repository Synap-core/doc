**Status:** Phases 1–3 COMPLETE (2026-04-10). Phase 4 (delivery router) is future work.

The event chain is the backbone of Synap's intelligence layer. Every mutation that matters — entity created, message sent, contact imported, capture completed — flows through a typed event system that:

1. Provides a complete audit trail (who did what, when, why)
2. Enables automations to react to any state change
3. Powers the proactive AI based on real signals
4. Bridges external data sources (Telegram, LinkedIn, connectors) into the same reactive graph
5. Feeds pod-to-pod sync without bespoke replication logic

---

## Architecture: The Two-Tier System

Every mutation does two things after the DB write succeeds:

```
DB write succeeds
      │
      ├─── 1. events table (audit + sync)
      │         append-only, immutable
      │         read by: sync-push worker (pod-to-pod replication)
      │         hooks: SSE broadcast, Socket.IO domain events, sync
      │
      └─── 2. pg-boss side-effects queue
                emitSideEffects() call, fire-and-forget
                queues: automation-trigger-match, search-index,
                        webhook-delivery, entity-embedding, cross-thread-notify
                read by: automation-trigger-matcher, search workers, etc.
```

Both tiers are always written together on all new events (P1-C pattern). Legacy entity repository paths write to the event log via their own hooks and to pg-boss via `emitSideEffects()` in the tRPC routers.

## Channel Types and channelPurpose Column

Personal channel variants (personal chat / proactive feed / capture thread) are now distinguished by the `channelPurpose` column instead of JSONB metadata flags:

```typescript
// Old (JSONB flags — still readable via OR fallback for migration)
{ metadata: { "isPersonal": true } }
{ metadata: { "isProactiveFeed": true } }
{ metadata: { "isCaptureThread": true } }

// New (indexed column — canonical going forward)
channelPurpose = 'chat'   // bidirectional user↔AI conversation
channelPurpose = 'feed'   // AI posts, user reads
channelPurpose = 'audit'  // hidden capture audit trail
```

**Migration:** All new channel inserts set `channelPurpose`. All queries use `channelPurpose = X OR metadata->>'isX' = 'true'` for backward compatibility. SQL migration `0064_channel_purpose.sql` backfills existing rows.

13 tables × 9 variants = 117 event types. See `packages/events/src/generator.ts`.

## Hub Protocol Events Endpoint

IS agents can query the event log via:
```
GET /api/hub/events?userId=...&workspaceId=...&type=...&subjectType=...&fromDate=...&limit=...
```
Returns: `{ events: EventRecord[] }` (defaults to last 7 days, max 200 events)

This enables IS tools like `query_recent_events` to surface relevant history for AI context.

## Automation System

### Trigger matching (automation-trigger-matcher.ts)

Supports wildcard patterns (`capture.*`, `connector_sync.*`, `proactive.*`).  
Filter expressions reference `data` fields: `config.profileSlug`, `config.proactiveType`, etc.

#### Supported trigger filters (with P1 additions)

| Pattern | Extra filters | Enabled by |
|---------|--------------|-----------|
| `entities.*` | profileSlug, entityType | entity-repository |
| `documents.*` | — | document-repository |
| `channel_message.*` | channelId | channels.ts |
| `proposal.*` | — | permission-check.ts |
| `capture.*` | profileSlug (array match) | P1-A  |
| `connector_sync.*` | provider (telegram/linkedin) | P1-B  |
| `proactive.*` | proactiveType | P1-C  |

### DAG executor (automation-executor.ts)

Output nodes write to:
- `channelType: 'proactive'` → proactive feed channel (uses `channelPurpose='feed'`) 
- `channelType: 'personal'` → personal chat channel (uses `channelPurpose='chat'`) 
- `outputType: 'notification'` → notifications table

## Phase 4 — Delivery Router (Future Work)

A routing layer that decides which surface receives a signal based on type + user preferences. Not yet implemented.

**Problem it solves:** Today each producer hardwires its delivery path:
- Cron workers → `postProactiveMessage()` → proactive feed
- Automations → output node → channel or notification
- IS → Hub Protocol → personal chat

There is no `routeSignal()` that reads `workspace.settings.deliveryPreferences` to decide. Building this requires:

1. Add `deliveryPreferences: DeliveryPreferences` to `WorkspaceSettings` schema
2. Define `DeliveryTarget` union type (personal_chat | proactive_feed | notification | entity_update)
3. Implement `routeSignal(eventType, payload, userId, workspaceId)` that resolves target from preferences
4. Migrate callers progressively (cron workers first, automation executor second)

This is Phase 4 work. All other phases are complete.
