## 2. Component Internals

### 2.1 Data Pod (synap-backend)

**Entry point:** `apps/api/src/index.ts`

#### Startup sequence (in order)

```
1. Load .env (dotenv/config)
2. Validate env: DATABASE_URL, storage provider, AI config, Ory Stack, IS connection, CORS
   → exit(1) if any validation fails
3. Create Hono app
4. Apply middleware: CORS, security headers, rate limiting (500 req/15min IP), size limit (10MB)
5. Register routes:
   GET  /health          ← no auth
   GET  /metrics         ← Prometheus
   POST /api/hub/*       ← Hub Protocol REST (Bearer or session token)
   POST /mcp             ← MCP JSON-RPC (Bearer token)
   POST /v1/*            ← OpenAI-compat (Bearer token)
   /trpc/*               ← tRPC (session token)
   /api/files            ← file upload
   /api/chat             ← chat stream
   /admin/*              ← SPA fallback
6. Run migrations (separate backend-migrate container)
7. validateSchemaCoherence()  ← blocks server start if critical columns missing
8. Start HTTP listener on port 4000
9. Start pg-boss job queue + register 27 workers + 3 cron jobs
10. Register process error handlers (unhandled rejection → exit(1))
```

**Schema coherence tripwire** (`packages/database/src/utils/schema-coherence.ts`):
- Validates 30+ critical columns across 14 tables exist in the live DB
- Catches drift between Drizzle schema and actual migrations
- Runs AFTER migrations, BEFORE serving traffic
- Exits with code 1 + structured error message pointing to the relevant migration

#### Hub Protocol REST (`packages/api/src/routers/hub-protocol-rest.ts`, 4649 lines)

Auth modes:
| Mode | Header | Who uses it |
|------|--------|-------------|
| Bearer API key | `Authorization: Bearer <key>` | IS agents, OpenClaw, CLI |
| Kratos session | `X-Session-Token: <token>` | Browser, extensions |

Key routes:
```
GET  /api/hub/health              ← no auth
GET  /api/hub/users/me            ← identity
GET  /api/hub/workspaces          ← list accessible workspaces
GET  /api/hub/entities            ← search entities
POST /api/hub/entities            ← create/update entity
GET  /api/hub/documents           ← search documents
POST /api/hub/channels/{id}/send  ← send message
POST /api/hub/memory              ← store fact (embedding optional)
POST /api/hub/setup/agent         ← create agent user + API key
```

Rate limits:
- Global IP: 500 req/15min
- Per API key: configurable in `api_keys` table
- MCP per key: 100 req/min

### 2.2 Control Plane API (synap-control-plane-api)

**Framework:** Hono REST (not tRPC)
**Auth:** Better Auth (email/password + OAuth + 2FA TOTP plugin)

Key services:
- **Billing**: Stripe checkout + webhooks (`src/services/billing/stripe.ts`)
- **Provisioning**: Hetzner API for server creation (`src/jobs/provision-pod.ts`)
- **DNS**: Cloudflare A records (`src/services/dns.ts`)
- **Email**: Resend for invitation emails
- **JWT signing**: ES256 ECDSA (`src/lib/signing-key.ts`)

Pod lifecycle state machine:
```
pending_configuration  ← created on Stripe webhook; awaiting user setup form
        ↓
     pending           ← user submitted setup form
        ↓
  creating_server      ← Hetzner API called
        ↓
  configuring_dns      ← DNS A record created
        ↓
   waiting_dns         ← polling DNS propagation
        ↓
  waiting_health       ← polling /health endpoint (cloud-init completing)
        ↓
syncing_intelligence   ← registering with IS
        ↓
sending_invitation     ← sending admin email
        ↓
      active           ← fully operational
        
      error            ← any step failed (retryable via admin)
    suspended          ← manually or billing lapse
     archived          ← soft-deleted
    warm_pool          ← pre-provisioned, no owner yet
```

## 3. Authentication Model

### CP ↔ Pod: ES256 asymmetric JWTs

```
CP API generates:   signCpJwt(payload, expiry) → ES256 JWT
                    Private key: CP_EC_PRIVATE_KEY (PKCS#8 PEM)

Pod verifies:       fetch({CONTROL_PLANE_URL}/.well-known/jwks.json)
                    → cache public key
                    → verify JWT signature + expiry + jti (replay prevention)

JWKS endpoint:      GET /.well-known/jwks.json on CP API
                    Returns: { keys: [{ kty, crv, use: "sig", alg: "ES256", kid, x, y }] }
```

JWT types used:

| Type | TTL | Purpose |
|------|-----|---------|
| `handshake` | 5 min | Shared pod pre-provisioning; creates Kratos identity + workspace |
| `provision` | 10 min | Dedicated pod IS sync; registers with Hub, delivers Hub API key |
| `agent_setup` | varies | OpenClaw agent setup; verified at `POST /api/hub/setup/agent` |
| `addon_activate` | varies | Addon activation; same endpoint |
| `tier_update` | 5 min | Push subscription tier changes to pod |
| `update` | 10 min | Rolling update trigger; sent to pod-agent |
| `update_callback` | 30 min | Pod authenticates result callback to CP |

**Key generation:**
```bash
openssl ecparam -name prime256v1 -genkey -noout | \
  openssl pkcs8 -topk8 -nocrypt -outform PEM > cp-private-key.pem
```

### Pod: API keys (Bearer tokens)

Generated at `POST /api/hub/setup/agent`. Format: `synap_hub_live_{32_random_hex}` (prod) or `synap_hub_test_` (dev). Stored hashed in `api_keys` table. Scopes: `hub-protocol.read`, `hub-protocol.write`, `mcp.read`, `mcp.write`.

### Pod: Kratos sessions

Browser/app sessions via Ory Kratos. Sent as `X-Session-Token` header to Hub Protocol. Full access (no scope restrictions).

### CP API: Better Auth sessions

httpOnly cookie. Email+password + optional TOTP 2FA. Used by CP Dashboard and landing page.

## 4. Signup → Active Pod Workflow

### Phase 1: Signup

```
POST /auth/sign-up/email
  → Better Auth creates user record
  → Email verification sent (Resend)
  → POST /auth/verify-email callback
      → auth.ts user.create.after hook:
          1. Update waitlist.phase: "waitlist" → "signed_up"
          2. fire-and-forget: provisionUserOnSharedPod()
             → sign handshake JWT (5 min)
             → POST pod:/api/handshake
             → Creates Kratos identity + default workspace on shared pod
          3. Create 7-day trial subscription (solo tier, no Stripe yet)
```

### Phase 2: Upgrade (Stripe checkout)

```
POST /billing/checkout
  → createCheckoutSession(customerId, priceId, successUrl, cancelUrl, userId)
  → Idempotent: same customer+price+day → same session

User completes checkout
  → POST /webhooks/stripe  (checkout.session.completed)
      → verifyWebhookSignature()
      → handleCheckoutCompleted():
          1. Resolve user (by stripeCustomerId → client_reference_id → metadata.userId)
          2. Resolve tier from priceId
          3. INSERT subscriptions (onConflictDoNothing for replays)
          4. Guard: if pod already created → skip
          5. Guard: stale pending_configuration pods (>2h) → relink
          6. Guard: re-subscribe → triggerRestorePod()
          7. INSERT dataPods { status: "pending_configuration" }
             subdomain: "{adj}-{noun}-{hex}"
             deploymentMode: "full_managed"
             provider: user.preferredProvider (default: hetzner)
          8. INSERT intelligenceAccess { monthlyTokenLimit: tier.aiTokens }
```

### Phase 3: User setup form

```
GET /pods/:id/provisioning-setup    ← returns form definition (fields depend on tier)
POST /pods/:id/provisioning-setup
  → validate subdomain uniqueness
  → validate custom domain (tier: pro+ only)
  → UPDATE dataPods { status: "pending", subdomain, serverLocation, adminEmail }
  → triggerProvisionPod(podId)       ← starts provision-pod job
```

### Phase 4: Server provisioning (provision-pod.ts)

```
Step 1: Create Hetzner server
  → Check idempotency: if pod.serverId already set, skip creation
  → Generate intelligenceApiKey (32 bytes hex)
  → Generate provisioning token (32 bytes hex, encrypted, stored in provisioningTokens)
  → Create Hetzner Volume (optional, persistent)
  → Generate cloud-init script
  → createServerWithFallback():
      → try preferred server type (cx33, cx43...)
      → fallback through alternatives on availability error
      → fallback through locations: nbg1 → fsn1 → hel1
  → Wait for IP (120s timeout)
  → UPDATE dataPods { status: "configuring_dns", serverId, serverIp }

Step 2: Configure DNS
  → Create Cloudflare A record: {subdomain}.synap.live → serverIp
  → UPDATE dataPods { status: "waiting_dns" }

Step 3: Poll DNS propagation
  → Resolve {subdomain}.synap.live up to 5 minutes
  → Fallback: HTTPS health probe if DNS timeout

Step 4: Poll health / cloud-init  [UPDATE: "deploying_services" → "waiting_health"]
  → Poll every 15s, up to 30min (120 attempts)
  → Two signals:
      a) Server POSTs to /install-callback (provisioning token auth) → progress events
      b) GET https://{domain}/health → fallback probe
  → On install_complete: UPDATE { status: "syncing_intelligence" }
  → On install_failed: DELETE server, abort (permanent error, no retry)
  → On timeout: UPDATE { status: "error" }, allow admin retry

Step 5: Sync Intelligence Hub
  → Sign provision JWT (type: "provision", 10 min)
  → POST pod:/api/provision/connect  ← establish trusted CP session
  → POST hub:/admin/provision-pod    ← register customer, set token limits
  → POST pod:/api/provision/register-intelligence  ← deliver Hub API key to pod
  → (optional) relay hubProtocolApiKey back to IS for proactive calls
  → UPDATE dataPods { status: "sending_invitation" }

Step 5b: Auto-install packages
  → For each pkg in autoInstallPackages:
      POST pod:/api/trpc/workspaces.seedPlugin (provisioning token auth)

Step 6: Send invitation email
  → sendPodReadyEmail(adminEmail, podUrl)
  → UPDATE dataPods { status: "active" }
```

**Total time (typical):** 3–8 minutes (server creation dominates)

## 5. Pod Update / Rolling Upgrade

**File:** `synap-control-plane-api/src/jobs/rolling-update.ts`

```
Trigger: admin action "update" from CP Dashboard → POST /admin/pods/:id/action

Rolling update flow:

1. For fleet > 1 pod: designate one canary
2. Sign two JWTs:
   a. Update trigger JWT (type: "update", 10 min):
      { podId, targetVersion, updateId, callbackUrl, callbackJwt }
   b. Callback JWT (type: "update_callback", 30 min):
      Pod uses this to authenticate its result POST back to CP

3. POST pod:/api/pod-agent/command { jwt: triggerJwt }
   Headers: X-JWKS-URL: {CP_URL}/.well-known/jwks.json
   → pod-agent pulls new image, restarts services, calls callback

4. Canary health check:
   Poll GET {pod_url}/health every 15s, up to 5 min
   → If canary fails: abort entire update
   → If canary healthy: proceed with remaining pods in parallel batches

5. Pod result callback:
   POST /pods/update-result/{podId}  { status, version }
   Authorization: Bearer {callbackJwt}
   → CP verifies callback JWT
   → UPDATE pod_updates { status, version }
   → UPDATE dataPods { version, status: "active" | "error" | "rolled_back" }
```

## 6. Warm Pool

**Files:** `synap-control-plane-api/src/jobs/warm-pool-replenish.ts` + `claim-warm-pod.ts`

Pre-provisioned pods with no owner. Configured via `WARM_POOL_SIZE` + `WARM_POOL_SERVER_TYPE` env vars.

```
Replenishment:
  1. Count in-flight warm pool pods (all non-error statuses)
  2. If errors exist: skip (manual cleanup required)
  3. For each pod needed up to WARM_POOL_SIZE:
     → INSERT dataPods { userId: null, status: "pending", subdomain: "safe-pool-{hex}" }
     → triggerProvisionPod(podId, { warmPool: true })
     → Provisioning skips Step 5 (IS sync) and Step 6 (email)
     → Final status: "warm_pool"

Claim (on Stripe checkout):
  → webhook routes to triggerClaimWarmPod(podId, userId, subscriptionId)
  → Step 5: sync Intelligence Hub (deferred from provision)
  → Step 6: send admin invitation email
  → UPDATE dataPods { userId, subscriptionId, status: "active" }
  → fire-and-forget: triggerWarmPoolReplenish() to refill pool
```

**Key advantage:** Pod is ready in seconds for users (DNS + services already up). Only IS registration + email on claim path (~30s).

## 7. OpenClaw / Self-Hosted Agent Setup

**Files:**
- `synap-backend/deploy/setup-openclaw.sh`
- `synap-backend/packages/api/src/routers/hub-protocol-rest.ts` (lines 4205–4647)

### `POST /api/hub/setup/agent` endpoint

Auth: ES256 JWT (`type: "agent_setup"` or `"addon_activate"`) OR `PROVISIONING_TOKEN` (self-hosted fallback).

Sequence:

```
1. Authenticate (CP JWT → PROVISIONING_TOKEN fallback)
2. Parse body: { agentType, workspaceId?, definition? }
3. Find or create workspace with packageSlug = 'agent-os'
   → If creating: load templates/agent-os.json
     → createWorkspaceFromDefinition() (6 profiles, 14 views)
     → Enqueue workspace-init job
4. Repair workspace: ensure owner member exists (self-healing)
5. Find or create agent user (userType: "agent", agentMetadata.agentType)
6. Grant workspace membership (role: "editor", idempotent)
7. Revoke old API keys for this agent user
8. Generate new Hub Protocol API key:
   synap_hub_live_{32_random_hex}
   Scopes: hub-protocol.read, hub-protocol.write, mcp.read, mcp.write
9. Return { agentUserId, workspaceId, hubApiKey, keyId }
   (key only returned once, not retrievable)
```

### setup-openclaw.sh flow

```bash
./setup-openclaw.sh [--pod-url URL] [--token TOKEN]

1. GET {POD_URL}/health            ← verify pod is up
2. POST {POD_URL}/api/hub/setup/agent { agentType: "openclaw" }
   Authorization: Bearer {PROVISIONING_TOKEN}
3. Write to .env:
   OPENCLAW_HUB_API_KEY={hubApiKey}
   SYNAP_AGENT_USER_ID={agentUserId}
   SYNAP_WORKSPACE_ID={workspaceId}
4. docker compose --profile openclaw up -d openclaw
5. Poll http://localhost:18789/health (20 attempts, 3s interval)
6. Print: openclaw skill install synap
```

## 8. Migration System Contract

**File:** `synap-backend/packages/database/src/scripts/migrate.ts`

### Directory search order

1. `/app/migrations` (Dockerfile layout)
2. `/app/api/migrations` (deploy/Dockerfile layout)
3. `./migrations` (dev, relative to CWD)
4. `${__dirname}/../../migrations` (fallback)

### Execution contract

```
- Read all .sql files from migrations directory
- Sort alphabetically (prefix number controls order)
- For each file NOT in _migrations table:
    sql.begin(async (tx) => {
      await tx.unsafe(migrationContent)
      await tx`INSERT INTO "_migrations" (filename) VALUES (${filename})`
    })
    // If any SQL fails → transaction rolls back atomically
    // Failed migration is NEVER recorded as applied
- On any failure: log structured error (code, position, detail, hint) → exit(1)
- On success: exit(0)
```

### _migrations table

Tracked by `filename` (unique constraint). Single column after the hardening migration (removed `type` column from old Drizzle schema). Auto-created on first boot if not exists.

### Baseline migration (0000_baseline_schema.sql)

- Creates all 73 tables with `CREATE TABLE IF NOT EXISTS`
- Creates 196 indexes with `CREATE INDEX IF NOT EXISTS`
- Uses `ALTER TABLE ADD COLUMN IF NOT EXISTS` for every column (handles partial DB state)
- Exception: `events` table has NO ALTER TABLE guards (TimescaleDB hypertable restriction — cannot add columns with non-constant defaults to compressed columnstores)
- Inserts 103 legacy migration filenames into `_migrations` to prevent re-execution
- The single source of truth for a fresh pod schema

### Schema reconciliation (0099_schema_reconciliation.sql)

- Catch-up migration for pods that have older schemas
- Idempotent: all `ADD COLUMN IF NOT EXISTS`
- Must be updated whenever a new column is added to `src/schema/*.ts`

### Adding a new column (required steps)

1. Add to Drizzle schema (`packages/database/src/schema/*.ts`)
2. Add to `0099_schema_reconciliation.sql` with `ADD COLUMN IF NOT EXISTS`
3. Add to `schema-coherence.ts` `REQUIRED_COLUMNS` array
4. Create new numbered migration file (e.g., `0100_my_feature.sql`)

Trigger: PRs to `main`/`develop`, push to `main`/`develop`. Concurrency: cancel in-progress for same ref. Services: `timescale/timescaledb-ha:pg16` at `localhost:5432`.

#### `docker-publish.yml`

Trigger: push to `main`, `develop`, or `v*` tags.

| Branch / Tag | Image tags produced |
|---|---|
| `develop` | `:dev` |
| `main` | `:main` + `:sha-{short}` |
| `v*` (e.g. `v1.2.3`) | `:v1.2.3` + `:latest` |

Steps:
1. Build multi-platform image (linux/amd64 + linux/arm64)
2. Push to `ghcr.io/synap-core/backend`
3. On `main` or tag: `POST $CP_API_URL/admin/releases/notify` with `{ repo, version, sha, imageTag }` (auth: `CP_ADMIN_KEY`)

#### `staging-deploy.yml`

Trigger: push to `develop` only.

Steps:
1. Wait for `docker-publish.yml` to complete (the `:dev` image must be pushed)
2. Trigger rolling update via CP API:
   ```
   POST $CP_API_URL/admin/pods/$STAGING_POD_ID/action
   { "action": "update", "targetVersion": "dev" }
   Authorization: Bearer $CP_ADMIN_KEY
   ```
3. Poll `$STAGING_POD_URL/health` until healthy (30 attempts × 10s)
4. Run smoke tests:
   ```
   HUB_API_KEY=$STAGING_HUB_API_KEY \
   BASE_URL=$STAGING_POD_URL \
   ./scripts/smoke-test.sh
   ```
5. Report pass/fail as commit status

### 9.4 Docker image tag strategy (all services)

| Git ref | Tags applied |
|---------|-------------|
| Push to `develop` | `:dev` |
| Push to `main` | `:main`, `:sha-{7-char short SHA}` |
| Push `v*` tag (e.g. `v1.2.3`) | `:v1.2.3`, `:latest` |

Rationale: `:dev` is always safe to deploy to staging. `:sha-{short}` provides a pinned rollback target without requiring a formal release tag. `:latest` only moves on explicit version tags to avoid accidental production upgrades.

## 12. Known Gaps & P0–P3 Issues

### P0 — Data safety / correctness

- **Intelligence Hub cross-contamination risk**: IS is shared across pods by default. If IS has a bug in agent context isolation, one pod's agent could leak context to another. Needs per-pod IS instance or strict context boundaries.
- **`events` table (TimescaleDB)**: Cannot use `ALTER TABLE ADD COLUMN IF NOT EXISTS` with non-constant defaults. Any new column added to the `events` table in a migration must use `ALTER TABLE ... ADD COLUMN colname type` WITHOUT a `DEFAULT expr` using runtime functions (use `DEFAULT NULL` or literal constants only).

### P1 — Reliability

- **Warm pool health failures require manual cleanup**: If warm pool pods error during provisioning, `WARM_POOL_SIZE` is checked against non-error pods only. But errored pods aren't auto-deleted. Manual cleanup in CP Dashboard required before pool replenishes.
- **Canary rollback edge cases**: If the canary pod update succeeds but fails silently (health check passes, app broken), remaining pods get the bad update. No semantic health validation beyond `/health` endpoint.
- **Callback JWT idempotency**: Update result callback (`POST /pods/update-result/{podId}`) has no deduplication. If the pod retries the callback (network issue), it could flip the update status back. Add `ON CONFLICT DO NOTHING` or a `processed_at` guard.

### P2 — Observability

- **No distributed tracing**: OpenTelemetry is commented out in production on all services. No correlation IDs across CP → IS → Pod requests.
- **No circuit breaker on Kratos/Hydra**: Slow auth services block all authenticated requests. Add circuit breaker pattern.
- **Pod error state opaque**: When a pod lands in `status: "error"`, the CP Dashboard shows the status but the root cause is buried in provisioning logs. Surface the last error message at the pod record level.

### P3 — Quality

- **CI tests are non-blocking**: The `test` step in synap-backend CI uses `continue-on-error: true`. Test failures don't block merges. Should be blocking for at least the core packages.
- **No E2E tests**: No integration tests for Hub Protocol or MCP endpoints in CI. Only unit/component tests.
- **`property_defs.property.get(slug)` nondeterministic**: Returns first match — undefined behaviour with workspace overlays (documented in `team/platform/property-def-scoping` §Known Gaps).
- **No smoke test after deploy**: CI only tests the Docker build — no health check against a running instance.
